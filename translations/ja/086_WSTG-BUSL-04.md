## WSTG-BUSL-04 — プロセス・タイミングのテスト (Test for Process Timing)

プロセス・タイミング攻撃 (Process Timing Attack) は、特定の要求を処理するためにアプリケーションが要する時間を測定することで、内部状態やデータの存在に関する機密情報を推測する手法です。早期終了条件ロジック (Early-exit conditional logic)、データベース検索、または非コンスタントタイム (Non-constant-time) の暗号比較などによって生じるミリ秒単位のレスポンス時間の差異を分析することにより、攻撃者はサイドチャネル分析 (Side-channel analysis) を実行し、有効なユーザー名の列挙、パスワード断片の検証、または特定のレコードの存在確認を行うことができます。これらの脆弱性は通常、認証エンドポイント、パスワードリセットフォーム、および入力の妥当性に基づいてバックエンドロジックが分岐するリソース負荷の高いAPIフィルターなどで発生します。攻撃者の視点からは、アプリケーションがユーザーに対して同一かつ一般的なエラーメッセージを返している場合でも、これらのタイミングの差はバックエンドデータの真偽を明らかにする「ブーリアン・オラクル (Boolean oracle)」として機能します。

### アプリケーションは、有効なリソース要求と無効なリソース要求に対して異なるレスポンス時間を示しますか？
- [ ] いいえ — 入力の妥当性に関わらず、レスポンス時間は同一です
- [ ] はい — わずかなタイミングの差が存在しますが、統計的に有意ではありません
- [ ] はい — 一貫したタイミングの差が観測可能 (Observable) であり、信頼性の高いオラクルを提供しています

### レスポンス時間を正規化するために、コンスタントタイム (Constant-time) 比較関数や人工的な遅延が実装されていますか？
- [ ] はい — すべての機密性の高い比較操作にコンスタントタイム・アルゴリズムが適用されています *(最も安全)*
- [ ] はい — 人工的な遅延やジッター (Jitter) が有効化されていますが、基盤となるロジックは非コンスタントタイムのままです
- [ ] いいえ — タイミング緩和策が適用されておらず、ロジックの分岐を直接測定可能です

### 攻撃者は統計分析を使用して、ネットワーク・ノイズ (Network noise) からタイミング信号を分離できますか？
- [ ] いいえ — ネットワークのジッターやアプリケーションのノイズにより、タイミング分析は不可能です
- [ ] はい — 十分なサンプルサイズがあれば、ノイズからタイミング信号を分離できます
- [ ] はい — タイミングの差 (Delta) が十分に大きいため、統計的な正規化は必要ありません

### ログインまたはパスワードリセット機能を通じて、アカウントの列挙 (Account Enumeration) が可能ですか？
- [ ] いいえ — タイミングによってアカウントの存在を特定することはできません
- [ ] はい — タイミングの不一致により、リモートの攻撃者が有効なユーザー名やメールアドレスを列挙できます

### リソースを大量に消費する操作（ファイル処理、複雑な検索など）は、タイミングを通じてデータの存在を漏洩させていますか？
- [ ] いいえ — レコードが見つかったかどうかに関わらず、リソース消費は均一です
- [ ] はい — バックエンド処理の持続時間を測定することで、機密レコードの存在を推測できます
