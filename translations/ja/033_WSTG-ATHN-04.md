## WSTG-ATHN-04 — 認証スキーマのバイパス（Bypassing Authentication Schema）のテスト

認証スキーマのバイパスとは、有効な認証情報（Credentials）を提供せずに、保護されたリソースへのアクセスを可能にするアプリケーションロジックの欠陥を特定し、悪用することです。この脆弱性は、通常、アプリケーションがクライアント側の制御に依存している場合、すべてのリクエストに対してサーバー側の認可（Authorization）を強制できていない場合、または管理用の「バックドア（Backdoor）」やデバッグエンドポイントが本番環境に露出している場合に発生します。攻撃者は、機密性の高いURLへの強制ブラウジング（Forced Browsing）を実行したり、`authenticated=true` のようなHTTPパラメータを操作したり、ヘッダーを偽装してセッションが既に確立されているとアプリケーションに誤認させたりすることで、これらの弱点を悪用します。悪用に成功すると、認証境界が完全に崩壊し、機密データの漏洩やシステム全体の侵害（Compromise）につながる可能性があります。

### 有効なセッションなしで機密性の高い内部ページに直接アクセスできますか？
- [ ] いいえ — 直接アクセスするとログインへのリダイレクト、または 403/404 レスポンスが返される  
- [ ] はい — 一部の機密ページにアクセス可能だが、提供されるデータは限定的または非機密である  
- [ ] はい — 機密性の高い管理用またはユーザー専用ページが認証なしで完全にアクセス**可能**である（緊急）  

### アプリケーションは認証状態の判断をクライアント側のフラグやパラメータに依存していますか？
- [ ] いいえ — 認証状態はサーバー側のセッション状態によって厳密に管理されている  
- [ ] はい — クライアント側のフラグは存在するが、変更しても保護された領域へのアクセスは許可**されない**  
- [ ] はい — パラメータ（例：`is_authenticated=true`、`user_role=admin`）を操作することで認証バイパスが**可能**である  

### 特定のHTTPヘッダーの偽装または操作によって認証をバイパスできますか？
- [ ] いいえ — `X-Forwarded-For`、`Referer`、またはカスタムヘッダーなどは認証に影響を与えない  
- [ ] はい — ヘッダー（例：`X-Custom-IP-Authorization`、`X-Remote-User`）の操作が**可能**であり、不正アクセスが許可される  

### 標準的なログインフローを回避する代替パスや開発時のアーティファクトは存在しますか？
- [ ] いいえ — すべての保護されたリソースは、本番環境向けの集中化された認証チェックによって保護されている  
- [ ] はい — レガシーなエンドポイント、デバッグルート（例：`/debug/login`）、または忘れられたAPIバージョンに認証情報なしでアクセス**可能**である  

### アプリケーションは、高い権限が必要なアクションに対して再認証やセッションの検証に失敗していますか？
- [ ] いいえ — 高い権限を必要とするアクションや機密性の高いアクションには、最新かつ有効なセッション（Session Token）が必要である  
- [ ] はい — 1つのエンドポイントでバイパスが見つかると、アプリケーション全体で管理者権限のアクションを実行するために悪用**可能**である
