## WSTG-INPV-01 — 反射型クロスサイトスクリプティング (Reflected Cross Site Scripting (XSS))

反射型クロスサイトスクリプティング (Reflected Cross Site Scripting (XSS)) は、アプリケーションが十分なバリデーションやエンコーディングを行わずに、信頼できないデータをHTTPレスポンスに含めることで発生します。これにより、被害者のブラウザのコンテキストでペイロード (Payload) が実行されます。攻撃者は通常、細工されたURLやフォームなどのソーシャルエンジニアリングを通じて悪意のあるペイロードを送り込み、ユーザーセッションの乗っ取り、機密性の高いクッキーの窃取、またはユーザーに代わって不正な操作を実行します。この脆弱性は、検索パラメータ、エラーメッセージ、および入力をユーザーインターフェースに直接反映するエンドポイントで一般的によく見られます。悪用には被害者が悪意のあるリンクを操作する必要があるため、非永続的ではありますが、特定の管理者や認証済みユーザーを標的とする非常に効果的な攻撃ベクトル (Attack vector) となります。

### ユーザー提供の入力はレスポンスボディに反映されていますか？
- [ ] いいえ — 入力はユーザーに**決して**反映されません  
- [ ] はい — 入力は反映されますが、適切にエンコードされており、バイパス (Bypass) は**不可能です**  
- [ ] はい — 入力はエンコーディングやサニタイズ**なしで**反映されます  

### コンテキストに応じた出力エンコーディングは実装されていますか？
- [ ] はい — 特定の反映コンテキストに基づいて、適切なエンコーディング（HTML、属性、JavaScript）が**適用されています**  
- [ ] はい — エンコーディングは**適用されています**が、特定のコンテキストに対して不十分です（例： `<script>` タグ内でのHTMLエンコーディング）  
- [ ] いいえ — エンコーディングは**適用されていません**  

### 入力バリデーションまたはWebアプリケーションファイアウォール (WAF) のフィルタをバイパスできますか？
- [ ] いいえ — フィルタは一般的および難読化されたすべてのXSSペイロードを効果的にブロックします  
- [ ] はい — フィルタは存在しますが、文字エンコーディング、非標準タグ、またはポリグロット (Polyglots) を使用してバイパスが**可能です**  
- [ ] いいえ — フィルタやバリデーションメカニズムは導入されていません  

### 反映された入力の実行コンテキストは何ですか？
- [ ] 安全 — 入力は実行不可能な場所（例：標準の `<div>` または `<span>` タグ内）に反映されます  
- [ ] リスクあり — 入力はHTML属性内、またはタグの `src`/`href` 内に反映されます  
- [ ] 致命的 — 入力は `<script>` ブロック、イベントハンドラ、またはテンプレートリテラル内に直接反映されます  

### XSSの影響を軽減するためのモダンなブラウザセキュリティヘッダーは存在しますか？
- [ ] はい — `Content-Security-Policy` (CSP) が制限的な script-src で**有効化されています**  
- [ ] はい — `Content-Security-Policy` (CSP) は**有効化されています**が、 `unsafe-inline` または脆弱なホワイトリストが含まれています  
- [ ] いいえ — CSPやレガシーな `X-XSS-Protection` ヘッダーは存在しません
