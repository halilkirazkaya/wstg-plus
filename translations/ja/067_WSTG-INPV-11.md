## WSTG-INPV-11 — コードインジェクション (Code Injection)

コードインジェクション (Code Injection) は、アプリケーションが `eval()`、`exec()`、`system()` などの安全でない言語固有の関数を通じて、実行環境内でユーザーから提供されたコードスニペットを評価するときに発生します。この脆弱性は、基礎となるオペレーティングシステムのシェルではなく、プログラミング言語の実行コンテキスト（PHP、Python、Node.js など）を標的とする点で、コマンドインジェクション (Command Injection) とは異なります。攻撃者はこれらのポイントを悪用して、任意のロジックを実行したり、環境変数を漏洩させたり、サーバー上で完全なリモートコード実行 (Remote Code Execution / RCE) を達成したりします。ペネトレーションテスター (Pentester) は、数学的表現、サーバーサイドテンプレート、または入力が実行可能コードとして解釈される可能性のある動的な構成パラメータを処理する機能を調査する必要があります。

### 言語固有の評価関数を介して動的な入力を評価するアプリケーション機能はありますか？
- [ ] いいえ — 動的評価関数はコードベースに**存在しません**  
- [ ] はい — `eval()`、`exec()`、または `include()` などの関数が使用されていますが、ユーザー入力は**受け入れません**  
- [ ] はい — 関数が使用されており、ユーザーから提供された入力を**処理します**  

### 評価の前に入力に対して入力バリデーション (Input Validation) またはサニタイズ (Sanitization) のメカニズムが適用されていますか？
- [ ] はい — 入力は**ハードコードされたホワイトリスト**に対して厳密に検証されています  
- [ ] はい — ブラックリスト方式で入力がサニタイズされていますが、バイパスは**可能です**  
- [ ] いいえ — 入力は評価関数に直接渡され、**制御は適用されていません**  

### 実行環境は、システムレベルのアクセスを防ぐためにサンドボックス (Sandbox) 化、または制限されていますか？
- [ ] はい — 実行は、システムコールを呼び出すことが**できない**高度に制限されたサンドボックス内で行われます  
- [ ] はい — いくつかの制限が存在しますが、サンドボックスの回避 (Sandbox Escape) は**可能です**  
- [ ] いいえ — 環境は、基礎となる言語 API および OS 関数へのフルアクセスを**許可しています** *(緊急)*  

### コードインジェクションを使用して機密情報を漏洩させることができますか？
- [ ] いいえ — 実行はブラインド (Blind) 形式であり、帯域外 (Out-of-band) 通信は**不可能です**  
- [ ] はい — 環境変数やローカルファイルを HTTP/DNS リクエスト経由で**漏洩させることができます**  
- [ ] はい — 実行されたコードの結果がアプリケーションのレスポンスに**直接反映されます**  

### アプリケーションはリモートファイルインクルージョン (Remote File Inclusion / RFI) または動的なライブラリのロードを許可していますか？
- [ ] いいえ — 事前に定義されたローカルのコードパスのみが実行可能です  
- [ ] はい — アプリケーションは、リモートまたは攻撃者が制御するソースからコードをロードして実行するように**強制される可能性があります**
