## WSTG-INPV-04 — HTTPパラメータ・ポリューションのテスト

HTTPパラメータ・ポリューション (HTTP Parameter Pollution, HPP) は、アプリケーションが同じ名前を持つ複数のHTTPパラメータを受け取り、それらを一貫性のない、あるいは不安全な方法で処理した際に発生します。攻撃者は重複したパラメータを提供することで、アプリケーションの内部ロジックを操作し、Webアプリケーション・ファイアウォール (Web Application Firewalls, WAF)、入力バリデーション (Input Validation) フィルタ、またはアクセス制御 (Access Control) メカニズムを回避できる可能性があります。この脆弱性は、使用されているWebサーバーやアプリケーションフレームワークに大きく依存しており、これらは重複したパラメータの最初、最後、あるいは連結されたバージョンを選択することがあります。通常、攻撃の対象となるのは、バックエンドAPI、データベースクエリ、またはURLリダイレクトメカニズムにパラメータを渡すエンドポイントであり、その影響はクロスサイト・スクリプティング (Cross-Site Scripting, XSS) から不正なデータ流出まで多岐にわたります。

### アプリケーションは、同じ名前を持つ複数のHTTPパラメータをどのように処理しますか？
- [ ] いいえ — 重複したパラメータはサーバーによって**拒否**または無視されます  
- [ ] はい — サーバーは一貫して**最初**または**最後**の出現のみを選択します  
- [ ] はい — サーバーは値を**連結**し、インジェクションを許す可能性があります  

### HPPを利用して、WAFや入力フィルタなどのセキュリティ制御を回避できますか？
- [ ] いいえ — セキュリティ制御はパラメータの**すべて**の出現箇所を正しく検査しています  
- [ ] はい — WAFまたはフィルタは**最初**の出現のみを検査するため、悪意のある**2番目**の出現が通過してしまいます  
- [ ] はい — 連結によって、攻撃者が検知を回避するためにペイロード (Payload) を複数のパラメータに**分割**することができます  

### アプリケーションは、適切なサニタイズ (Sanitization) を行わずに、汚染されたパラメータをレスポンスに反映していますか？
- [ ] いいえ — パラメータはUIに反映される前にサニタイズまたはエンコードされます  
- [ ] はい — ポリューションによってコンテンツが反映されますが、サニタイズは**適用されています**  
- [ ] はい — ポリューションによってコンテンツが反映され、サニタイズは**適用されていません**。これによりXSSやロジックエラーが発生します  

### 内部API呼び出しやバックエンドシステムに渡されるパラメータは、ポリューションに対して脆弱ですか？
- [ ] いいえ — バックエンドのリクエストは安全で、動的ではない構築方法を使用しています  
- [ ] はい — HPPにより、攻撃者はバックエンドのリクエスト構造内のパラメータを**インジェクション**または**上書き**することができます  

### GETリクエストとPOSTリクエストでパラメータが汚染された場合、アプリケーションは異なる動作を示しますか？
- [ ] いいえ — 動作はすべてのHTTPメソッドで一貫しています  
- [ ] はい — GETパラメータのみがポリューションに対して脆弱です  
- [ ] はい — POST（ボディ）パラメータのみがポリューションに対して脆弱です
