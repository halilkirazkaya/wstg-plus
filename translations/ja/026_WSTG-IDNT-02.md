## WSTG-IDNT-02 — ユーザー登録プロセスのテスト

ユーザー登録プロセスのテストでは、新しいアイデンティティが作成されるワークフローを評価し、それが不正アクセスや自動化された悪用（Exploitation）に利用されないことを確認します。このプロセスにおける脆弱性は、本人確認（Identity Verification：メール/SMS）の欠如、ボットによる大量のアカウント作成への脆弱性、または既存のユーザー名を露呈させる詳細すぎるエラーメッセージによる情報漏洩（Information Disclosure）として現れることがよくあります。攻撃者はこれらの欠陥を悪用して、ユーザー列挙（User Enumeration）を実行したり、大規模なスパムキャンペーンを行ったり、あるいはアカウント作成段階で登録パラメータを操作して権限昇格（Privilege Escalation）を図ったりします。このテストは、アプリケーションの整合性を保護し、攻撃者が不正なアカウントや悪意のあるアカウントをシステム内に増殖させるのを防ぐために不可欠です。

### 新しいアカウントが有効になる前に本人確認が必要ですか？
- [ ] はい — 登録には、メールまたはSMSで送信される一意の期限付きトークン（Token）が必要です
- [ ] はい — 本人確認は必要ですが、トークンが脆弱（Weak）または予測可能です
- [ ] いいえ — 本人確認ステップなしで、アカウントが即座にアクティブ化されます

### 登録プロセスはユーザー列挙を防止していますか？
- [ ] はい — アプリケーションは、新規および既存のメールアドレス/ユーザー名の両方に対して同一のレスポンスを返します
- [ ] いいえ — エラーメッセージ（例：「このメールアドレスは既に使用されています」）により、攻撃者が登録済みユーザーを特定（マッピング）できてしまいます
- [ ] いいえ — サーバーレスポンスのタイミングの差（Timing Differences）により、ユーザーが存在するかどうかが判明してしまいます

### 大量登録を防止するための自動化対策（Anti-automation controls）は実装されていますか？
- [ ] はい — CAPTCHAまたはプルーフ・オブ・ワーク（Proof-of-Work）メカニズムが存在し、効果的です
- [ ] はい — 対策は存在しますが、APIエンドポイントやヘッダー操作によるバイパスが可能です
- [ ] いいえ — 登録エンドポイントにレート制限（Rate Limiting）やCAPTCHAが適用されていません

### 登録パラメータを操作して権限を昇格させることができますか？
- [ ] いいえ — ユーザーロール（User Roles）はサーバー側のロジックによって厳格に割り当てられます
- [ ] はい — リクエスト内の `role`、`admin`、または `group_id` などのパラメータを変更することで、より高いアクセス権限を取得できてしまいます

### 登録段階でパスワードポリシーは適用されていますか？
- [ ] はい — 強力なパスワード要件がサーバー側で強制（Enforced）されています
- [ ] いいえ — 脆弱なパスワード（例：「password123」）がシステムに受け入れられてしまいます
