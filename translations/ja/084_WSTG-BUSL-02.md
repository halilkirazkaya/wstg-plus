## WSTG-BUSL-02 — リクエスト偽造能力のテスト

ビジネスロジックの文脈におけるリクエストの偽造 (Request Forging) は、アプリケーションで定義されたパラメータを操作し、意図されたワークフローや認可レベル (Authorization level) 外のアクションを実行することを指します。攻撃者は、チェックアウトシステムやアカウント登録などの多段階プロセスを標的にします。これらのプロセスでは、サーバー側が信頼できるバックエンドのソースに対して再検証を怠り、クライアント側のパラメータによって状態 (State) が維持されている場合があります。隠しフィールド (Hidden fields)、JSON値、または価格、数量、内部ステータスフラグなどのRESTパラメータを変更することで、攻撃者は支払い要件をバイパスしたり、権限昇格 (Privilege Escalation) を行ったり、意図しない状態変化を引き起こしたりする可能性があります。この脆弱性は通常、トランザクション中にサーバーがクライアントからのビジネス状態の表現を暗黙的に信頼しているステートフルなWebフォームやAPIで発生します。

| 項目 | 値 |
|---|---|
| **WSTG ID** | WSTG-BUSL-02 |
| **CWE** | CWE-602 |
| **テストステータス** | 未実施 |
| **深刻度** | 高 / 緊急* |

> *偽造されたリクエストによって金銭的損失、不正な管理者操作、または完全なアカウント乗っ取り (Account Takeover) が可能になる場合、深刻度は「緊急 (Critical)」となります。

* https://owasp.org/www-project-web-security-testing-guide/latest/4-Web_Application_Security_Testing/10-Business_Logic_Testing/02-Test_Ability_to_Forge_Requests

**ツール:** `Burp Suite (Proxy, Repeater, Intruder)`, `Postman`, `OWASP ZAP`, `CyberChef`

### アプリケーションは、トランザクションパラメータをサーバー側の「信頼できる情報源 (Source of Truth)」に対して検証していますか？
- [ ] はい — すべての機密パラメータ（価格、ロール、ID）はデータベースに対して検証されており、バイパス (Bypass) は**不可能**です。
- [ ] はい — 検証は適用されていますが、パラメータ汚染 (Parameter Pollution) や代替エンコーディングを介してバイパス可能です。
- [ ] いいえ — アプリケーションはクライアント側の値を信頼して、トランザクションのコストや結果を決定しています *(緊急)*。

### ビジネス上重要な値（数量、金額など）を不正な値に変更することは可能ですか？
- [ ] いいえ — 負の値、ゼロ、または過度に大きな値はサーバーによって拒否されます。
- [ ] はい — サーバーは変更された値を受け入れますが、限られた範囲内に限定されます。
- [ ] はい — 負の値やゼロを送信することが**可能**であり、金銭的またはロジックのバイパスにつながります。

### 多段階プロセスの状態維持に、隠しフィールドやAPIパラメータが使用されていますか？
- [ ] いいえ — 状態はサーバー側のセッションまたは署名済みトークンを介して安全に維持されています。
- [ ] はい — 状態はクライアントを介して渡されますが、整合性チェック (HMAC/署名) が**有効**であり、堅牢です。
- [ ] はい — 状態はクライアントを介して渡され、整合性チェックが**無効**であるか、削除可能です。

### ワークフロー内の必須の中間ステップをスキップするリクエストを偽造することは可能ですか？
- [ ] いいえ — サーバーはすべてのトランザクションに対して厳格な操作シーケンスを強制しています。
- [ ] はい — ステップをスキップすることは**可能**ですが、最終的なアクションには依然として前のステップからの有効なデータが必要です。
- [ ] はい — 最終的な「送信」または「完了」リクエストを直接偽造して、認可や支払いのステップをバイパスすることが**可能**です。
