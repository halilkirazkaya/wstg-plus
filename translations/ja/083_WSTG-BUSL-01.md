## WSTG-BUSL-01 — ビジネスロジックのデータバリデーション (Business Logic Data Validation) のテスト

ビジネスロジックのデータバリデーションのテストは、構文的には正しいがビジネスドメインの文脈において論理的に無効なデータを、アプリケーションが特定し拒否できる能力に焦点を当てています。攻撃者は、ショッピングカートでの負の数量、未来の日付のイベントに対する過去の日付、あるいは極端に大きな整数など、予期しない値を送信することでこれらの脆弱性を悪用し、制約を回避してアプリケーションの状態を操作します。これらの脆弱性は、サーバー側のロジックがユーザー提供データの文脈的な整合性を検証できていない、多段階のワークフロー、チェックアウトプロセス、およびアカウント管理モジュールによく見られます。悪用に成功すると、金銭的損失、整合性の欠如、および制限された機能やデータへの不正アクセスにつながる可能性があります。

### アプリケーションは数値入力に対して論理的な範囲制限を強制しているか？
- [ ] はい — 不適切な場合に、負の値、ゼロ、または過度に大きな値を正しく拒否している *(最も安全)*  
- [ ] はい — 一部の無効な範囲は拒否するが、整数オーバーフロー (Integer Overflow) や境界値のバイパス (Boundary Bypass) に対して脆弱である  
- [ ] いいえ — ビジネスコンテキストに関係なく、あらゆる数値を受け入れている *(深刻)*  

### サーバー側のバリデーションチェックは、アプリケーションのすべてのレイヤーで一貫しているか？
- [ ] はい — バリデーションは API/サービス層とデータベース層の両方で一貫して適用されている  
- [ ] はい — UI/フロントエンドでバリデーションは適用されているが、直接的な API リクエストによるバイパスが可能である  
- [ ] いいえ — バリデーションが適用されていない、あるいは一貫性がなく、論理的なデータの破損を許容している  

### 多段階のワークフローにおいてデータを操作することで、ビジネスロジックを覆すことは可能か？
- [ ] いいえ — 状態はサーバー上で安全に保持されており、前のステップからのデータを改ざんすることはできない  
- [ ] はい — 初期のステップでバリデーションが行われるが、最終的な送信時にデータの整合性を再検証していない  
- [ ] はい — ステップをスキップしたり、順不同のデータを送信したりすることで、ワークフローのバイパスが可能である  

### アプリケーションは、ビジネス制約をバイパスする「エッジケース」のデータを適切に処理しているか？
- [ ] はい — 制約は堅牢であり、構文的には正しいが異例な入力（例：うるう年、タイムゾーンの変更など）を処理している  
- [ ] はい — 一部のエッジケースは処理されているが、特定の組み合わせによって予期しない動作が引き起こされる可能性がある  
- [ ] いいえ — エッジケースが直接、無料購入や不正なステータス変更などのロジックの失敗につながっている
