## WSTG-INPV-03 — HTTP動詞の改ざんのテスト (Testing for HTTP Verb Tampering)

HTTP動詞の改ざん (HTTP Verb Tampering) は、使用されるHTTPメソッドに基づいて特定のリソースへのアクセスを認可するWebサーバーやアプリケーションフレームワークの脆弱性を悪用します。攻撃者は、`GET` や `POST` といった標準的なメソッドを `HEAD`、`PUT`、`OPTIONS`、あるいはバックエンドが不整合な処理を行う可能性のある任意の非標準的な文字列に置き換えることで、セキュリティ制約のバイパス (Bypass) を試みます。この脆弱性は通常、Java EEの web.xml フィルタや .NET の認可ルールなどのセキュリティ設定において、許可されたメソッドを明示的にリストアップしているものの、他のメソッドを考慮し損ねている場合、あるいは「すべて拒否 (Deny-all)」ではなく「メソッドごとに拒否 (Deny-by-method)」というアプローチが使用されている場合に発生します。エクスプロイト (Exploit) に成功すると、管理者機能への不正アクセス、データの改ざん、またはサーバーの内部設定に関する情報漏えいを招く可能性があります。


| 項目 | 値 |
|---|---|
| **WSTG ID** | WSTG-INPV-03 |
| **CWE** | CWE-288 |
| **テストステータス** | 未実施 |
| **深刻度** | 中 / 高* |

> *管理者機能やデータの改ざん (PUT/DELETE) が認証なしで実行可能な場合、深刻度は「高」となります。

### アプリケーションは非標準または任意のHTTPメソッドに応答しますか？
- [ ] いいえ — アプリケーションは明示的に必要なメソッド以外のすべてを拒否します
- [ ] はい — アプリケーションは標準的なメソッド (OPTIONS, TRACE) に応答しますが、セキュリティをバイパスすることはできません
- [ ] はい — アプリケーションは任意の文字列 (例: FOO, TEST) を受け入れ、それらを `GET` または `POST` リクエストとして処理します

### HTTPメソッドを変更することで認証/認可をバイパスできますか？
- [ ] いいえ — HTTP動詞に関係なく、セキュリティ制御がグローバルに適用されています
- [ ] はい — 制御は導入されていますが、`HEAD` メソッドを使用することでバイパスが可能です
- [ ] はい — 代替メソッドに対してセキュリティ制御が適用されておらず、制限されたエンドポイントへの不正アクセスが可能です

### `PUT` や `DELETE` などの危険なメソッドがサーバーで有効になっていますか？
- [ ] いいえ — 危険なメソッドは無効化されているか、405 Method Not Allowed を返します
- [ ] はい — メソッドは有効ですが、有効で権限の高い認証が必要です
- [ ] はい — メソッドが有効であり、認証なしでのファイルアップロードやリソースの削除を許容します *(緊急)*

### `OPTIONS` メソッドは、許可されている動詞に関する機密情報を明らかにしますか？
- [ ] いいえ — `OPTIONS` は無効化されているか、汎用的な応答を返します
- [ ] はい — `OPTIONS` は `Allow` ヘッダーを返しますが、制限された内部メソッドは公開しません
- [ ] はい — `OPTIONS` は、さらなる攻撃に役立つ内部専用または管理者用メソッドを明らかにします

### `TRACE` メソッドが有効になっており、クロスサイトトレーシング (XST) を許容する可能性がありますか？
- [ ] いいえ — `TRACE` および `TRACK` メソッドは無効化されています
- [ ] はい — `TRACE` が有効であり、機密性の高いクッキー/トークンを含むHTTPヘッダーをそのまま返します
