## WSTG-CLNT-11 — ウェブメッセージング (Web Messaging) のテスト

ウェブメッセージング、または `postMessage` は、親ページと生成されたポップアップ、あるいは埋め込まれた iframe（インラインフレーム）などのウィンドウオブジェクト間でのクロスオリジン通信 (Cross-origin communication) を可能にします。この仕組みは現代的なウェブ機能において不可欠ですが、受信側のアプリケーションが送信者のアイデンティティを検証しなかったり、メッセージペイロード (Message Payload) を不適切に処理したりすると、重大なリスクが生じます。攻撃者は、ターゲットアプリケーションを埋め込んだ悪意のあるサイトをホストし、細工したメッセージを送信することで、意図しないアクションのトリガー、機密データの窃取、あるいは DOMベースのクロスサイトスクリプティング (DOM-based XSS) の実行といった脆弱性を悪用します。攻撃が成功するのは、メッセージリスナーが `origin` プロパティを厳格に検証していない場合や、`message.data` を危険な実行シンク (Execution Sinks) に直接渡している場合です。

### アプリケーションは、ドキュメント間通信のために `postMessage` API を利用していますか？
- [ ] いいえ — クライアント側のコードに `postMessage` リスナーは存在しません
- [ ] はい — `postMessage` は内部コンポーネント間の通信に使用されています
- [ ] はい — `postMessage` はサードパーティのドメインやウィジェットとの通信に使用されています

### 受信メッセージのオリジン (Origin) は厳格に検証されていますか？
- [ ] はい — `===` または同一比較を用いて、オリジンが厳格なホワイトリストに対してチェックされている（最も安全）
- [ ] はい — 正規表現を使用してオリジンを検証しているが、ロジックがバイパス (Bypass) 可能である (例: `^https://trusted.com`)
- [ ] いいえ — `origin` プロパティがチェックされておらず、任意のドメインからのメッセージ送信を許可している
- [ ] いいえ — 送信側の `targetOrigin` にワイルドカード `*` が使用されており、悪意のあるリスナーにデータが漏洩する可能性がある

### メッセージペイロードは、アプリケーションによって処理される前にサニタイズ (Sanitize) されていますか？
- [ ] はい — データはプレーンテキストとして扱われ、危険なシンクは使用されていない
- [ ] はい — データはパースされ（例: `JSON.parse`）、使用前に検証されており、バイパスは不可能である
- [ ] いいえ — メッセージデータが `innerHTML`、`eval()`、`setTimeout()` などの危険なシンクに直接渡されている
- [ ] いいえ — 検証なしに、メッセージデータがウィンドウの遷移や `location.href` の変更に使用されている

### 攻撃者は、悪意のあるメッセージを介して不正なアクションをトリガーしたり、データを窃取したりできますか？
- [ ] いいえ — メッセージが偽装されても、機密性の高いアクションやデータにアクセスすることはできない
- [ ] はい — 細工されたメッセージによって、機密性の高い状態変更（例：パスワード変更、プロフィール更新）がトリガーされる可能性がある
- [ ] はい — 細工されたメッセージによって、DOMベースの XSS や CSRF トークン/セッションデータの窃取が発生する可能性がある
