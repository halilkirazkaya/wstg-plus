## WSTG-ATHZ-05 — OAuthの脆弱性のテスト

OAuth (OAuth) の脆弱性は、OAuth 2.0 または OpenID Connect (OIDC) プロトコルの実装における様々な欠陥を網羅しており、多くの場合、完全なアカウントの乗っ取り (Account Takeover) や不正なデータアクセスを招きます。これらの脆弱性は通常、認可フロー中に発生し、特に `redirect_uri` の検証 (Validation)、`state` パラメータのエントロピーと検証、およびアクセストークン (Access Token) や IDトークン (ID Token) の安全な取り扱いに関連しています。攻撃者は、認可コードのインターセプト、オープンリダイレクトを介した攻撃者管理ドメインへのトークンのリダイレクト、またはクロスサイトリクエストフォージェリ (CSRF) を実行して被害者のセッションを攻撃者のアカウントに紐付けることで、これらを悪用します。OAuth は主要な認証メカニズムとして機能することが多いため、単一の誤設定がユーザー識別情報システム全体の整合性を損なう可能性があります。

### CSRF を防止するために `state` パラメータが実装および検証されていますか？
- [ ] はい — `state` は必須であり、セッションごとに一意で、サーバー側で厳密に検証されています  
- [ ] はい — `state` は存在しますが、異なるセッション間で静的または推測可能です  
- [ ] はい — `state` は存在しますが、アプリケーションはコールバック時にそれを検証しません  
- [ ] いいえ — 認可リクエストで `state` パラメータが使用されていません *(緊急)*  

### `redirect_uri` はホワイトリストに対して厳密に検証されていますか？
- [ ] はい — 事前に登録されたホワイトリスト (Whitelist) と完全に一致する場合のみ受け入れられます  
- [ ] はい — 検証は適用されていますが、パスドラバーサルやサブドメインの操作によってバイパスが可能です  
- [ ] はい — 検証は適用されていますが、パラメータ汚染 (Parameter Pollution) やフラグメントインジェクションによってバイパスが可能です  
- [ ] いいえ — アプリケーションは `redirect_uri` パラメータに任意の URL を許可しています *(緊急)*  

### フローを変更するために `response_type` を操作できますか？
- [ ] いいえ — アプリケーションは意図したフロー (例: `code`) のみを受け入れ、それ以外は拒否します  
- [ ] はい — `code` から `token` (インプリシットフロー) への切り替えが可能であり、URL 内にトークンが漏洩します  
- [ ] はい — ハイブリッドフローや許可されていない `response_type` の値が有効であり、処理されます  

### アクセストークンや認可コードが Referer ヘッダーやブラウザ履歴を介して漏洩していませんか？
- [ ] いいえ — トークンやコードは安全に処理されており、機密性の高いページでは適切な `Referrer-Policy` が使用されています  
- [ ] はい — `Referer` ヘッダーを介してトークンやコードがサードパーティドメインに漏洩しています  
- [ ] はい — 不完全なキャッシュ設定や GET リクエストにより、トークンやコードがブラウザ履歴に表示されます  

### アプリケーションは OAuth スコープに対して「最小権限 (Least Privilege)」の原則を適用していますか？
- [ ] はい — アプリケーションは機能に必要な最小限のスコープのみを要求します  
- [ ] いいえ — アプリケーションはデフォルトで過剰な権限または管理者権限のスコープを要求します  
- [ ] いいえ — ユーザーは同意フェーズで特定のスコープを確認したり、除外したりすることができません
