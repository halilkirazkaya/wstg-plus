## WSTG-BUSL-07 — アプリケーションの悪用に対する防御のテスト

アプリケーションの悪用に対する防御（Defenses against application misuse）では、意図されたビジネスロジック (Business Logic) から逸脱した異常なユーザー行動を識別し、ログに記録し、それに対応するアプリケーションの能力を評価します。従来のシグネチャベースのセキュリティとは異なり、これらの防御は、自動化された悪用を示す短時間での大量リクエスト、クレデンシャルスタッフィング (Credential Stuffing)、または連続的なリソースの列挙 (Sequential Resource Enumeration) などのパターンを検出することに焦点を当てています。攻撃者の視点からは、このテストにより、標準的なセキュリティアラートを発生させずにデータを持ち出したり、リソースを枯渇させたりすることを目的とした自動化や「ローアンドスロー」(Low and Slow) 攻撃に対するアプリケーションの耐性が決定されます。これらの防御策の実装に失敗すると、多くの場合、大規模なデータスクレイピング (Data Scraping)、アカウントテイクオーバー (Account Takeover)、または正規ではあるがリソースを大量に消費するアプリケーション機能を介したサービス拒否 (DoS) が発生します。

### 機微なエンドポイントに対して、レート制限 (Rate Limiting) またはスロットリング (Throttling) メカニズムが適用されていますか？
- [ ] はい — すべての機微なエンドポイントにおいて、厳格なレート制限が適用され、強制されている *(最も安全)*
- [ ] はい — レート制限は適用されているが、IPローテーションやヘッダーの改ざんによってバイパス (Bypass) が可能である
- [ ] はい — レート制限は認証エンドポイントにのみ適用されており、他のエンドポイントは露出したままである
- [ ] いいえ — アプリケーションのいかなる機能にも、レート制限やスロットリングは適用されていない

### アプリケーションは自動化されたインタラクションパターンを検出し、ブロックしていますか？
- [ ] はい — 行動分析 (Behavioral Analysis) または CAPTCHA が有効であり、ボットを効果的にブロックしている
- [ ] はい — アンチ自動化 (Anti-automation) は有効であるが、ヘッドレスブラウザ (Headless Browser) やセッションサイクリング (Session Cycling) を使用してバイパスが可能である
- [ ] いいえ — アプリケーションは、人間のユーザーと自動化されたスクリプトを区別することができない

### 異常な動作がログに記録され、防御的な対応が行われていますか？
- [ ] はい — 悪用によって自動ブロックがトリガーされ、セキュリティチームにアラートが送信される
- [ ] はい — 悪用は監査のためにログに記録されるが、自動化された防御アクションは実行されない
- [ ] いいえ — アプリケーションは、異常なアクティビティパターンをログに記録せず、対応も行わない

### クライアント側の識別子やヘッダーを操作することで、防御をバイパスできますか？
- [ ] いいえ — 防御はサーバー側の状態 (Server-side State) および検証済みテレメトリ (Verified Telemetry) に依存している
- [ ] はい — クッキーの削除や `User-Agent` 文字列のローテーションによって、コントロールをバイパスできる
- [ ] はい — `X-Forwarded-For` や `X-Real-IP` などのヘッダーを注入することで、コントロールをバイパスできる
