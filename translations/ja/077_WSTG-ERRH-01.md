## WSTG-ERRH-01 — 不適切なエラー処理（Improper Error Handling）のテスト

不適切なエラー処理（Improper Error Handling）は、ウェブアプリケーションがエラーレスポンスを通じて、スタックトレース（Stack Traces）、データベーススキーマ情報、内部ファイルパスなどの機密性の高い技術的詳細を公開してしまうことで発生します。攻撃者は、形式に不備のある入力（Malformed Input）の送信、存在しないリソースへのアクセス、またはサーバー側の例外の誘発によってこれらのエラーを引き起こし、アプリケーションの基盤となるアーキテクチャをマッピングして、さらなる悪用（Exploitation）のための潜在的なベクトルを特定します。このような情報漏洩は、攻撃者に正確な環境仕様を提供することになり、SQLインジェクション（SQL Injection）やパス・トラバーサル（Path Traversal）などのより深刻な攻撃の前兆となることがよくあります。安全な実装では、詳細な診断情報は安全なサーバー側のログにのみ記録し、ユーザーには一般的で分かりやすいメッセージのみを提供する必要があります。

### アプリケーションは、未処理の例外に対して汎用的なグローバルエラーハンドラーを使用していますか？
- [ ] はい — 汎用的なエラーページが**有効**であり、技術的な詳細は**一切**公開されていない  
- [ ] はい — カスタムエラーページが使用されているが、特定のヘッダーを介して漏洩の**可能性がある**  
- [ ] いいえ — デフォルトのサーバーエラーページ（例：Tomcat, IIS, Nginx）が**表示されている**  

### 形式に不備のある入力や境界値のケースを通じて、技術情報を列挙（Enumeration）することは可能ですか？
- [ ] いいえ — エラーは適切に処理されており、ロギング用の一意のリファレンスIDが表示される  
- [ ] はい — レスポンス内にスタックトレースやデータベースクエリが**露出している**  
- [ ] はい — 内部ファイルパス、環境変数、またはサーバーのバージョン文字列が**露出している**  

### APIレスポンスが詳細なエラーオブジェクトやデバッグ情報を漏洩させていませんか？
- [ ] いいえ — APIは標準化されたエラーコードとサニタイズされたJSONメッセージを返す  
- [ ] はい — APIがレスポンスボディに詳細なデバッグオブジェクトや完全な例外の詳細を返している  
- [ ] はい — JSONレスポンスの `details` フィールドや `exception` フィールドにスタックトレースが含まれている  

### エラー発生時にアプリケーションの動作（時間ベースまたはコンテンツベース）に差異が生じますか？
- [ ] いいえ — エラーの種類に関わらず、レスポンスのシグネチャとタイミングは一貫している  
- [ ] はい — 応答の差異を利用して、有効な状態と無効な状態を列挙（例：ユーザー列挙（User Enumeration））**できる**
