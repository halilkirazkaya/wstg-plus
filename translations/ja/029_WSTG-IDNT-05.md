## WSTG-IDNT-05 — 脆弱な、または強制されていないユーザー名ポリシーのテスト (Testing for Weak or Unenforced Username Policy)

脆弱な、または強制されていないユーザー名ポリシーのテストは、アカウント識別子の作成を管理するルールと、列挙 (Enumeration) やスプーフィング (Spoofing) に対する脆弱性に焦点を当てています。脆弱なポリシーは、予測可能なシーケンス、一般的な辞書単語、または内部の従業員 ID を反映した識別子を許可することが多く、これによりブルートフォース (Brute Force) やクレデンシャルスタッフィング (Credential Stuffing) 攻撃の探索空間が大幅に減少します。攻撃者の視点からは、これらのパターンの特定は大規模なアカウント発見の第一歩であり、多くの場合、登録時のエラーメッセージ、パスワードリセットの動作、または公開プロファイル内のメタデータを分析することによって達成されます。堅牢なポリシーを確保することで、攻撃者が体系的にユーザーベースをマッピングすることを防ぎ、標的型フィッシングや自動化された認証バイパス (Authentication Bypass) の試みに対する防御を容易にします。

### ユーザー名は、非常に予測可能または連続的なパターンに基づいていますか？
- [ ] いいえ — ユーザー名はランダム、ユーザー定義、または高エントロピーの文字列である  
- [ ] はい — ユーザー名は予測可能な形式 (例: `user1001`, `user1002`) に従っているが、認証は堅牢である  
- [ ] はい — ユーザー名は**厳密に連続**しているか、既知の企業形式に従っており、容易な列挙を可能にしている  

### アプリケーションは、ユーザー名に対して最小の長さと複雑さの要件を強制していますか？
- [ ] はい — 登録時に厳格な長さと文字セットのポリシーが**強制されている**  
- [ ] はい — ポリシーは存在するが、極端に短い (例: 1-2 文字) または過度に単純なユーザー名を許可している  
- [ ] いいえ — ユーザー名の長さや文字の種類に関する**ポリシーが強制されていない**  

### アプリケーションのレスポンスを通じて、有効なユーザー名を列挙できますか？
- [ ] いいえ — アプリケーションは汎用的なエラーメッセージを返し、すべての試行に対して一貫したタイミングを示す  
- [ ] はい — アプリケーションは個別のエラーメッセージ (例: 「ユーザー名は既に使用されています」) を返すが、レート制限 (Rate Limiting) が**適用されている**  
- [ ] はい — アプリケーションは登録エラー、パスワードリセット、またはタイミングの差異を通じて、有効なユーザー名を**漏洩 (Leak)** させている  

### ポリシーは、一般的または予約済みの管理者ユーザー名の登録を防止していますか？
- [ ] はい — アプリケーションは一般的な名前 (例: `admin`, `root`, `support`) やシステム予約語をブロックしている  
- [ ] いいえ — ユーザーは機密性の高い名前や管理者名でアカウントを**登録できる**  

### ユーザー名ポリシーは、すべてのエントリーポイント (API, モバイル, Web) で一貫していますか？
- [ ] はい — ポリシーはすべてのインターフェースとバージョンにわたって一貫して**適用されている**  
- [ ] いいえ — レガシーなエンドポイントや特定の API バージョンでは、同じユーザー名の制約が**強制されていない**
