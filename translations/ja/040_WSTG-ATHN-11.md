## WSTG-ATHN-11 — 多要素認証 (MFA) のテスト

多要素認証 (Multi-Factor Authentication: MFA) のテストでは、プライマリ認証情報が漏洩した場合でも、不正アクセスを防止するために設計された二次的なセキュリティレイヤーの堅牢性を評価します。攻撃者は、レガシーな API バージョン、モバイルバックエンド、パスワードリセットのワークフローなど、MFA が一貫して適用されていないエンドポイントを特定することで、MFA のバイパスを頻繁に試みます。悪用 (Exploit) の手法には、サーバーレスポンスの改ざん、短期間有効なコード (One-Time Password: OTP) に対する総当たり攻撃 (Brute Force)、または第2要素を提供せずに認証済み状態へ移行できてしまう競合状態 (Race Condition) やセッション管理 (Session Management) の不備の利用が含まれます。

### MFA はすべての認証ポータルで一貫して強制されていますか？
- [ ] はい — すべての Web、モバイル、および API ベースのログイン試行において MFA が要求される  
- [ ] はい — ただし、特定のエンドポイント（例：レガシーな `/api/v1/login` やモバイル専用ポータル）では MFA が**強制されていない**  
- [ ] いいえ — アプリケーションに MFA が**実装されていない** *(緊急)*  

### 直接的な URL 遷移やレスポンス改ざんによって MFA 検証ステップをバイパスできますか？
- [ ] いいえ — 直接の遷移やパラメータ改ざんによって認証チャレンジを**バイパスすることはできない**  
- [ ] はい — MFA チャレンジを完了せずに内部ダッシュボードの URL へ直接遷移することが**可能である**  
- [ ] はい — サーバーレスポンスを改ざん（例： `401 Unauthorized` を `200 OK` に変更）することでアクセス権を得ることが**可能である**  

### MFA コードの検証プロセスは総当たり攻撃 (Brute Force) から保護されていますか？
- [ ] はい — 複数回の OTP 試行失敗後、厳格なレート制限 (Rate Limiting) またはアカウントロックアウトが**適用される**  
- [ ] はい — レート制限は存在するが、IP ローテーションやヘッダー改ざんによってバイパスすることが**可能である**  
- [ ] いいえ — レート制限が強制されておらず、コードの自動化された総当たり攻撃が**可能である**  

### アプリケーションは MFA の遷移中に安全なセッション状態を維持していますか？
- [ ] はい — 高権限のセッショントークンは、MFA が正常に完了した**後にのみ**発行される  
- [ ] いいえ — MFA が完了する**前に**完全に機能するセッション Cookie が発行され、一部の機能へのアクセスが許可されている  
- [ ] いいえ — アプリケーションが静的な識別子や予測可能なセッション遷移を使用しており、ハイジャックが**可能である**  

### 代替の MFA 要素 (SMS、メール、バックアップコード) は悪用に対して脆弱ですか？
- [ ] いいえ — すべての要素において、安全で予測不可能かつ有効期限が制限された値が使用されている  
- [ ] はい — バックアップコードが予測可能、または列挙 (Enumerated) することが**可能である**  
- [ ] はい — 「コードを再送する」機能を悪用して SMS/メールのフラッディングを行ったり、連絡先詳細の一部を露出させたりすることが可能である
