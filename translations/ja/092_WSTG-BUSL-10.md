## WSTG-BUSL-10 — 決済機能のテスト

決済機能のテストでは、攻撃者が決済ゲートウェイをバイパスしたり、注文合計を操作したり、あるいは取引成功の信号を偽装することを可能にする業務ロジック (Business Logic) の不備を特定します。これらの脆弱性は通常、アプリケーション、クライアントのブラウザ、および Stripe、PayPal などのサードパーティ製決済プロセッサ、または内部の元帳システム間の通信に存在します。攻撃者は、転送中の価格パラメータの変更、合計コストを削減するための負の数量の送信、あるいは実際の支払いをせずに出荷を完了させるために成功したトランザクションのコールバックをリプレイしようとする可能性があります。悪用に成功すると、直接的な金銭的損失、在庫の不一致、およびアプリケーションの eコマースとしての完全性が損なわれる可能性があります。

### 処理の前に、サーバー側でトランザクション金額が検証されていますか？
- [ ] はい — 金額はサーバー側の製品データベースに対して厳密に検証されています *(最も安全)*  
- [ ] はい — 金額は検証されていますが、パラメータ汚染 (Parameter Pollution) や通貨の切り替えによってロジックのバイパスが可能です  
- [ ] いいえ — トランザクション金額はクライアント側のリクエストで変更可能であり、バックエンドでそのまま処理されます  

### 商品の数量を操作して、合計金額をゼロまたは負の値にすることは可能ですか？
- [ ] いいえ — 負またはゼロの数量は、アプリケーションの検証ロジックによって拒否されます  
- [ ] はい — カート内では負の数量が受け入れられますが、チェックアウト時に合計金額が修正されます  
- [ ] はい — 負の数量を使用して注文合計を減らしたり、クレジットを取得したりすることが可能です *(緊急)*  

### アプリケーションは、決済ゲートウェイのコールバック (Callback) またはウェブフック (Webhooks) を安全に処理していますか？
- [ ] はい — コールバックには、偽装不可能な有効な暗号署名 (Cryptographic Signature) が必要です  
- [ ] はい — 署名はチェックされていますが、シークレットが弱いか、検証をバイパスすることが可能です  
- [ ] いいえ — アプリケーションは、認証されていない、または未検証のコールバックに依存して支払いステータスを確認しています  

### アプリケーションは、チェックアウトまたは支払いフェーズにおいてレースコンディション (Race Conditions) に対して脆弱ですか？
- [ ] いいえ — トランザクションの整合性とデータベースのロックメカニズムが有効になっています  
- [ ] はい — レースコンディションは可能ですが、最終的な価格や注文完了には影響しません  
- [ ] はい — 同時実行リクエストにより、1回の支払いで複数の注文が完了したり、ギフトカードの「二重使用」が発生したりする可能性があります  

### 割引コードやロイヤリティポイントは、操作や再利用の対象となりますか？
- [ ] いいえ — コードは1回限りの使用として検証され、ロジックの制約が適用されています  
- [ ] はい — コードの再利用や対象外の商品への適用が可能ですが、影響は限定的です  
- [ ] はい — 攻撃者は、複数の矛盾するコードを適用したり、有効期限や使用制限をバイパスしたりすることが可能です
