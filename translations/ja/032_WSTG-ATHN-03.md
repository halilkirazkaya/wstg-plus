## WSTG-ATHN-03 — 脆弱なロックアウトメカニズムのテスト

アカウントロックアウトメカニズムは、指定された回数の認証試行に失敗した後にアカウントを無効化することで、総当たり攻撃（Brute-force）やクレデンシャルスタッフィング（Credential stuffing）攻撃を軽減するために設計された、多層防御（Defense-in-depth）の重要な制御策です。堅牢なロックアウトポリシーがない場合、攻撃者は自動化されたツールを使用して、複数のアカウントにわたって体系的にパスワードを推測したり（Password spraying）、成功するまで特定の高価値なアカウントを標的にしたりする可能性があります。この脆弱性は通常、ログインポータル、パスワードリセットフォーム、およびレート制限（Rate Limiting）やアカウントレベルの保護が欠如しているAPIエンドポイントで発生します。攻撃者の視点からは、ロックアウトメカニズムが脆弱または存在しない場合、無限のパスワード試行が可能になります。一方で、過度に厳格なロックアウトメカニズムは、正当なユーザーのアカウントを意図的にロックさせることにより、分散型サービス拒否攻撃（DoS）に悪用される可能性があります。

### 設定された回数の試行失敗後に、アカウントロックアウトメカニズムが強制適用されますか？
- [ ] はい — 定義された適切なしきい値（例：5〜10回）の後にアカウントがロックされる
- [ ] はい — アカウントはロックされるが、試行回数が過度に多い
- [ ] いいえ — アカウントをロックできず、無制限の総当たり攻撃（Brute-force）が可能である

### 一般的な回避手法を使用して、ロックアウトメカニズムをバイパス（Bypass）できますか？
- [ ] いいえ — ロックアウトはサーバーサイドで強制されており、クライアントサイドの変更には影響されない
- [ ] はい — IPアドレスのローテーションまたはプロキシプールの使用によりバイパスが可能である
- [ ] はい — `X-Forwarded-For` や `User-Agent` などのHTTPヘッダーを操作することでバイパスが可能である
- [ ] はい — Cookieの消去または新しいセッションの開始によりバイパスが可能である

### ロックアウトメカニズムは、アカウントの回復または有効期限の手段を提供していますか？
- [ ] はい — 一定時間経過後、またはメール認証を介してアカウントが自動的にアンロックされる
- [ ] はい — アカウントのアンロックには管理者による介入が必要である
- [ ] いいえ — ロックアウトは永続的であり、明確な回復手段がないため、DoSのリスクを高めている

### アプリケーションは、ロックアウト中に有効なユーザー名と無効なユーザー名を区別しますか？
- [ ] いいえ — 存在するユーザーと存在しないユーザーの両方に対して、アプリケーションのレスポンスが同一である
- [ ] はい — アカウントがロックされているかどうかが判明し、ユーザー名の列挙（Username Enumeration）が可能である

### ロックアウトメカニズムは、サービス拒否攻撃（DoS）に対して脆弱ですか？
- [ ] いいえ — CAPTCHAやIPベースのレート制限（Rate Limiting）などの制御により、大量のアカウントロックが防止されている
- [ ] はい — 攻撃者が誤ったパスワードを入力することで、既知の全ユーザーを体系的にロックアウトさせることが可能である
