## WSTG-INPV-06 — LDAPインジェクションのテスト (Testing for LDAP Injection)

LDAPインジェクション (LDAP Injection) は、アプリケーションがユーザーから提供されたデータを、適切なサニタイズ (Sanitization) やエスケープ (Escaping) を行わずにLDAP (Lightweight Directory Access Protocol) フィルタに組み込むことで、攻撃者がクエリロジックを操作できるようになった場合に発生します。アスタリスク、括弧、論理演算子などの特殊文字を注入することで、攻撃者は検索フィルタを修正し、認証メカニズムをバイパスしたり、ユーザー名、グループメンバーシップ、組織属性などの機密性の高いディレクトリ情報を抽出（エクスフィルトレーション）したりすることが可能になります。この脆弱性は、Active DirectoryやOpenLDAPと連携する企業ディレクトリ検索、従業員ポータル、またはシングルサインオン (SSO) システムなどの機能で一般的に見られます。攻撃者の視点では、アプリケーションによって直接的なクエリ出力が抑制されている場合、ブラインド (Blind) 手法を用いてディレクトリ構造や属性値を1ビットずつ列挙する手法が成功に繋がることが多くあります。

### アプリケーションは認証またはディレクトリ検索にLDAPを使用していますか？
- [ ] いいえ — アプリケーションアーキテクチャでLDAPは**使用されていません**  
- [ ] はい — LDAPが使用されており、すべての入力は厳密にサニタイズされるか、パラメータ化されています  
- [ ] はい — LDAPが使用されており、ユーザー入力が適切なエスケープ**なし**でフィルタに連結されています  

### ユーザー入力においてLDAPメタ文字は適切にエスケープまたはフィルタリングされていますか？
- [ ] はい — `(`, `)`, `&`, `|`, `*`, `\` などの文字は**許可されていない**か、正しくエスケープされています  
- [ ] いいえ — メタ文字を注入してLDAPフィルタの構造を変更することが**可能です**  

### インジェクションによって認証メカニズムをバイパスできますか？
- [ ] いいえ — ログインロジックはクエリ操作の**影響を受けません**  
- [ ] はい — ユーザー名またはパスワードフィールドにおける論理和 (`|`) またはワイルドカード (`*`) インジェクションを使用して、認証を**バイパスできます**  

### ディレクトリサービスからブラインド手法によるデータの抽出は可能ですか？
- [ ] いいえ — 検索結果は制限されており、タイミングや真偽値 (Boolean) の差は観察されません  
- [ ] はい — 真偽値ベースのブラインドインジェクション手法により、ディレクトリ属性を1ビットずつ**列挙できます**  
- [ ] はい — クエリ操作により、完全なディレクトリレコードがアプリケーションのレスポンスに**反映されます**  

### 二次的なアプリケーションコンポーネント（メーラー、SSOなど）は、LDAPベースの属性インジェクションに対して脆弱ですか？
- [ ] いいえ — すべてのコンポーネントでLDAP属性が安全に処理されています  
- [ ] はい — 攻撃者がインジェクションを介して自身の属性（メールアドレス、グループメンバーシップなど）を修正し、権限昇格 (Privilege Escalation) や通信の転送を**行うことができます**
