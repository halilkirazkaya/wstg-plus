## WSTG-CLNT-06 — クライアントサイドのリソース操作のテスト

クライアントサイドのリソース操作 (Client-Side Resource Manipulation) は、アプリケーションがユーザー制御の入力を許可し、ブラウザによってロードされるスクリプト、スタイルシート、または iframe などのリソースのソースやパスに影響を与える場合に発生します。URIフラグメント、クエリパラメータ、またはその他の DOMソース (DOM sources) を操作することで、攻撃者はアプリケーションに悪意のある外部コンテンツをロードさせたり、許可されていないコードを実行させたりすることができます。この脆弱性は通常、信頼された許可リスト (Allow-list) に対して十分な検証を行わずに、HTML要素の `src` や `href` 属性を動的に設定する JavaScript ロジックに見られます。攻撃者の視点では、リソースのロード先を攻撃者が制御するサーバーにリダイレクトする URL を作成することで、DOMベースのクロスサイトスクリプティング (DOM-based Cross-Site Scripting (XSS))、機密データの窃取 (Data Exfiltration)、または UIレッドレッシング (UI Redressing) を引き起こす可能性があります。

| **CWE** | CWE-99 |
| **テストステータス** | 未実施 (Not Performed) |

### アプリケーションはリソースを動的にロードまたは埋め込むためにクライアントサイドのシンク (Sinks) を使用していますか？
- [ ] いいえ — リソースは静的な HTML またはサーバーサイドのロジックのみを介してロードされます
- [ ] はい — クライアントサイドの JavaScript が `src`、`href`、または `data` などのリソース属性を動的に設定しています

### リソース URI のソースに対して検証コントロールまたは許可リストが実装されていますか？
- [ ] はい — すべての動的リソースに対して厳格な許可リストとオリジン検証が**適用されています**
- [ ] はい — 検証は存在しますが、脆弱なブラックリストや容易にバイパス可能な正規表現に依存しています
- [ ] いいえ — ユーザー制御のリソースパスに対して検証コントロールが適用されていません

### 代替プロトコルやエンコーディングを使用して URI 検証をバイパスすることは可能ですか？
- [ ] いいえ — リソースフィルタのバイパス (Bypass) は**不可能です**
- [ ] はい — `data:`、`vbscript:`、または `javascript:` などの異なるプロトコルを使用してバイパスが**可能です**
- [ ] はい — 単純な文字列一致を回避するために、エンコードされた文字やヌルバイトを使用してバイパスが**可能です**

### 攻撃者はリソースのロードを外部の信頼できないドメインにリダイレクトできますか？
- [ ] いいえ — リソースパスは同一オリジン (Same-origin) または固定のサブディレクトリに制限されています
- [ ] はい — アプリケーションは、任意の外部ドメインからスクリプトやスタイルを強制的にロード**させられる可能性があります**

### リソース操作による最大の影響は何ですか？
- [ ] 低 (Low) — 操作は画像や機密性のないテキストなどの非実行要素にのみ影響します
- [ ] 中 (Medium) — 操作により CSSインジェクション (CSS Injection) や重大な UIレッドレッシングが可能です
- [ ] 高 (High) — 操作が DOMベースの XSS または任意の JavaScript の実行につながります *(致命的)*
