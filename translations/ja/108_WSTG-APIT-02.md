## WSTG-APIT-02 — API Broken Object Level Authorization

オブジェクトレベルの認可の不備 (Broken Object Level Authorization: BOLA) は、不適切な直接オブジェクト参照 (Insecure Direct Object Reference: IDOR) とも呼ばれ、APIがIDによって特定される特定のリソースにアクセスまたは操作するための適切な権限をユーザーが持っているかどうかの検証に失敗したときに発生します。攻撃者は、リクエストパス、クエリパラメータ、またはJSONボディ内にある数値型IDやUUIDなどの識別子を体系的に列挙または推測することで、他のユーザーに属するデータにアクセスし、この脆弱性を悪用します。この脆弱性は、現代のAPIセキュリティにおいて最も一般的かつ影響力の大きい問題であり、大規模なデータ漏洩 (Mass data exfiltration)、不正な改ざん、または完全なアカウントの乗っ取り (Account takeover) につながる可能性があります。攻撃者の視点からは、オブジェクト識別子を受け取るエンドポイントを特定し、サーバー側の認可ロジックが欠如しているか、あるいはそれらのIDを操作することでバイパス (Bypass) 可能かどうかをテストすることが目的となります。

### APIリクエスト内のオブジェクト識別子は予測可能、または列挙可能ですか？
- [ ] いいえ — 識別子は長く、ランダムで、暗号学的に安全です（例：UUIDv4）  
- [ ] はい — 識別子は連番の整数です（例：`101`, `102`）  
- [ ] はい — 識別子は予測可能なパターンに従っているか、公開情報から派生しています  

### APIはすべてのリクエストに対して、オブジェクト所有権のサーバー側検証を実行していますか？
- [ ] はい — すべてのリクエストに対して認可チェックが適用されており、バイパスは**不可能です**  
- [ ] はい — チェックは適用されていますが、パラメータ汚染 (Parameter Pollution) やメソッドトンネリング (Method Tunneling) を介したバイパスが**可能です**  
- [ ] いいえ — アプリケーションは特定のソース所有権を確認せず、ユーザーが認証されていることのみに依存しています  

### 識別子を変更することで、他のユーザーのリソースにアクセスしたり変更したりすることは可能ですか？
- [ ] いいえ — 他のユーザーのリソースへの不正アクセスは**不可能です**  
- [ ] はい — 不正な**読み取り**アクセス（水平方向のBOLA: Horizontal BOLA）が**可能です**  
- [ ] はい — 不正な**改ざん**または**削除**（水平方向のBOLA）が**可能です**  

### APIは、IDを置換することで管理者レベルまたはシステムレベルのオブジェクトへのアクセスを許可していますか？
- [ ] いいえ — 管理リソースは二次的な認可レイヤーによって保護されています  
- [ ] はい — システムレベルのリソースへのアクセスまたは改ざん（垂直方向のBOLA: Vertical BOLA）が**可能です**  

### 識別子をリクエストの別の部分に移動させることで、認可チェックをバイパスすることは可能ですか？
- [ ] いいえ — パラメータの場所に依存せず、認可ロジックは一貫しています  
- [ ] はい — IDがURLパスからJSONボディやヘッダーに移動された場合にバイパスが**可能です**  
- [ ] はい — 複数のIDが提供され、サーバーが権限のない方のIDを処理した場合にバイパスが**可能です**
