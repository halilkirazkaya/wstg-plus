## WSTG-INPV-20 — マスアサインメント (Mass Assignment)

マスアサインメント (Mass Assignment) は、オーバーポスティング (Overposting) やパラメータの安全でないデシリアライゼーション (Insecure Deserialization) とも呼ばれ、Webアプリケーションが適切な属性フィルタリングを行わずに、受信したHTTPリクエストパラメータを内部オブジェクトのプロパティに自動的にバインドする場合に発生します。この脆弱性は、JSON、XML、またはフォームエンコードされたデータがアプリケーション側のデータモデルに直接デシリアライズされる、現代のMVCフレームワーク (MVC frameworks) やRESTful APIにおいて一般的です。攻撃者は、`isAdmin`、`role`、`account_balance` などのリストにない追加パラメータをリクエストに注入 (Injecting) することで、開発者がユーザーによる制御を意図していなかった機密性の高いフィールドを修正し、この動作を悪用します。攻撃の成功は通常、権限昇格 (Privilege Escalation)、不正なデータ改ざん、または重要なビジネスロジック (Business Logic) ワークフローのバイパスを招きます。

### アプリケーションは受信リクエストパラメータに対して自動バインドを使用していますか？
- [ ] いいえ — パラメータは特定の変数または安全なオブジェクトに手動でマッピングされています。  
- [ ] はい — 自動バインドが使用されていますが、厳格なホワイトリスト (White-lists) またはDTO (データ転送オブジェクト / Data Transfer Objects) が適用されています。  
- [ ] はい — 自動バインドが使用されており、機密性の高い内部属性を修正**可能**です。  

### 管理者属性または権限関連の属性を正常に注入できますか？
- [ ] いいえ — 注入されたパラメータは無視されるか、削除されるか、あるいはバリデーションエラーが発生します。  
- [ ] はい — 追加のパラメータは受け入れられますが、バックエンドのオブジェクト状態には影響し**ません**。  
- [ ] はい — `is_admin`、`role`、`permissions` などのパラメータを注入することで、権限昇格 (Privilege Escalation) に**成功**しました *(緊急 / Critical)*  

### オーバーポスティングによって、機密性の高いビジネスロジックまたは財務フィールドを修正することは可能ですか？
- [ ] いいえ — `price`、`balance`、`status` などのフィールドは、マスアサインメントから保護されています。  
- [ ] はい — JSONまたはフォームリクエストのボディに機密性の高いビジネス属性を追加することで、それらを修正**可能**です。  

### アプリケーションは、パラメータの推測を容易にする内部オブジェクト構造を公開していますか？
- [ ] いいえ — レスポンスには意図された公開フィールドのみが含まれ、ドキュメントへのアクセスは制限されています。  
- [ ] はい — GETレスポンス内で内部オブジェクトのフィールドが可視化されており、パラメータの特定が**可能**です。  
- [ ] はい — APIドキュメント (Swagger/OpenAPI) に、割り当て**可能**な内部プロパティが明示的にリストされています。
