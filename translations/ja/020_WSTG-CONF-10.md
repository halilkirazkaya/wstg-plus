## WSTG-CONF-10 — サブドメイン・テイクオーバー (Subdomain Takeover)

サブドメイン・テイクオーバー (Subdomain Takeover) は、DNSエントリ（通常は CNAME ですが、場合によっては A レコードや MX レコード）が、サードパーティのクラウドプロバイダーやホスティングサービス上の廃止された、または存在しないリソースを指している場合に発生します。攻撃者は、AWS、GitHub、Azure などのプロバイダーから返される、リソースがもはや所有されていないことを示す特定のエラーメッセージを探すことで、これらの「宙ぶらりんな (Dangling)」DNSレコードを特定します。放棄されたリソース名をプロバイダーのプラットフォームで登録することにより、攻撃者はサブドメインの完全な制御権を獲得し、悪意のあるコンテンツのホスト、ベースドメインにスコープされたセッションクッキーの窃取、および コンテンツセキュリティポリシー (CSP) や CORS 保護のバイパスが可能になります。この脆弱性は、組織の正当なドメイン構造に関連する信頼を悪用して、影響の大きいフィッシングやデータ盗難を容易にするため、特に危険です。


| 項目 | 値 |
|---|---|
| **WSTG ID** | WSTG-CONF-10 |
| **CWE** | CWE-1329 |
| **テストステータス** | 未実施 |
| **深刻度** | 高 (High) / 緊急 (Critical)* |

> *サブドメインを使用してベースドメインからセッションクッキーをハイジャックしたり、SSO/OAuth のリダイレクトをバイパスしたりできる場合、深刻度は「緊急 (Critical)」になります。

### アプリケーションはサブドメインを介してサードパーティのホスティングまたはクラウドサービスを利用していますか？
- [ ] いいえ — すべてのサブドメインは組織が管理する IP アドレス空間を指しています
- [ ] はい — サブドメインがサードパーティプロバイダー（S3、GitHub Pages、Heroku など）を指しています

### 存在しないリソースを指している「宙ぶらりんな (Dangling)」DNSレコードはありますか？
- [ ] いいえ — 特定されたすべての DNS レコードは、アクティブで管理下にあるリソースに解決されます
- [ ] はい — プロバイダー側で**もはや存在しない**リソースに対して CNAME または A レコードが存在します
- [ ] はい — DNS レコードが NXDOMAIN またはプロバイダー固有のエラーページ（例: "NoSuchBucket"）に解決されます

### 特定された未使用のリソースを正常に取得（クレーム）することは可能ですか？
- [ ] いいえ — プロバイダーが放棄された名前の取得に対する保護策を備えているか、またはアカウントの確認が必要です
- [ ] はい — リソース名は、任意のユーザーによってサードパーティプラットフォーム上で登録**可能**です

### ベースドメインは、侵害される可能性のある「Domain」スコープのクッキーを使用していますか？
- [ ] いいえ — クッキーは特定のサブドメインに厳密にスコープされているか、`Host-Only` フラグを使用しています
- [ ] はい — 機密性の高いクッキー（セッション ID、JWT など）が親ドメインにスコープされており、乗っ取られたサブドメインを介して窃取される**可能性**があります

### サブドメインを使用してセキュリティヘッダーや認証フローをバイパスできますか？
- [ ] いいえ — 特定されたサブドメインに依存するセキュリティ機能はありません
- [ ] はい — サブドメインが CSP の `script-src` または `connect-src` ディレクティブでホワイトリストに登録されています
- [ ] はい — サブドメインが OAuth または SSO 設定における有効なリダイレクト URI (Redirect URI) になっています
