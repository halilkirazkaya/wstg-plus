## WSTG-IDNT-03 — アカウントプロビジョニングプロセスのテスト (Account Provisioning Process)

アカウントプロビジョニングプロセスは、アプリケーション環境内で新しいアイデンティティ（ID）がどのように作成、検証、および権限割り当てされるかを管理するものです。攻撃者はこのワークフローを標的にし、スパムやサービス拒否 (DoS) を目的とした自動化された大量登録を実行したり、本人確認 (Identity Verification) ステップをバイパスして不正アカウントを作成したり、あるいはサインアップフェーズでパラメータを操作して自身に上位権限を付与したりします。脆弱性は、セルフサービス登録フォーム、招待制システム、または管理用プロビジョニングコンソールにおいて、不正なアカウント作成や権限昇格 (Privilege Escalation) を許容するビジネスロジックの脆弱性 (Business Logic Flaws) として現れることがよくあります。攻撃者の視点からは、プロビジョニングフローを侵害することは正当な足がかりを得ることを意味し、さらなる悪用 (Exploit)、データ流出 (Data Exfiltration)、またはソーシャルエンジニアリング攻撃 (Social Engineering Attacks) の拠点として利用される可能性があります。


| 項目 | 値 |
|---|---|
| **WSTG ID** | WSTG-IDNT-03 |
| **CWE** | CWE-285 |
| **テストステータス** | 未実施 (Not Performed) |
| **深刻度** | 中 (Medium) / 高 (High)* |

> *攻撃者が管理者アカウントをプロビジョニングできる場合、あるいはグローバルな登録制限を回避できる場合、深刻度は緊急 (Critical) となります。

### 自己登録（セルフ登録）は厳格に制御されているか、または許可されたユーザーのみに制限されているか？
- [ ] はい — 登録は**無効化**されているか、管理者による手動承認が必要である  
- [ ] はい — 登録は開放されているが、**許可された**メールドメインまたは招待トークンに制限されている  
- [ ] いいえ — 登録はすべてのユーザーに対して**開放**されており、ドメインやトークンの制限はない  

### アカウントの有効化前に、堅牢な本人確認メカニズム（メール、SMSなど）が必要か？
- [ ] はい — 確認が**必須**であり、バイパスは**不可能**である  
- [ ] はい — 確認は**必須**であるが、パラメータの改ざん (Parameter Tampering) や推測可能なトークンによってバイパスが**可能**である  
- [ ] いいえ — アカウントは申請後、確認なしに**即座に有効化**される  

### ユーザーは登録リクエスト中にロール（役割）や権限のパラメータを操作できるか？
- [ ] いいえ — ロールは**サーバーサイド (Server-side)** で割り当てられ、クライアントサイドのパラメータは存在しない  
- [ ] はい — ロールパラメータは存在するが、サーバーサイドのバリデーションにより操作は**不可能**である  
- [ ] はい — ロールや権限のパラメータ（例：`role=admin`, `is_staff=true`）を修正して上位権限を取得することが**可能**である *(緊急)*  

### 大量のアカウント作成を防ぐためのレート制限 (Rate Limiting) や自動化防止コントロールが実装されているか？
- [ ] はい — レート制限および CAPTCHA が**有効**かつ効果的に機能している  
- [ ] はい — コントロールは存在するが、ヘッダー偽装 (Header Spoofing) や CAPTCHA 解決サービスによってバイパスが**可能**である  
- [ ] いいえ — レート制限や自動化防止コントロールが**適用されていない**  

### プロビジョニングプロセスにおいて、既存のアカウントに関する情報の漏洩（ユーザー列挙）が発生するか？
- [ ] いいえ — レスポンスメッセージは、既存ユーザーと存在しないユーザーで**同一**である  
- [ ] はい — レスポンスの内容やレスポンスタイムの差異により、既存ユーザーの**列挙 (User Enumeration) が可能**である
