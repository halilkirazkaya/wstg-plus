## WSTG-SESS-07 — セッションタイムアウトのテスト (Testing Session Timeout)

セッションタイムアウト (Session Timeout) のテストでは、あらかじめ定義された無操作時間または合計時間の経過後に、アプリケーションがユーザーのセッションを効果的に終了するかどうかを評価します。この制御は、特に共有ワークステーションや、攻撃者がネットワーク傍受またはクロスサイトスクリプティング (XSS) を通じてセッション識別子を取得するシナリオにおいて、セッションハイジャック (Session Hijacking) のリスクを軽減するために不可欠です。ペネトレーションテスターは、一定期間の無操作後にトリガーされるアイドルタイムアウト (Idle Timeout) と、アクティビティに関係なくセッションの総寿命を制限する絶対タイムアウト (Absolute Timeout) の両方を分析します。攻撃者の視点からは、無期限または過度に長いセッションは、不正アクセスを維持し、再認証の必要性を回避するための絶好の機会を与えてしまいます。

### アプリケーションはアイドルセッションタイムアウトを実装していますか？
- [ ] はい — 短期間（例：15〜30分）の無操作後にセッションが期限切れになる  
- [ ] はい — セッションは期限切れになるが、タイムアウト期間が**過度に長い**（例：24時間超）  
- [ ] いいえ — 無操作中もセッションが**アクティブなまま**無期限に維持される  

### セッションタイムアウトはサーバー側 (Server-side) で強制されていますか？
- [ ] はい — クライアント側の状態に関係なく、サーバーは期限切れのトークン (Token) を含むリクエストを拒否する  
- [ ] いいえ — タイムアウトがクライアント側 (Client-side) の JavaScript または meta-refresh **のみ**で強制されている  

### アプリケーションは絶対セッションタイムアウトを実装していますか？
- [ ] はい — アクティビティに関係なく、固定された最大期間の経過後にセッションが終了する  
- [ ] いいえ — 継続的なアクティビティがある限り、セッションを無期限に**延長できる**  

### タイムアウト時にサーバー上でセッション識別子が無効化されますか？
- [ ] はい — タイムアウトの閾値に達すると、セッショントークンを**再利用できない**  
- [ ] いいえ — クライアントがログインページにリダイレクトされた後でも、サーバー上でセッショントークンが**依然として有効**である  

### 自動化されたハートビートリクエスト (Heartbeat requests) を使用して、セッションを無期限に維持できますか？
- [ ] いいえ — 絶対タイムアウトまたは二次的な制御により、セッションの無制限の延長が**防止されている**  
- [ ] はい — 定期的なバックグラウンドリクエスト（例：AJAX）により、セッションを無期限に**維持できる**
