## WSTG-SESS-01 — セッション管理スキーマのテスト (Testing for Session Management Schema)

セッション管理スキーマのテストは、予測やハイジャック (Hijacking) に対して堅牢であることを確認するために、アプリケーションがセッション識別子 (Session Identifiers) のライフサイクルと構造をどのように処理するかを分析することに焦点を当てます。攻撃者は複数のセッショントークン (Session Tokens) を取得して統計分析を行い、パターン、低いエントロピー (Entropy)、または予測可能な増分を探し、他のユーザーの有効なセッションを偽造しようとします。スキーマの脆弱性は、セッション固定攻撃 (Session Fixation) の脆弱性や、不十分なランダム値の使用として現れることが多く、通常は HTTP クッキー (Cookies)、URL パラメータ、またはカスタムヘッダーの実装で見つかります。セッションスキーマの侵害は、攻撃者が被害者の主要な認証情報を必要とせずに、認証済み状態への完全なアクセス権を得ることができる、影響度の高い攻撃です。

| **CWE** | CWE-330 |
| **テストステータス** | 未実施 |

### セッション識別子は十分に複雑でランダムですか？
- [ ] はい — トークンは統計的ランダム性テスト（高エントロピー）に合格し、バイパス (Bypass) は不可能である  
- [ ] はい — トークンは長いが、予測可能なパターンや静的なセグメントが含まれている  
- [ ] いいえ — トークンが連続的である、予測可能なタイムスタンプに基づいている、またはエントロピーが**極めて低い** *(Critical)*  

### セッション識別子はどこで送信および保存されますか？
- [ ] 識別子は `Secure` および `HttpOnly` 属性を持つクッキーに保存されている *(最も安全)*  
- [ ] 識別子はクッキーに保存されているが、`Secure` または `HttpOnly` フラグが欠如している  
- [ ] 識別子が URL パラメータまたは `GET` リクエスト経由**のみ**で送信される *(高リスク)*  

### アプリケーションは認証成功時に新しいセッション識別子を発行しますか？
- [ ] はい — ログイン直後に新しい一意のセッション ID が**生成される**  
- [ ] いいえ — ログインの前後でセッション ID が変わらない *(セッション固定攻撃が可能)*  

### セッション識別子は、再利用を防止するために特定の IP アドレスまたは User-Agent に紐付けられていますか？
- [ ] はい — クライアントのフィンガープリント (Fingerprint) が大幅に変更された場合、セッションは無効化される  
- [ ] いいえ — 追加の検証なしに、異なる IP アドレスやデバイス間でセッションを**再利用できる**  

### ログアウト後またはタイムアウト後に、セッション識別子はサーバー側で適切に無効化されますか？
- [ ] はい — ログアウト時にサーバー側のセッション状態が**完全に破棄される**  
- [ ] いいえ — クライアント側のクッキーが削除されても、サーバー上でセッションが有効なまま残る  
- [ ] いいえ — セッションが**期限切れにならない**、またはタイムアウト期間が過度に長い
