## WSTG-IDNT-01 — ロール定義のテスト

ロール定義のテストでは、アプリケーション内で定義されているさまざまなユーザーロールと権限レベルを特定および文書化し、それらが論理的に分離され、最小権限の原則 (Principle of Least Privilege) に従っていることを確認します。攻撃者はアプリケーションを分析して、高権限ロールと標準ユーザーロールをマッピングし、権限境界の曖昧さや文書化されていない管理者機能を探します。これらのロールを特定することは、権限昇格 (Privilege Escalation) パスを発見するための第一歩となります。ロール定義の不整合は、機密データや管理者機能への不正アクセスにつながることが多いためです。このフェーズは通常、初期の偵察 (Reconnaissance) およびビジネスロジックのマッピング中に行われ、ユーザープロファイル、管理ダッシュボード、および API 権限構造に焦点を当てます。

### ロールはアプリケーションまたはそのドキュメント内で明確に定義および文書化されていますか？
- [ ] はい — ロールは完全に文書化されており、**最小権限の原則**に従っています  
- [ ] はい — ロールは文書化されていますが、**権限の重複**が含まれています  
- [ ] いいえ — 正式なロールの文書化または定義が存在しません  

### 管理者機能と標準ユーザー機能の間に明確な分離はありますか？
- [ ] はい — 管理者機能は標準ユーザーのビューから**完全に隔離**されています  
- [ ] はい — 分離されていますが、管理エンドポイントが**予測可能または推測可能**です  
- [ ] いいえ — 管理機能と標準機能が同じインターフェース内に**混在**しています  

### アプリケーションはロールベースのアクセス制御 (RBAC) に中央集権的なメカニズムを使用していますか？
- [ ] はい — 中央集権的な RBAC または属性ベースのアクセス制御 (ABAC) が**実装**され、一貫して適用されています  
- [ ] はい — 分散された制御が存在しますが、モジュール間で**適用に一貫性がありません**  
- [ ] いいえ — アクセス制御ロジックが個別のページやコンポーネントに**ハードコード**されています  

### システム内に文書化されていないロールや隠しロールは存在しますか？
- [ ] いいえ — 発見されたすべてのロールは文書化された機能と一致しています  
- [ ] はい — 隠しロールまたは「シャドウ」ロール（例：`super_admin`、`support`、`test`）が**存在します**  

### 低権限のユーザーが他のユーザーの権限やロールを列挙できますか？
- [ ] いいえ — ユーザーは他のエンティティのロール/権限を表示または列挙することは**できません**  
- [ ] はい — ユーザーはプロファイルページ、メタデータ、または API レスポンスを通じてロールを列挙**できます** *(中)*
