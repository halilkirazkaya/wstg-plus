## WSTG-IDNT-04 — アカウントの列挙 (Account Enumeration) および推測可能なユーザーアカウントのテスト

アカウントの列挙（Account Enumeration）は、HTTPレスポンスの差異、エラーメッセージ、または計測可能なタイミングの差異を通じて、アプリケーションがユーザーアカウントの存在の有無を露呈させた場合に発生します。攻撃者は、ユーザー名のリストを体系的に送信することで有効なアカウントを特定し、その後のクレデンシャルスタッフィング (Credential Stuffing)、総当たり攻撃 (Brute Force)、またはソーシャルエンジニアリング (Social Engineering) に悪用します。この脆弱性は、ログインポータル、パスワードリセット機能、登録フォーム、およびユーザー固有の識別子を処理するAPIエンドポイントでよく見られます。攻撃者の視点では、列挙に成功することで、ブラインドな総当たり攻撃を既知の有効なアイデンティティに対する標的型の攻撃へと変えることができ、攻撃対象領域 (Attack Surface) を大幅に絞り込むことが可能になります。

### 認証エラーメッセージはすべてのシナリオにおいて一貫していますか？
- [ ] はい — 無効なユーザー名と無効なパスワードの両方に対して、汎用的なエラーメッセージが使用されている  
- [ ] はい — メッセージは類似しているが、句読点や大文字小文字に僅かな差異が**存在する**  
- [ ] いいえ — エラーメッセージに「ユーザーが見つかりません」や「パスワードが正しくありません」と明示的に表示される *(脆弱性あり)*  

### パスワードリセットまたは「パスワードを忘れた場合」のフローで、アカウントの存在が漏洩していませんか？
- [ ] いいえ — メールアドレスやユーザー名が存在するかどうかにかかわらず、アプリケーションは汎用的なメッセージを返す  
- [ ] はい — 対策は講じられているが、レスポンスの長さやステータスコードによるバイパスが**可能である**  
- [ ] はい — リセットメールが送信されたか、あるいはアカウントが**存在しない**かをアプリケーションが明示的に回答する  

### 登録プロセスはユーザーの列挙から保護されていますか？
- [ ] いいえ — 登録処理によって情報が漏洩せず、CAPTCHAやレート制限 (Rate Limiting) を使用して一括チェックを防止している  
- [ ] はい — 対策は講じられているが、タイミングの差異やAPIレスポンスの差異によるバイパスが**可能である**  
- [ ] はい — ユーザー名またはメールアドレスが**既に使用されている**場合、アプリケーションが即座にユーザーに通知する  

### 有効なユーザー名と無効なユーザー名を比較した際、タイミングの差異を計測できますか？
- [ ] いいえ — 定数時間比較 (Constant-time comparisons) により、アカウントの有効性にかかわらずレスポンス時間は一貫している  
- [ ] はい — タイミングの差異は存在するが、ネットワーク経由で確実に計測するには小さすぎる  
- [ ] はい — 計測可能なタイミングの差異が**存在する**（例：有効なユーザーに対してのみパスワードハッシュの計算が実行されるため）  

### アプリケーションはアカウントに対して予測可能または推測可能な命名規則を使用していますか？
- [ ] いいえ — アカウント識別子はランダム（UUID等）であるか、ユーザー定義の複雑なものである  
- [ ] はい — アカウント識別子が特定のパターン（例：`emp001`, `emp002`）に従っており、列挙が**可能である**  
- [ ] はい — 識別子が連続した整数であり、データベース全体のフル列挙が**可能である**
