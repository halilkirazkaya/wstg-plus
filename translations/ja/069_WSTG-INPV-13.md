## WSTG-INPV-13 — 書式文字列挿入 (Format String Injection)

書式文字列挿入（Format String Injection）は、Webアプリケーションがユーザー制御の入力を、C言語の `printf` ファミリや低レベルのロギングラッパーなどの可変長引数関数（Variadic function）の書式文字列引数に直接渡すことで発生します。最近のマネージドコード（Managed code）を使用したWebフレームワークでは一般的ではありませんが、この脆弱性は、低レベルのシステムライブラリと相互作用するレガシーなCGIアプリケーション、ネイティブ拡張、またはエッジケースのロギングシステムにおいて依然として重大です。攻撃者は、`%x` や `%p` などの書式指定子（Format specifier）を提供して、スタック（Stack）から機密性の高いメモリアドレスやデータを漏洩させたり、`%n` 指定子を使用して任意のメモリ書き込みを実行したりすることで、この脆弱性を悪用します。エクスプロイト（Exploit）に成功すると、通常、リモートコード実行（Remote Code Execution - RCE）を介した完全なシステム侵害、あるいは少なくともプロセスクラッシュによる影響の大きいサービス拒否攻撃（Denial-of-Service - DoS）の状態を招きます。

### アプリケーションはネイティブコードまたはレガシーなCGIインターフェースを介してユーザー入力を処理していますか？
- [ ] いいえ — アプリケーションは完全にマネージドコードのフレームワーク（JVM、CLRなど）で構築されています。
- [ ] はい — アプリケーションは JNI、ネイティブC/C++拡張、または書式文字列の脆弱性が存在する**可能性がある**レガシーなバイナリCGIを使用しています。

### ユーザー供給の入力は、書式を認識する関数の主要な引数として渡されていますか？
- [ ] いいえ — 入力はデータ引数として渡されており、書式文字列は**静的**かつ**ハードコード**されています。
- [ ] はい — ユーザー入力が書式文字列引数に直接連結されていますが、サニタイズ（Sanitization）は**適用されています**。
- [ ] はい — ユーザー入力が書式文字列として直接使用されており、サニタイズは**適用されていません**。

### 書式指定子を使用してメモリの内容を漏洩させることは可能ですか？
- [ ] いいえ — 入力はバリデーションされており、`%p`、`%x`、`%s` などの指定子は**処理されません**。
- [ ] はい — `%p` または `%x` 指定子を指定すると、スタックまたはヒープ（Heap）のメモリアドレスがレスポンスに反映されます。
- [ ] はい — 書式指定子を介して機密データ（ポインタ、スタックカナリア（Stack Canaries）など）を**抽出（Exfiltrate）できます**。

### `%n` 指定子を使用してアプリケーションをクラッシュさせたり操作したりできますか？
- [ ] いいえ — メモリ書き込みを許可する指定子は、コンパイラや環境によって**無効化**またはブロックされています。
- [ ] はい — `%s%s%s%s` または同様の文字列が提供されると、アプリケーションがクラッシュ（DoS）します。
- [ ] はい — `%n` による任意のメモリ書き込みが**可能**であり、リモートコード実行（RCE）につながる可能性があります。

### このインジェクションを通じて、最新のバイナリ保護機能（ASLR, DEP, スタックカナリア）をバイパスできますか？
- [ ] いいえ — 環境は要塞化されており、メモリ漏洩を保護機能のバイパスに利用することは**できません**。
- [ ] はい — 書式文字列挿入を使用してベースアドレスを漏洩させることができ、ASLR/DEPのバイパスが**可能です**。
