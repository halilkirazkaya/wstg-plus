## WSTG-BUSL-05 — 機能の使用回数制限のテスト (Test Number of Times a Function Can Be Used Limits)

このテストでは、アプリケーションが特定のユーザー、セッション、またはIPアドレスによって実行される特定のアクションや機能の頻度および総数に対して、ビジネスロジック (Business Logic) 上の制約を強制しているかどうかを評価します。これらの制限の実装に失敗すると、攻撃者がSMS OTPの送信、パスワードリセットメールのトリガー、割引コードの適用、または大量のデータエクスポートなどの価値の高い機能を悪用できるようになります。その結果、金銭的損失、リソース枯渇 (Resource Exhaustion)、または評判の低下を招く恐れがあります。これらの脆弱性は通常、トランザクションを伴うエンドポイントや通信機能で見られ、意図されたビジネス上の閾値を大幅に超えてアクションを繰り返す自動化スクリプトを通じて悪用されます。攻撃者の視点からは、明示的なレート制限 (Rate Limiting) や回数ベースのスロットルを欠いたワークフローにおいて、SMSポンピング (SMS Pumping)、メール爆弾、またはブルートフォース (Brute Force) を仕掛けるための格好の標的となります。

| 項目 | 値 |
|---|---|
| **WSTG ID** | WSTG-BUSL-05 |
| **CWE** | CWE-799 |
| **テストステータス** | 未実施 |
| **深刻度** | 中 / 高* |

> *深刻度は、当該機能に金銭的取引やSMSコストが関与する場合、またはシステム全体の可用性に影響を与える場合に「高（High）」となります。

### アプリケーションは機密性の高いビジネス機能に対して制限を定義し、強制していますか？
- [ ] はい — 制限は厳格に強制されており、回避は**不可能**である  
- [ ] はい — 制限は強制されているが、悪用を防ぐには閾値が高すぎる  
- [ ] いいえ — 機能の実行に対して制限が適用されていない  

### レート制限 (Rate Limiting) および使用カウンターはサーバー側 (Server-side) で強制されていますか？
- [ ] はい — 強制は厳格にサーバー側で行われており、操作することは**できない**  
- [ ] いいえ — 強制がクライアント側 (Client-side) のロジック（例：JavaScriptや隠しフィールド）に依存している  
- [ ] いいえ — どのレベルにおいても強制が存在しない  

### ヘッダーまたはセッションの操作によって使用制限を回避できますか？
- [ ] いいえ — `X-Forwarded-For`、`Origin`、またはセッションのローテーションによる回避は**不可能**である  
- [ ] はい — IPヘッダーの偽装やクッキーの消去により制限を**回避できる**  
- [ ] はい — 複数のアカウントまたはセッションを作成することで制限を**回避できる**  

### 制限を超えた場合に適切な防御的レスポンスがトリガーされますか？
- [ ] はい — アプリケーションは `429 Too Many Requests` を返し、イベントをログに記録する *(最も安全)*  
- [ ] はい — アプリケーションはエラーを返すが、潜在的な悪用をログに記録**しない**  
- [ ] いいえ — アプリケーションは処理を継続するが、出力を無視する  
- [ ] いいえ — アプリケーションはレスポンスを返さない、または負荷によりクラッシュする  

### 機能の悪用によるビジネスへの影響は重大ですか？
- [ ] いいえ — 機能の悪用によるコストやリソースへの影響は無視できる  
- [ ] はい — 悪用により、軽微なリソース消費やユーザーへの迷惑につながる**可能性がある**  
- [ ] はい — 悪用により、多大な金銭的コスト（例：SMS料金）やサービスの拒否につながる**可能性がある** *(重要)*
