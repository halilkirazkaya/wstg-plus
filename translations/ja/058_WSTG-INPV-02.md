## WSTG-INPV-02 — 蓄積型クロスサイトスクリプティング (Stored Cross Site Scripting (XSS))

蓄積型クロスサイトスクリプティング (Stored Cross Site Scripting (XSS)) は、持続型 XSS (Persistent XSS) とも呼ばれ、アプリケーションがユーザーからデータを受け取り、適切な入力バリデーション (Input Validation) やエンコーディング (Encoding) を行わずに、永続的なデータベース、ファイルシステム、またはその他のストレージメディアに保存した場合に発生します。被害者がその後、このサニタイズ (Sanitization) されていないデータを取得して表示するページに移動すると、悪意のあるスクリプトがアプリケーションのオリジン (Origin) の下で被害者のブラウザコンテキスト (Browser context) 内で実行されます。この脆弱性は、被害者が特別に細工されたリンクをクリックする必要がないため、特に危険です。影響を受けるページを閲覧するすべてのユーザーが標的となり、セッションハイジャック (Session hijacking)、不正な操作、または資格情報の窃取 (Credential harvesting) につながる可能性があります。攻撃者は通常、他のユーザーや管理者のためにデータがレンダリングされるコメント欄、ユーザープロファイル、掲示板、および管理ログを標的にします。

### 他のユーザーに後で表示するためにユーザー入力を保存するエントリポイントはありますか？
- [ ] いいえ — アプリケーションは、後にレンダリングされるユーザー制御可能な入力を保存しません
- [ ] はい — 入力は保存されますが（例：プロファイル、コメント）、HTML コンテキスト内ではレンダリングされません
- [ ] はい — 入力は保存され、他のユーザーまたは管理者にレンダリングされます

### 保存されたデータがレンダリングされる際、出力エンコーディングまたはサニタイズが適用されていますか？
- [ ] はい — コンテキストを考慮した出力エンコーディングが**適用されており**、バイパスは**不可能です** *(最も安全)*
- [ ] はい — サニタイズ（例：`DOMPurify`）が**適用されていますが**、設定ミスによりバイパスが**可能です**
- [ ] いいえ — エンコーディングやサニタイズ**なし**で、データが直接 DOM にレンダリングされます *(高)*

### 代替ペイロード (Payload) を使用して、入力バリデーションやサニタイズをバイパスすることは可能ですか？
- [ ] いいえ — フィルタは、さまざまなエンコーディング、非標準タグ、およびイベントハンドラを安全に処理します
- [ ] はい — HTML エンティティエンコーディングまたは特定のタグ（例：`<svg>`, `<img>`）を使用してバイパスが**可能です**
- [ ] はい — ポリグロットペイロード (Polyglot payloads) または Mutation-based XSS (mXSS) を介してバイパスが**可能です**

### コンテンツセキュリティポリシー (Content Security Policy (CSP)) は、二次的な防御層を提供していますか？
- [ ] はい — 厳格な CSP が**有効であり**、インラインスクリプトの実行や許可されていないソースからの読み込みを防止しています
- [ ] はい — CSP は**有効ですが**、設定ミス（例：`unsafe-inline` や広範な `script-src`）があります
- [ ] いいえ — アプリケーションで CSP が**有効になっていません**

### 「蓄積型 XSS から RCE (Remote Code Execution)」や、その他の影響の大きい攻撃チェーン (ブラインド XSS (Blind XSS)) を達成することは可能ですか？
- [ ] いいえ — 影響はクライアント側のブラウザコンテキストに限定されます
- [ ] はい — 蓄積型 XSS を使用して、内部パネルの管理者を標的（ブラインド XSS）にすることが**可能です**
- [ ] はい — 蓄積型 XSS を他の脆弱性（例：CSRF、機密データの流出）と連鎖させることが**可能です**
