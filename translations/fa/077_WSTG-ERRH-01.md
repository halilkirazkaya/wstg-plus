## WSTG-ERRH-01 — تست مدیریت نامناسب خطا (Testing for Improper Error Handling)

مدیریت نامناسب خطا (Improper Error Handling) زمانی رخ می‌دهد که یک اپلیکیشن وب، جزئیات فنی حساس — مانند ردپای پشته (Stack Trace)، اطلاعات شمای پایگاه داده (Database Schema) یا مسیرهای فایل‌های داخلی — را از طریق پاسخ‌های خطای خود فاش کند. مهاجمان با ارسال ورودی‌های بدشکل (Malformed Input)، دسترسی به منابع ناموجود یا ایجاد استثناهای سمت سرور (Server-side Exceptions) برای نقشه‌برداری از معماری زیربنایی اپلیکیشن و شناسایی بردارهای احتمالی برای بهره‌برداری‌های بیشتر، این خطاها را تحریک می‌کنند. این نشت اطلاعات اغلب به عنوان مقدمه‌ای برای حملات شدیدتر، از جمله تزریق اس‌کیوال (SQL Injection) یا پیمایش مسیر (Path Traversal)، با ارائه مشخصات دقیق محیط به مهاجم عمل می‌کند. یک پیاده‌سازی امن باید پیام‌های عمومی کاربرپسند ارائه دهد، در حالی که تشخیص‌های دقیق را فقط در لاگ‌های (Logs) امن سمت سرور ثبت کند.

### آیا اپلیکیشن از یک مدیریت‌کننده خطای عمومی و سراسری برای استثناهای مدیریت‌نشده استفاده می‌کند؟
- [ ] بله — صفحات خطای عمومی **فعال** هستند و **هیچ** جزئیات فنی را فاش نمی‌کنند  
- [ ] بله — صفحات خطای سفارشی استفاده می‌شوند اما نشت اطلاعات از طریق هدرهای خاص **امکان‌پذیر است**  
- [ ] خیر — صفحات خطای پیش‌فرض سرور (مانند Tomcat، IIS، Nginx) **قابل مشاهده** هستند  

### آیا اطلاعات فنی می‌تواند از طریق ورودی‌های بدشکل یا موارد مرزی (Boundary Cases) احصا شود؟
- [ ] خیر — خطاها به درستی مدیریت شده و شناسه‌های مرجع منحصربه‌فرد برای ثبت در لاگ تخصیص داده می‌شود  
- [ ] بله — ردپای پشته یا کوئری‌های پایگاه داده در پاسخ‌ها **فاش می‌شوند**  
- [ ] بله — مسیرهای فایل‌های داخلی، متغیرهای محیطی یا رشته‌های نسخه سرور **فاش می‌شوند**  

### آیا پاسخ‌های API در حال نشت اشیاء خطای پرجزئیات یا اطلاعات عیب‌یابی (Debug) هستند؟
- [ ] خیر — رابط‌های برنامه‌نویسی اپلیکیشن (API) کدهای خطای استاندارد و پیام‌های JSON پاکسازی‌شده را برمی‌گردانند  
- [ ] بله — APIها اشیاء دیباگ پرجزئیات یا جزئیات کامل استثنا را در بدنه پاسخ برمی‌گردانند  
- [ ] بله — ردپای پشته در فیلدهای `details` یا `exception` در پاسخ JSON گنجانده شده است  

### آیا اپلیکیشن هنگام بروز خطا رفتار متفاوتی (مبتنی بر زمان یا مبتنی بر محتوا) از خود نشان می‌دهد؟
- [ ] خیر — امضاهای پاسخ و زمان‌بندی‌ها بدون توجه به نوع خطا، ثابت هستند  
- [ ] بله — پاسخ‌های متفاوت **می‌توانند** برای احصای حالت‌های معتبر در مقابل نامعتبر (مانند User Enumeration) استفاده شوند
