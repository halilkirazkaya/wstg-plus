## WSTG-IDNT-04 — تست شمارش حساب کاربری و حساب کاربری قابل حدس (Testing for Account Enumeration and Guessable User Account)

شمارش حساب کاربری (Account Enumeration) زمانی رخ می‌دهد که یک اپلیکیشن از طریق تفاوت در پاسخ‌های HTTP، پیام‌های خطا یا تفاوت‌های زمانی (Timing Differences) قابل اندازه‌گیری، وجود یا عدم وجود یک حساب کاربری را فاش کند. مهاجمان با ارسال سیستماتیک لیست‌هایی از نام‌های کاربری برای شناسایی حساب‌های معتبر، از این رفتار سوءاستفاده می‌کنند تا در مراحل بعدی حملاتی مانند پر کردن اعتبارنامه (Credential Stuffing)، حملات جستجوی فراگیر (Brute Force) یا مهندسی اجتماعی (Social Engineering) را پیاده‌سازی کنند. این آسیب‌پذیری معمولاً در پورتال‌های ورود، قابلیت‌های بازیابی رمز عبور، فرم‌های ثبت‌نام و نقاط پایانی API که با شناسه‌های خاص کاربر سروکار دارند، ظاهر می‌شود. از دیدگاه یک مهاجم، شمارش موفقیت‌آمیز با تبدیل یک تلاش جستجوی فراگیر کورکورانه به یک بهره‌برداری (Exploit) هدفمند از هویت‌های معتبر و شناخته شده، سطح حمله (Attack Surface) را به میزان قابل توجهی کاهش می‌دهد.

### آیا پیام‌های خطای احراز هویت در تمام سناریوها یکسان هستند؟
- [ ] بله — پیام‌های خطای عمومی برای هر دو حالت نام کاربری نامعتبر و رمز عبور نامعتبر استفاده می‌شود.
- [ ] بله — پیام‌ها مشابه هستند، اما تفاوت‌های جزئی در علائم نگارشی یا بزرگی و کوچکی حروف **وجود دارد**.
- [ ] خیر — پیام‌های خطا صراحتاً عباراتی مانند "کاربر یافت نشد" یا "رمز عبور اشتباه است" را اعلام می‌کنند *(آسیب‌پذیر)*.

### آیا فرآیند بازیابی رمز عبور (Password Reset) منجر به نشت اطلاعات وجود حساب می‌شود؟
- [ ] خیر — اپلیکیشن صرف‌نظر از اینکه ایمیل/نام کاربری وجود داشته باشد یا خیر، یک پیام عمومی باز می‌گرداند.
- [ ] بله — کنترل‌هایی در نظر گرفته شده است، اما گذر (Bypass) از آن‌ها از طریق طول پاسخ یا کدهای وضعیت (Status Codes) **امکان‌پذیر است**.
- [ ] بله — اپلیکیشن صراحتاً تأیید می‌کند که آیا ایمیل بازیابی ارسال شده است یا اینکه حساب کاربری **وجود ندارد**.

### آیا فرآیند ثبت‌نام در برابر شمارش کاربر محافظت شده است؟
- [ ] خیر — ثبت‌نام اطلاعاتی نشت نمی‌دهد یا از کپچا (CAPTCHA) و محدودسازی نرخ (Rate Limiting) برای جلوگیری از بررسی‌های انبوه استفاده می‌کند.
- [ ] بله — کنترل‌هایی در نظر گرفته شده است، اما گذر از آن‌ها از طریق تفاوت‌های زمانی یا تفاوت در پاسخ‌های API **امکان‌پذیر است**.
- [ ] بله — اپلیکیشن بلافاصله به کاربر اطلاع می‌دهد که نام کاربری یا ایمیل **قبلاً گرفته شده است**.

### آیا تفاوت‌های زمانی هنگام مقایسه نام‌های کاربری معتبر و نامعتبر قابل اندازه‌گیری است؟
- [ ] خیر — زمان پاسخگویی به دلیل استفاده از مقایسه‌های زمان‌ثابت (Constant-time comparisons)، فارغ از اعتبار حساب، یکسان است.
- [ ] بله — تفاوت‌های زمانی وجود دارد اما برای اندازه‌گیری قابل اعتماد در شبکه بسیار ناچیز است.
- [ ] بله — تفاوت‌های زمانی قابل اندازه‌گیری **وجود دارد** (به‌عنوان مثال، به دلیل اجرای فرآیند درهم‌سازی رمز عبور (Password Hashing) تنها برای کاربران معتبر).

### آیا اپلیکیشن از قراردادهای نام‌گذاری پیش‌بینی‌پذیر یا قابل حدس برای حساب‌ها استفاده می‌کند؟
- [ ] خیر — شناسه‌های حساب تصادفی (UUIDs) یا تعریف شده توسط کاربر و پیچیده هستند.
- [ ] بله — شناسه‌های حساب از یک الگو پیروی می‌کنند (مانند `emp001`, `emp002`) و شمارش **امکان‌پذیر است**.
- [ ] بله — شناسه‌ها اعداد صحیح متوالی هستند که شمارش کامل پایگاه داده را **امکان‌پذیر می‌سازد**.
