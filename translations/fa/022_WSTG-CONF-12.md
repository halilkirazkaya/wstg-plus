## WSTG-CONF-12 — سیاست امنیت محتوا (Content Security Policy - CSP)

سیاست امنیت محتوا (CSP) یک مکانیزم دفاع در عمق حیاتی است که از طریق هدرهای پاسخ HTTP پیاده‌سازی می‌شود تا ریسک‌هایی مانند تزریق اسکریپت از طریق سایت (Cross-Site Scripting - XSS)، کلیک‌ربایی (Clickjacking) و حملات تزریق داده را کاهش دهد. یک CSP با پیکربندی مناسب، با تعریف منابع داینامیک مجاز برای بارگذاری، توانایی مهاجم را برای اجرای اسکریپت‌های غیرمجاز یا استخراج داده‌های حساس (Exfiltrate) به دامنه‌های خارجی محدود می‌کند. نفوذگران (Pentesters) این سیاست را برای دستورالعمل‌های (Directives) بیش از حد مجاز، استفاده از کلمات کلیدی ناامن مانند `'unsafe-inline'` و اتکا به CDNهای ناامن که ممکن است دور زدن (Bypass) را تسهیل کنند، ارزیابی می‌کنند. یک CSP ضعیف یا مفقود به طور مستقیم آسیب‌پذیری ایجاد نمی‌کند، اما با عدم ارائه لایه حفاظتی ثانویه، تأثیر حملات تزریق موفق را به طور قابل توجهی افزایش می‌دهد.

### آیا هدر سیاست امنیت محتوا (CSP) در پاسخ‌های اپلیکیشن وجود دارد؟
- [ ] بله — هدر `Content-Security-Policy` **موجود** و **اعمال شده** است  
- [ ] بله — هدر `Content-Security-Policy-Report-Only` جهت تست **موجود** است  
- [ ] خیر — هیچ هدر CSP در پاسخ‌ها **موجود نیست**  

### آیا دستورالعمل‌های `script-src` یا `default-src` به درستی محدود شده‌اند؟
- [ ] بله — دستورالعمل‌ها از لیست سفید (Whitelisting) سخت‌گیرانه، Nonceها یا Hashها استفاده می‌کنند و دور زدن (Bypass) **امکان‌پذیر نیست**  
- [ ] بله — دستورالعمل‌ها **به درستی پیکربندی شده‌اند** اما لیست سفید شامل CDNهای قابل دور زدن شناخته شده است  
- [ ] خیر — دستورالعمل‌ها از Wildcardها (`*`) یا طرح‌های `data:` استفاده می‌کنند که دور زدن را **امکان‌پذیر می‌سازد**  

### آیا این سیاست اجازه اجرای اسکریپت‌های Inline ناامن یا `eval` را می‌دهد؟
- [ ] خیر — `'unsafe-inline'` و `'unsafe-eval'` **موجود نیستند**  
- [ ] بله — `'unsafe-inline'` **موجود** است اما توسط یک `nonce` یا `hash` محافظت می‌شود  
- [ ] بله — `'unsafe-inline'` یا `'unsafe-eval'` بدون محافظت‌های اضافی **فعال هستند** *(حیاتی)*  

### آیا اپلیکیشن در برابر کلیک‌ربایی (Clickjacking) از طریق دستورالعمل `frame-ancestors` محافظت می‌شود؟
- [ ] بله — `frame-ancestors` روی `'none'` یا `'self'` تنظیم شده است  
- [ ] بله — `frame-ancestors` **فعال** است اما به دامنه‌های شخص ثالث مورد اعتماد خاصی اجازه می‌دهد  
- [ ] خیر — `frame-ancestors` **مفقود** است و به `X-Frame-Options` قدیمی متکی است یا هیچ محافظتی ندارد  

### آیا مکانیزمی برای گزارش تخلفات CSP وجود دارد؟
- [ ] بله — `report-uri` یا `report-to` **پیکربندی شده** و فعال است  
- [ ] خیر — گزارش تخلفات **غیرفعال** است یا **پیکربندی نشده** است
