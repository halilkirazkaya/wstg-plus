## WSTG-APIT-01 — شناسایی API (API Reconnaissance)

شناسایی API یک فرآیند سیستماتیک برای شناسایی و نقشه‌برداری از سطح حمله (Attack Surface) به منظور کشف نقاط انتهایی (Endpoints)، متدهای پشتیبانی شده و ساختارهای داده‌ای زیربنایی است. مهاجمان این مرحله را برای کشف APIهای مستند نشده یا سایه (Shadow APIs)، نسخه‌های منسوخ شده (Deprecated) با آسیب‌پذیری‌های قدیمی و فایل‌های مستندات فاش شده مانند تعاریف Swagger یا OpenAPI که منطق کسب‌وکار داخلی (Internal Business Logic) را آشکار می‌کنند، انجام می‌دهند. با تحلیل الگوهای قابل پیش‌بینی URL، فایل‌های جاوااسکریپت سمت کلاینت (Client-side JavaScript) و نتایج بروت فورس (Brute Force) دایرکتوری، یک نفوذگر می‌تواند نقشه جامعی از API برای شناسایی اهداف حملات شکست در کنترل دسترسی در سطح شیء (BOLA) یا تخصیص انبوه (Mass Assignment) ایجاد کند. این شناسایی معمولاً مسیرهای رایج مانند `/api/v1/`، `/swagger.json` یا `/graphql` را هدف قرار می‌دهد تا نقشه‌ راهی از قابلیت‌های بک‌اند (Backend) اپلیکیشن به دست آورد.


| فیلد | مقدار |
|---|---|
| **شناسه WSTG** | WSTG-APIT-01 |
| **CWE** | CWE-200 |
| **وضعیت تست** | انجام نشده |

### آیا فایل‌های مستندات API (مانند Swagger، OpenAPI، WSDL) به صورت عمومی در دسترس هستند؟
- [ ] خیر — مستندات API در دسترس نیست یا توسط احراز هویت (Authentication) محدود شده است  
- [ ] بله — مستندات در دسترس است اما نیاز به اعتبارنامه‌های (Credentials) معتبر دارد  
- [ ] بله — مستندات API (مانند `/swagger-ui.html`) به صورت عمومی و بدون احراز هویت در دسترس است  

### آیا نقاط انتهایی مستند نشده یا "سایه" از طریق فازینگ (Fuzzing) قابل شناسایی هستند؟
- [ ] خیر — فازینگ دایرکتوری و نقاط انتهایی هیچ مسیر مستند نشده‌ای را نشان نداد  
- [ ] بله — نقاط انتهایی مستند نشده پیدا شدند اما بدون مجوز (Authorization) قابل دسترسی نیستند  
- [ ] بله — نقاط انتهایی مستند نشده پیدا شدند و بدون مجوز معتبر قابل دسترسی هستند  

### آیا اپلیکیشن نسخه‌های متعددی از API (مانند /v1/، /v2/، /beta/) را فاش می‌کند؟
- [ ] خیر — فقط نسخه فعلی و ایمن‌سازی شده API در دسترس است  
- [ ] بله — نسخه‌های قدیمی وجود دارند اما همان کنترل‌های امنیتی نسخه فعلی را پیاده‌سازی کرده‌اند  
- [ ] بله — نسخه‌های قدیمی (مانند `/v1/`) در دسترس هستند و فاقد کنترل‌های امنیتی نسخه‌های جدیدتر می‌باشند  

### آیا منابع سمت کلاینت (JavaScript/Mobile Apps) ساختار نقاط انتهایی API یا کلیدها را فاش می‌کنند؟
- [ ] خیر — کد سمت کلاینت حاوی مسیرهای API هاردکد شده (Hardcoded) یا کلیدهای حساس نیست  
- [ ] بله — کد سمت کلاینت شامل نقشه‌های نقاط انتهایی API است اما کلید حساس ندارد  
- [ ] بله — کلیدهای API، اسرار (Secrets) یا URLهای نقاط انتهایی داخلی در منابع سمت کلاینت هاردکد شده‌اند  

### آیا پارامترها یا هدرهای مخفی در نقاط انتهایی شناخته شده قابل شناسایی هستند؟
- [ ] خیر — فازینگ پارامتر (مثلاً از طریق `Arjun`) ورودی‌های مخفی را فاش نکرد  
- [ ] بله — پارامترهای مخفی (مانند `debug=true` یا `admin=1`) کشف شدند اما کارایی ندارند  
- [ ] بله — پارامترهای مخفی کشف شدند و می‌توان از آن‌ها برای تغییر رفتار اپلیکیشن استفاده کرد
