## WSTG-CLNT-01 — تست تزریق اسکریپت از طریق وب‌گاه مبتنی بر DOM

تزریق اسکریپت از طریق وب‌گاه مبتنی بر DOM (DOM-based Cross-Site Scripting یا DOM XSS) زمانی رخ می‌دهد که یک اپلیکیشن حاوی کدهای جاوااسکریپت سمت کلاینت باشد که داده‌ها را از یک منبع غیرقابل اعتماد (Untrusted Source) به شکلی ناامن پردازش می‌کند؛ این کار معمولاً از طریق بازنویسی داده‌ها در مدل شیء سند (Document Object Model یا DOM) به‌وسیله یک سینک (Sink) خطرناک انجام می‌شود. برخلاف XSS منعکس‌شده (Reflected) یا ذخیره‌شده (Stored)، این آسیب‌پذیری به‌طور کامل در کدهای سمت کلاینت وجود دارد؛ پی‌لود (Payload) اغلب اصلاً به سرور ارسال نمی‌شود، زیرا توسط سینک‌هایی مانند `innerHTML`، `eval()` یا `document.write()` پردازش می‌گردد. مهاجمان با طراحی URLهایی حاوی قطعات (Fragments) یا پارامترهای مخرب که هنگام بارگذاری توسط مرورگر قربانی، جاوااسکریپت دلخواه را در کانتکست نشست کاربر اجرا می‌کنند، از این نقص سوءاستفاده می‌کنند. این موضوع می‌تواند منجر به استخراج توکن‌های نشست (Session tokens)، سرقت داده‌های حساس و انجام اقدامات غیرمجاز از طرف کاربر احرازهویت‌ شده شود.

### آیا از منابع غیرقابل اعتماد برای دریافت داده‌های تحت کنترل کاربر در جاوااسکریپت استفاده می‌شود؟
- [ ] خیر — جاوااسکریپت از `location.hash`، `location.search` یا `document.referrer` استفاده نمی‌کند.
- [ ] بله — منابع غیرقابل اعتماد استفاده می‌شوند اما داده‌ها به هیچ سینک اجرایی (Execution Sink) منتقل نمی‌شوند.
- [ ] بله — منابع غیرقابل اعتماد استفاده می‌شوند و داده‌ها مستقیماً به منطق سمت کلاینت وارد می‌شوند.

### آیا داده‌های تحت کنترل کاربر به سینک‌های خطرناک DOM منتقل می‌شوند؟
- [ ] خیر — سینک‌هایی مانند `innerHTML`، `outerHTML`، `document.write()` یا `eval()` استفاده نمی‌شوند.
- [ ] بله — سینک‌های خطرناک وجود دارند اما داده‌ها به‌طور دقیق با استفاده از یک کتابخانه امن مانند DOMPurify پاک‌سازی (Sanitized) می‌شوند.
- [ ] بله — سینک‌های خطرناک وجود دارند و داده‌ها با فیلترینگ مبتنی بر عبارت باقاعده (Regex) سفارشی یا ناکافی پردازش می‌شوند.
- [ ] بله — سینک‌های خطرناک وجود دارند و داده‌ها بدون هیچ‌گونه پاک‌سازی یا کدگذاری (Encoding) منتقل می‌شوند *(بحرانی)*.

### آیا سینک اجرایی از طریق یک پی‌لود مبتنی بر URL قابل دسترسی است؟
- [ ] خیر — جریان از منبع به سینک (Source-to-sink) از طریق ورودی خارجی قابل تحریک نیست.
- [ ] بله — جریان امکان‌پذیر است اما به تعامل پیچیده کاربر نیاز دارد.
- [ ] بله — جریان امکان‌پذیر است و می‌تواند از طریق یک قطعه URL ساده یا پارامتر پرس‌وجو (Query Parameter) تحریک شود.

### آیا هدرهای امنیتی مدرن یا اقدامات تقابلی مبتنی بر مرورگر، ریسک را خنثی می‌کنند؟
- [ ] بله — یک سیاست امنیت محتوا (Content-Security-Policy یا CSP) سخت‌گیرانه با هدرهای `script-src` و `trusted-types` فعال است و از اجرا جلوگیری می‌کند.
- [ ] بله — یک CSP وجود دارد اما `unsafe-inline` یا هش‌های ضعیف، دور زدن (Bypass) آن را ممکن می‌سازد.
- [ ] خیر — هیچ CSP برای کاهش ریسک اجرای مبتنی بر DOM وجود ندارد.

### آیا اپلیکیشن از فریم‌ورک‌های سمت کلاینت با حفاظت‌های داخلی استفاده می‌کند؟
- [ ] بله — فریم‌ورک (مانند Angular یا React) به‌طور خودکار داده‌ها را کدگذاری می‌کند و دور زدن آن امکان‌پذیر نیست.
- [ ] بله — فریم‌ورک استفاده شده است اما "راه‌های فرار" (Escape hatches) مانند `dangerouslySetInnerHTML` فعال هستند.
- [ ] خیر — اپلیکیشن از جاوااسکریپت خام (Vanilla JavaScript) یا کتابخانه‌های قدیمی (مانند نسخه‌های قدیمی jQuery) بدون کدگذاری خودکار استفاده می‌کند.
