## WSTG-CLNT-11 — تست پیام‌رسانی وب (Web Messaging)

پیام‌رسانی وب (Web Messaging) یا `postMessage` ارتباط بین مبدائی (Cross-origin communication) را بین اشیاء پنجره (window objects)، مانند یک صفحه والد و یک پاپ‌آپ ایجاد شده یا یک iframe تعبیه‌شده، فراهم می‌کند. این مکانیسم برای عملکردهای مدرن وب حیاتی است اما اگر اپلیکیشنِ دریافت‌کننده نتواند هویت فرستنده را تأیید کند یا با محموله (Payload) پیام به‌درستی رفتار نکند، ریسک قابل توجهی ایجاد می‌کند. مهاجمان با میزبانی یک سایت مخرب که اپلیکیشن هدف را در خود جای داده و ارسال پیام‌های دست‌کاری شده برای تحریک اقدامات ناخواسته، استخراج داده‌های حساس یا اجرای اسکریپت‌نویسی بین‌سایتی مبتنی بر DOM (DOM-based XSS) از این آسیب‌پذیری‌ها سوءاستفاده می‌کنند. بهره‌برداری موفق زمانی رخ می‌دهد که شنودکننده‌های پیام (message listeners) ویژگی `origin` را به دقت اعتبارسنجی نکنند یا `message.data` را مستقیماً به سینک‌های اجرایی (execution sinks) خطرناک منتقل کنند.

### آیا اپلیکیشن از API مربوط به `postMessage` برای ارتباط بین سندی (Cross-document communication) استفاده می‌کند؟
- [ ] خیر — شنودکننده‌های `postMessage` در کد سمت کلاینت (Client-side) وجود ندارند  
- [ ] بله — از `postMessage` برای ارتباطات داخلی اجزا استفاده می‌شود  
- [ ] بله — از `postMessage` برای ارتباط با دامنه‌ها یا ویجت‌های شخص ثالث (Third-party) استفاده می‌شود  

### آیا مبدأ (Origin) پیام‌های ورودی به دقت اعتبارسنجی می‌شود؟
- [ ] بله — مبدأ با استفاده از `===` یا مقایسه یکسان با یک لیست مجاز (Whitelist) دقیق بررسی می‌شود *(امن‌ترین حالت)*  
- [ ] بله — مبدأ با استفاده از یک عبارت باقاعده (Regex) اعتبارسنجی می‌شود، اما منطق آن مستعد دور زدن (Bypass) است (مثلاً `^https://trusted.com`)  
- [ ] خیر — ویژگی `origin` بررسی نمی‌شود، که به هر دامنه‌ای اجازه ارسال پیام می‌دهد  
- [ ] خیر — از یک علامت ویلدکارد (Wildcard) `*` در `targetOrigin` فرستنده استفاده شده است که بالقوه منجر به نشت داده‌ها به شنودکننده‌های مخرب می‌شود  

### آیا محموله (Payload) پیام قبل از پردازش توسط اپلیکیشن پاک‌سازی (Sanitize) می‌شود؟
- [ ] بله — با داده‌ها به عنوان متن ساده (Plain text) رفتار می‌شود و از هیچ سینک (Sink) خطرناکی استفاده نمی‌شود  
- [ ] بله — داده‌ها پارس (Parse) شده (مثلاً `JSON.parse`) و قبل از استفاده اعتبارسنجی می‌شوند و بای‌پس (Bypass) امکان‌پذیر نیست  
- [ ] خیر — داده‌های پیام مستقیماً به سینک‌های خطرناکی مانند `innerHTML` ،`eval()` یا `setTimeout()` منتقل می‌شوند  
- [ ] خیر — از داده‌های پیام برای پیمایش پنجره یا تغییر `location.href` بدون اعتبارسنجی استفاده می‌شود  

### آیا یک مهاجم می‌تواند از طریق یک پیام مخرب، اقدامات غیرمجاز را تحریک یا داده‌ها را استخراج کند؟
- [ ] خیر — حتی با یک پیام جعل شده (Spoofed message)، هیچ اقدام یا داده حساسی قابل دسترسی نیست  
- [ ] بله — یک پیام دست‌کاری شده می‌تواند منجر به تغییرات وضعیت حساس شود (مثلاً تغییر رمز عبور، بروزرسانی پروفایل)  
- [ ] بله — یک پیام دست‌کاری شده می‌تواند منجر به XSS مبتنی بر DOM یا استخراج توکن‌های جعل درخواست بین‌سایتی (CSRF) یا داده‌های نشست (Session data) شود
