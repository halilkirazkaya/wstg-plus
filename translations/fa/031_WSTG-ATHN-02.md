## WSTG-ATHN-02 — تست برای اعتبارات پیش‌فرض (Testing for Default Credentials)

تست برای اعتبارات پیش‌فرض (Default Credentials) شامل شناسایی رابط‌های مدیریتی (Administrative Interfaces)، سرویس‌ها یا اجزای برنامه‌ای است که همچنان از نام‌های کاربری و رمزهای عبور تنظیم شده در کارخانه یا ارائه‌شده توسط سازنده استفاده می‌کنند. این آسیب‌پذیری هدفی با اولویت بالا برای مهاجمان محسوب می‌شود، زیرا اغلب مسیری فوری برای نفوذ کامل به سیستم (System Compromise)، دسترسی مدیریتی یا اجرای کد از راه دور (Remote Code Execution - RCE) با کمترین تلاش فراهم می‌کند. این وضعیت معمولاً در نرم‌افزارهای تجاری آماده (Off-the-shelf)، پلتفرم‌های مدیریت محتوا (CMS)، ابزارهای مدیریت پایگاه داده و سخت‌افزارهای یکپارچه مانند دستگاه‌های اینترنت اشیا (IoT) یا تجهیزات شبکه رخ می‌دهد. بهره‌جویی (Exploitation) معمولاً از طریق مطابقت دادن نسخه‌های نرم‌افزاری شناسایی شده با پایگاه‌های داده عمومی رمزهای عبور پیش‌فرض یا استفاده از ابزارهای خودکار پر کردن اعتبارات (Credential Stuffing) در مراحل اولیه شناسایی (Reconnaissance) انجام می‌شود.

### آیا رابط‌های مدیریتی یا اجرایی در معرض اینترنت یا بخش‌های شبکه غیرقابل اعتماد قرار دارند؟
- [ ] خیر — هیچ رابط مدیریتی در دسترس نیست  
- [ ] بله — رابط‌ها موجود هستند اما توسط کنترل‌های سطح شبکه محدود شده‌اند (مانند VPN یا لیست مجاز IP)  
- [ ] بله — رابط‌ها **به صورت عمومی** در دسترس هستند و نیاز به احراز هویت (Authentication) دارند  

### آیا اعتبارات پیش‌فرض ارائه‌شده توسط سازنده برای تمامی اجزای شناسایی شده تغییر یافته‌اند؟
- [ ] بله — تمامی اعتبارات پیش‌فرض در تمام اجزا **تغییر یافته‌اند** (امن‌ترین حالت)  
- [ ] بله — اعتبارات برای برنامه اصلی **تغییر یافته‌اند**، اما اجزای ثانویه (مانند CMS یا پایگاه داده) همچنان پیش‌فرض باقی مانده‌اند  
- [ ] خیر — اعتبارات پیش‌فرض در یک یا چند رابط شناسایی شده **فعال هستند** (بحرانی)  

### آیا برنامه در اولین ورود برای حساب‌های جدید یا مدیریتی، تغییر رمز عبور را اجبار می‌کند؟
- [ ] بله — تغییر رمز عبور قبل از انجام هرگونه اقدام مدیریتی **الزامی است**  
- [ ] خیر — برنامه **اجازه می‌دهد** از اعتبارات پیش‌فرض به صورت نامحدود استفاده شود  

### آیا مکانیزم‌های قفل حساب یا کنترل‌های محدودکننده نرخ برای جلوگیری از تست خودکار اعتبارات پیش‌فرض فعال هستند؟
- [ ] بله — محدودیت نرخ (Rate Limiting) تهاجمی یا قفل حساب (Lockout) **اعمال شده است**  
- [ ] بله — کنترل‌ها موجود هستند اما دور زدن آن‌ها از طریق دستکاری هدر یا چرخش IP **امکان‌پذیر است**  
- [ ] خیر — هیچ مکانیزم محدودکننده نرخ یا قفل حسابی **فعال نیست**  

### آیا پلاگین‌های شخص ثالث، میان‌افزارها یا ابزارهای مدیریتی (مانند phpMyAdmin یا Tomcat Manager) از اعتبارات منحصربه‌فرد استفاده می‌کنند؟
- [ ] خیر — اجزای شخص ثالث در محیط وجود ندارند  
- [ ] بله — تمامی اجزای شخص ثالث از اعتبارات منحصربه‌فرد و غیر پیش‌فرض استفاده می‌کنند  
- [ ] خیر — حداقل یک جزء شخص ثالث از طریق اعتبارات پیش‌فرض شناخته شده **در دسترس است**
