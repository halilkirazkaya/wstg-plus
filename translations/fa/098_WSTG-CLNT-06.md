## WSTG-CLNT-06 — تست دست‌کاری منابع سمت کلاینت (Client-Side Resource Manipulation)

دست‌کاری منابع سمت کلاینت (Client-side resource manipulation) زمانی رخ می‌دهد که یک اپلیکیشن اجازه می‌دهد ورودی‌های تحت کنترل کاربر بر منبع (Source) یا مسیر منابع بارگذاری شده توسط مرورگر، مانند اسکریپت‌ها (Scripts)، استایل‌شیت‌ها (Stylesheets) یا آی‌فریم‌ها (Iframes) تأثیر بگذارند. با دست‌کاری فراگمنت‌های URI (URI fragments)، پارامترهای پرس‌وجو (Query parameters) یا سایر منابع DOM، یک نفوذگر می‌تواند اپلیکیشن را مجبور به بارگذاری محتوای مخرب خارجی یا اجرای کد غیرمجاز نماید. این آسیب‌پذیری معمولاً در منطق جاوااسکریپت (JavaScript) یافت می‌شود که به صورت پویا ویژگی‌های (Attributes) `src` یا `href` در عناصر HTML را بدون اعتبارسنجی کافی در برابر یک لیست مجاز (Allow-list) معتبر مقداردهی می‌کند. از دیدگاه یک نفوذگر، این مورد با ایجاد یک URL که بارگذاری منابع را به سرور تحت کنترل نفوذگر هدایت می‌کند، اکسپلویت (Exploit) می‌شود که به طور بالقوه منجر به اسکریپت‌نویسی بین‌سایتی مبتنی بر DOM (DOM-based XSS)، نشت داده‌های حساس (Sensitive data exfiltration) یا جعل رابط کاربری (UI redressing) می‌گردد.

### آیا اپلیکیشن از سینک‌های سمت کلاینت (Client-side sinks) برای بارگذاری پویا یا جاسازی منابع استفاده می‌کند؟
- [ ] خیر — منابع تنها از طریق HTML استاتیک یا منطق سمت سرور بارگذاری می‌شوند  
- [ ] بله — جاوااسکریپت سمت کلاینت به صورت پویا ویژگی‌های منبع مانند `src` ،`href` یا `data` را مقداردهی می‌کند  

### آیا کنترل‌های اعتبارسنجی یا لیست‌های مجاز برای منابع URI پیاده‌سازی شده‌اند؟
- [ ] بله — لیست‌های مجاز سخت‌گیرانه و اعتبارسنجی مبدأ (Origin validation) روی تمام منابع پویا **اعمال شده‌اند**  
- [ ] بله — اعتبارسنجی وجود دارد اما به لیست‌های سیاه (Blacklists) ضعیف یا عبارات منظم (Regex) که به راحتی دور زده می‌شوند، متکی است  
- [ ] خیر — هیچ کنترل اعتبارسنجی روی مسیرهای منبع تحت کنترل کاربر اعمال نشده است  

### آیا دور زدن اعتبارسنجی URI با استفاده از پروتکل‌های جایگزین یا کدگذاری (Encoding) امکان‌پذیر است؟
- [ ] خیر — دور زدن (Bypass) فیلترهای منبع **امکان‌پذیر نیست**  
- [ ] بله — دور زدن با استفاده از پروتکل‌های مختلف مانند `:data` ،`:vbscript` یا `:javascript` **امکان‌پذیر است**  
- [ ] بله — دور زدن با استفاده از کاراکترهای کدگذاری شده یا نال‌بایت‌ها (Null bytes) برای دور زدن تطبیق‌های رشته‌ای ساده **امکان‌پذیر است**  

### آیا نفوذگر می‌تواند بارگذاری منبع را به یک دامنه خارجی و نامعتبر هدایت کند؟
- [ ] خیر — مسیرهای منبع به همان مبدأ (Same origin) یا یک زیرشاخه ثابت محدود شده‌اند  
- [ ] بله — اپلیکیشن **می‌تواند** مجبور به بارگذاری اسکریپت‌ها یا استایل‌ها از یک دامنه خارجی دلخواه شود  

### حداکثر تأثیر (Impact) دست‌کاری منابع چیست؟
- [ ] پایین — دست‌کاری تنها بر عناصر غیرقابل اجرا مانند تصاویر یا متن‌های غیرحساس تأثیر می‌گذارد  
- [ ] متوسط — دست‌کاری اجازه تزریق CSS (CSS Injection) یا جعل رابط کاربری قابل توجه را می‌دهد  
- [ ] بالا — دست‌کاری منجر به DOM-based XSS یا اجرای جاوااسکریپت دلخواه می‌شود *(بحرانی)*
