## WSTG-INPV-19 — تست جعل درخواست سمت سرور (SSRF)

جعل درخواست سمت سرور (Server-Side Request Forgery - SSRF) زمانی رخ می‌دهد که یک مهاجم بتواند یک اپلیکیشن وب را تحت تأثیر قرار دهد تا درخواست‌های غیرمجازی را از سمت سرور به منابع داخلی یا خارجی ارسال کند. مهاجم با دستکاری پارامترهایی که انتظار URL، نام میزبان (Hostname) یا آدرس‌های IP را دارند، می‌تواند کنترل‌های سطح شبکه مانند فایروال‌ها (Firewalls) و لیست‌های کنترل دسترسی (ACLs) را دور بزند تا بخش‌های داخلی شبکه را پویش کند، به سرویس‌های متادیتای ابری (Cloud Metadata Services - IMDS) دسترسی پیدا کند یا با APIها و پایگاه‌های داده داخلی آسیب‌پذیر تعامل داشته باشد. این آسیب‌پذیری معمولاً در قابلیت‌هایی مانند واکشی تصویر، تولید PDF، وبهوک‌ها (Webhooks) یا آپلود فایل از طریق URL ظاهر می‌شود. از دیدگاه یک مهاجم، SSRF یک ابزار قدرتمند برای جابجایی (Pivot) به محیط‌های ایزوله، استخراج داده‌های حساس پیکربندی یا پتانسیل دستیابی به اجرای کد از راه دور (Remote Code Execution - RCE) از طریق تعامل با سرویس‌های داخلی مانند Redis یا Memcached است.

### آیا اپلیکیشن URLها یا نام‌های میزبان ارائه شده توسط کاربر را پردازش می‌کند؟
- [ ] خیر — هیچ قابلیتی URLها یا نام‌های میزبان خارجی را نمی‌پذیرد  
- [ ] بله — قابلیت وجود دارد اما ورودی در برابر یک لیست مجاز (Allow-list) **به‌طور دقیق اعتبارسنجی می‌شود**  
- [ ] بله — قابلیت وجود دارد و URLهای ارائه شده توسط کاربر را می‌پذیرد  

### آیا کنترل‌های اعتبارسنجی یا لیست‌های سیاه بر روی مقصد درخواستی اعمال می‌شوند؟
- [ ] بله — یک **لیست مجاز (Allow-list)** سخت‌گیرانه از دامنه‌ها/IPهای قابل اعتماد اجباری است  
- [ ] بله — از یک **لیست سیاه (Deny-list)** استفاده می‌شود (مثلاً مسدود کردن `127.0.0.1` یا `169.254.169.254`)  
- [ ] خیر — **هیچ اعتبارسنجی** یا فیلترینگی روی ورودی اعمال نمی‌شود  

### آیا امکان دور زدن فیلترها از طریق کدگذاری، تغییر مسیر یا DNS Rebinding وجود دارد؟
- [ ] خیر — فیلترها مستحکم هستند و تغییر مسیرها/DNS Resolution را به‌صورت امن مدیریت می‌کنند  
- [ ] بله — دور زدن فیلتر با استفاده از کدگذاری URL (URL Encoding) یا فرمت‌های جایگزین IP (هگز، اکتال) **امکان‌پذیر است**  
- [ ] بله — دور زدن فیلتر از طریق تغییر مسیرهای HTTP 3xx (HTTP Redirects) به اهداف داخلی **امکان‌پذیر است**  
- [ ] بله — دور زدن فیلتر از طریق حملات DNS Rebinding برای اشاره به IPهای داخلی **امکان‌پذیر است**  

### آیا اپلیکیشن می‌تواند به سرویس‌های متادیتای داخلی یا رابط‌های محلی دسترسی پیدا کند؟
- [ ] خیر — شبکه داخلی و نقاط انتهایی متادیتای ابری **قابل دسترس نیستند**  
- [ ] بله — دسترسی به localhost (`127.0.0.1`) یا سرویس‌های Loopback **امکان‌پذیر است**  
- [ ] بله — دسترسی به سرویس‌های متادیتای ابری (مانند AWS/GCP/Azure IMDS) **امکان‌پذیر است** *(حیاتی)*  

### ماهیت پاسخ SSRF چیست؟
- [ ] کور (Blind) — هیچ پاسخ یا متادیتایی به کاربر بازگردانده نمی‌شود  
- [ ] جزئی (Partial) — متادیتای پاسخ (هدرها، اندازه، زمان‌بندی) به کاربر بازگردانده می‌شود  
- [ ] کامل (Full) — بدنه کامل پاسخ (Response Body) از منبع داخلی به کاربر **منعکس می‌شود**
