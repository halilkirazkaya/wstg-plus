## WSTG-SESS-04 — تست برای متغیرهای نشست فاش شده (Exposed Session Variables)

فاش شدن متغیرهای نشست (Exposed Session Variables) زمانی رخ می‌دهد که شناسه‌های حساس نشست یا داده‌های مرتبط با وضعیت از طریق کانال‌های ناامن مانند رشته‌های پرس‌وجوی URL (URL query strings)، هدرهای ارجاع‌دهنده (Referer headers) یا گزارش‌های سیستم (System logs) منتقل شوند. این افشاسازی ریسک ربایش نشست (Session Hijacking) را به‌طور قابل‌توجهی افزایش می‌دهد، زیرا شناسه‌ها می‌توانند در تاریخچه مرورگر کش شده، توسط پروکسی‌های میانی (Proxies) ثبت شده یا از طریق هدر ارجاع‌دهنده به سایت‌های شخص ثالث نشت کنند. آزمون‌گران نفوذ (Pentesters) این موارد را با نظارت بر تمام درخواست‌های خروجی و بررسی گزارش‌های برنامه یا پیکربندی‌های سرور برای نشت ناخواسته داده‌ها شناسایی می‌کنند. در دنیای واقعی، مهاجمان از این نشتی‌ها برای استخراج توکن‌های نشست (Session tokens) معتبر از ماشین‌های اشتراکی یا تحلیل الگوهای ترافیک برای دور زدن احراز هویت (Authentication) و کسب دسترسی غیرمجاز به حساب‌های کاربری استفاده می‌کنند.


| فیلد | مقدار |
|---|---|
| **شناسه WSTG** | WSTG-SESS-04 |
| **CWE** | CWE-598 |
| **وضعیت تست** | انجام نشده |
| **شدت** | متوسط / بالا* |

> *شدت در صورتی "بالا" محسوب می‌شود که متغیر فاش شده یک توکن نشست باشد که امکان تسخیر فوری حساب کاربری (Account takeover) را فراهم کند.

### آیا شناسه نشست در رشته پرس‌وجوی URL منتقل می‌شود؟
- [ ] نه — شناسه‌های نشست در رشته پرس‌وجوی URL وجود **ندارند**  
- [ ] بله — شناسه‌ها وجود دارند اما نشست کوتاه‌مدت یا کم‌ریسک است  
- [ ] بله — شناسه‌های منحصربه‌فرد نشست در رشته پرس‌وجوی URL وجود **دارند** *(بحرانی)*  

### آیا برنامه توکن‌های نشست را از طریق هدر Referer نشت می‌دهد؟
- [ ] نه — `Referrer-Policy` از نشت جلوگیری می‌کند یا لینک شخص ثالثی وجود ندارد  
- [ ] بله — توکن‌های نشست از طریق هدر Referer به دامنه‌های شخص ثالث منتقل **می‌شوند**  

### آیا متغیرهای نشست در گزارش‌های سمت سرور یا برنامه ذخیره می‌شوند؟
- [ ] نه — متغیرهای نشست ماسک شده یا از گزارش‌ها حذف شده‌اند  
- [ ] بله — متغیرهای نشست در گزارش‌های برنامه یا وب‌سرور به‌صورت متن ساده (Plain text) ثبت **می‌شوند**  

### آیا برنامه از درخواست‌های GET برای عملیات تغییردهنده وضعیت استفاده می‌کند؟
- [ ] نه — برنامه از `POST` یا سایر متدهای غیر یکنوا (Non-idempotent) برای عملیات حساس استفاده می‌کند  
- [ ] بله — از `GET` استفاده می‌شود که باعث کش شدن داده‌های نشست در تاریخچه مرورگر و پروکسی‌های میانی می‌گردد  

### آیا هدر `Cache-Control` برای جلوگیری از کش شدن داده‌های مرتبط با نشست پیکربندی شده است؟
- [ ] بله — `Cache-Control: no-store` (یا مشابه آن) برای پاسخ‌های حساس اعمال **شده است**  
- [ ] بله — کنترل‌ها وجود دارند اما دور زدن آن‌ها از طریق موارد خاص مرورگر **امکان‌پذیر است**  
- [ ] نه — پاسخ‌های حساس حاوی داده‌های نشست توسط مرورگر **قابل کش شدن هستند**
