## WSTG-APIT-02 — API Broken Object Level Authorization

شکست در مجوزدهی سطح شیء (Broken Object Level Authorization - BOLA)، که با نام ارجاع مستقیم و ناامن به اشیاء (Insecure Direct Object Reference - IDOR) نیز شناخته می‌شود، زمانی رخ می‌دهد که یک رابط برنامه‌نویسی کاربردی (API) در اعتبارسنجی این موضوع که آیا یک کاربر دسترسی‌های لازم برای مشاهده یا دستکاری یک منبع خاص که با یک شناسه (ID) مشخص شده است را دارد یا خیر، شکست می‌خورد. مهاجمان با شمارش سیستماتیک یا حدس زدن شناسه‌ها - مانند شناسه‌های عددی یا UUIDها - در مسیرهای درخواست (Request paths)، پارامترهای کوئری (Query parameters)، یا بدنه‌های JSON برای دسترسی به داده‌های متعلق به سایر کاربران، از این آسیب‌پذیری سوءاستفاده می‌کنند. این آسیب‌پذیری رایج‌ترین و تاثیرگذارترین مشکل در امنیت مدرن API است که می‌تواند منجر به استخراج انبوه داده‌ها، تغییر غیرمجاز یا تصاحب کامل حساب کاربری (Account takeover) شود. از دیدگاه یک مهاجم، هدف شناسایی نقاط پایانی (Endpoints) است که شناسه‌های اشیاء را می‌پذیرند و تست این موضوع که آیا منطق مجوزدهی (Authorization) سمت سرور وجود ندارد یا می‌توان آن را با دستکاری آن شناسه‌ها دور زد (Bypass).

### آیا شناسه‌های اشیاء در درخواست‌های API قابل پیش‌بینی یا شمارش هستند؟
- [ ] خیر — شناسه‌ها طولانی، تصادفی و از نظر رمزنگاری امن هستند (مانند UUIDv4)  
- [ ] بله — شناسه‌ها اعداد صحیح متوالی هستند (مانند `101` و `102`)  
- [ ] بله — شناسه‌ها از یک الگوی قابل پیش‌بینی پیروی می‌کنند یا از اطلاعات عمومی مشتق شده‌اند  

### آیا API اعتبارسنجی سمت سرور برای مالکیت شیء را برای هر درخواست انجام می‌دهد؟
- [ ] بله — بررسی‌های مجوزدهی برای هر درخواست اعمال می‌شوند و دور زدن آن‌ها **امکان‌پذیر نیست**  
- [ ] بله — بررسی‌ها اعمال می‌شوند اما دور زدن آن‌ها از طریق آلودگی پارامتر (Parameter pollution) یا تونل‌سازی متد (Method tunneling) **امکان‌پذیر است**  
- [ ] خیر — برنامه صرفاً به احراز هویت کاربر تکیه می‌کند بدون اینکه مالکیت منبع خاص را بررسی کند  

### آیا امکان دسترسی یا تغییر منبع کاربر دیگر با تغییر شناسه وجود دارد؟
- [ ] خیر — دسترسی غیرمجاز به منابع سایر کاربران **امکان‌پذیر نیست**  
- [ ] بله — دسترسی غیرمجاز برای **خواندن** (BOLA افقی) **امکان‌پذیر است**  
- [ ] بله — **تغییر** یا **حذف** غیرمجاز (BOLA افقی) **امکان‌پذیر است**  

### آیا API اجازه دسترسی به اشیاء در سطح مدیریتی یا سیستم را با جایگزینی شناسه‌ها می‌دهد؟
- [ ] خیر — منابع مدیریتی توسط لایه‌های مجوزدهی ثانویه محافظت می‌شوند  
- [ ] بله — دسترسی یا تغییر منابع سطح سیستم (BOLA عمودی) **امکان‌پذیر است**  

### آیا می‌توان بررسی مجوزدهی را با انتقال شناسه به بخش دیگری از درخواست دور زد؟
- [ ] خیر — منطق مجوزدهی بدون توجه به مکان پارامتر ثابت است  
- [ ] بله — دور زدن زمانی **امکان‌پذیر است** که شناسه از مسیر URL به بدنه JSON یا هدرها منتقل شود  
- [ ] بله — دور زدن زمانی **امکان‌پذیر است** که چندین شناسه ارائه شود و سرور شناسه‌ی غیرمجاز را پردازش کند
