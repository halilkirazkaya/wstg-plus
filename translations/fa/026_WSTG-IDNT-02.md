## WSTG-IDNT-02 — تست فرآیند ثبت‌نام کاربر (Test User Registration Process)

تست فرآیند ثبت‌نام کاربر شامل ارزیابی گردش کاری است که طی آن شناسه‌های جدید ایجاد می‌شوند تا اطمینان حاصل شود که امکان سوءاستفاده از آن برای دسترسی غیرمجاز یا بهره‌برداری خودکار (Automated Exploitation) وجود ندارد. آسیب‌پذیری‌ها در این فرآیند اغلب به صورت عدم تایید هویت (Identity Verification) (مانند ایمیل/SMS)، حساسیت به ایجاد انبوه حساب کاربری توسط ربات‌ها، یا افشای اطلاعات (Information Disclosure) از طریق پیام‌های خطای پرجزئیات که نام‌های کاربری موجود را فاش می‌کنند، ظاهر می‌شوند. مهاجمان از این نقص‌ها برای انجام شمارش کاربران (User Enumeration)، اجرای کمپین‌های هرزنامه در مقیاس بزرگ یا دستکاری پارامترهای ثبت‌نام جهت ارتقاء سطح دسترسی (Privilege Escalation) در مرحله ایجاد حساب استفاده می‌کنند. این تست برای محافظت از یکپارچگی (Integrity) اپلیکیشن و جلوگیری از پر شدن سیستم با حساب‌های جعلی یا مخرب توسط مهاجمان حیاتی است.

### آیا پیش از فعال شدن حساب جدید، تایید هویت الزامی است؟
- [ ] بله — ثبت‌نام نیازمند یک توکن (Token) منحصربه‌فرد و دارای محدودیت زمانی است که از طریق ایمیل یا SMS ارسال می‌شود.
- [ ] بله — تایید هویت الزامی است اما توکن‌ها ضعیف یا قابل پیش‌بینی هستند.
- [ ] خیر — حساب‌ها بلافاصله و بدون هیچ مرحله تاییدی فعال می‌شوند.

### آیا فرآیند ثبت‌نام از شمارش کاربران (User Enumeration) جلوگیری می‌کند؟
- [ ] بله — اپلیکیشن برای هر دو حالت ایمیل/نام‌کاربری جدید و موجود، پاسخ‌های یکسان (Identical) برمی‌گرداند.
- [ ] خیر — پیام‌های خطا (مثلاً "ایمیل قبلاً استفاده شده است") به مهاجم اجازه می‌دهد کاربران ثبت‌نام شده را شناسایی (Map) کند.
- [ ] خیر — تفاوت‌های زمانی (Timing differences) در پاسخ‌های سرور فاش می‌کند که آیا کاربر وجود دارد یا خیر.

### آیا کنترل‌های ضد خودکارسازی برای جلوگیری از ثبت‌نام انبوه پیاده‌سازی شده‌اند؟
- [ ] بله — مکانیزم‌های کپچا (CAPTCHA) یا اثبات کار (Proof-of-work) وجود دارند و موثر هستند.
- [ ] بله — کنترل‌ها وجود دارند اما دور زدن (Bypass) آن‌ها از طریق نقاط انتهایی API یا دستکاری هدرها (Header Manipulation) امکان‌پذیر است.
- [ ] خیر — هیچ محدودیت نرخ (Rate Limiting) یا کپچایی روی نقطه انتهایی ثبت‌نام اعمال نشده است.

### آیا پارامترهای ثبت‌نام برای ارتقاء سطح دسترسی قابل دستکاری هستند؟
- [ ] خیر — نقش‌های کاربری دقیقاً توسط منطق سمت سرور (Server-side logic) تعیین می‌شوند.
- [ ] بله — پارامترهایی مانند `role` ،`admin` یا `group_id` می‌توانند در درخواست دستکاری شوند تا دسترسی بالاتری کسب شود.

### آیا سیاست رمز عبور (Password Policy) در مرحله ثبت‌نام اجباری شده است؟
- [ ] بله — الزامات رمز عبور قوی در سمت سرور اجبار شده‌اند.
- [ ] خیر — رمزهای عبور ضعیف (مثلاً "password123") توسط سیستم پذیرفته می‌شوند.
