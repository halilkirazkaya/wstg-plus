## WSTG-CONF-06 — تست متدهای HTTP (Test HTTP Methods)

تست متدهای HTTP شامل شناسایی این است که کدام فعل‌ها (Verbs) توسط وب‌سرور و اپلیکیشن پشتیبانی می‌شوند تا اطمینان حاصل شود که تنها عملکردهای ضروری در دسترس هستند. فراتر از متدهای استاندارد `GET` و `POST` ، ممکن است سرورها به طور ناخواسته متدهای خطرناکی مانند `PUT` ، `DELETE` یا `TRACE` را فعال کرده باشند که می‌تواند به مهاجمان اجازه دهد فایل‌های مخرب آپلود کنند، محتوای موجود را حذف کنند یا کوکی‌های نشست (Session Cookies) را از طریق ردیابی بین‌سایتی (Cross-Site Tracing - XST) استخراج کنند. تست‌کنندگان نفوذ (Pentesters) هدرهای پاسخ مانند `Allow` یا `Public` را بررسی کرده و سعی می‌کنند با استفاده از تکنیک‌های بازنویسی متد (Method Overriding) یا دستکاری متد (Verb Tampering)، محدودیت‌ها را برای دسترسی به عملکردهای غیرمجاز دور بزنند. این تست برای مقاوم‌سازی سطح حمله (Attack Surface Hardening) و جلوگیری از پیکربندی‌های اشتباهی که منجر به نفوذ به سرور یا دستکاری غیرمجاز داده‌ها می‌شود، حیاتی است.


| فیلد | مقدار |
|---|---|
| **شناسه WSTG** | WSTG-CONF-06 |
| **CWE** | CWE-650 |
| **وضعیت تست** | انجام نشده |
| **شدت** | پایین / متوسط* |

> *اگر متدهای `PUT` یا `DELETE` اجازه دستکاری غیرمجاز فایل را بدهند، یا اگر متد `TRACE` استخراج توکن نشست را تسهیل کند، شدت آسیب‌پذیری به سطح بالا (High) افزایش می‌یابد.

### آیا متدهای خطرناک HTTP مانند `PUT` یا `DELETE` در سراسر اپلیکیشن غیرفعال شده‌اند؟
- [ ] بله — تنها متدهای امن (GET, POST, HEAD) **فعال هستند**  
- [ ] خیر — متدهای `PUT` یا `DELETE` **فعال هستند** اما نیاز به احراز هویت (Authentication) معتبر دارند  
- [ ] خیر — متدهای `PUT` یا `DELETE` **فعال هستند** و بدون احراز هویت در دسترس می‌باشند *(حیاتی)*  

### آیا متد `TRACE` برای جلوگیری از ردیابی بین‌سایتی (XST) غیرفعال شده است؟
- [ ] بله — متد `TRACE` **غیرفعال است** یا کد وضعیت 405 Method Not Allowed را برمی‌گرداند  
- [ ] خیر — متد `TRACE` **فعال است** اما هدرهای حساس را منعکس نمی‌کند  
- [ ] خیر — متد `TRACE` **فعال است** و هدرهای `Cookie` یا `Authorization` را منعکس می‌کند *(متوسط)*  

### آیا بازنویسی متد HTTP (مانند `X-HTTP-Method-Override`) توسط سرور پشتیبانی می‌شود؟
- [ ] خیر — سرور هدرهای بازنویسی متد را **پردازش نمی‌کند**  
- [ ] بله — سرور هدرهای بازنویسی را پردازش می‌کند اما کنترل‌های دسترسی (Access Controls) **اعمال می‌شوند**  
- [ ] بله — سرور هدرهای بازنویسی را پردازش می‌کند و دور زدن کنترل دسترسی (Access Control Bypass) **امکان‌پذیر است**  

### آیا سرور به متدهای HTTP دلخواه یا نامعتبر به صورت امن پاسخ می‌دهد؟
- [ ] بله — سرور کد 405 Method Not Allowed یا 501 Not Implemented را برمی‌گرداند  
- [ ] خیر — سرور برای متدهای دلخواه مانند `JEFF` یا `TEST` کد 200 OK یا 500 Error را برمی‌گرداند  
- [ ] خیر — استفاده از متدهای دلخواه اجازه دور زدن فیلترهای احراز هویت یا مجوزدهی را می‌دهد  

### آیا متد `OPTIONS` محدود شده یا برای به حداقل رساندن افشای اطلاعات (Information Disclosure) پیکربندی شده است؟
- [ ] بله — متد `OPTIONS` غیرفعال است یا یک هدر `Allow` حداقلی را برمی‌گرداند  
- [ ] خیر — متد `OPTIONS` طیف گسترده‌ای از متدهای فعال، از جمله متدهای DAV یا فعل‌های خطایابی (Debugging) را فاش می‌کند
