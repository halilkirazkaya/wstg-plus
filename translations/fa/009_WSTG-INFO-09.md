## WSTG-INFO-09 — شناسایی اثر انگشت اپلیکیشن وب (Fingerprint Web Application)

شناسایی اثر انگشت (Fingerprinting) یک اپلیکیشن وب، فرآیند شناسایی فریم‌ورک (Framework) خاص، سیستم مدیریت محتوا (CMS) یا پشته تکنولوژی (Technology stack) زیربنایی و نسخه‌های مرتبط با آن است. این فاز شناسایی (Reconnaissance) حیاتی است، زیرا به نفوذگر اجازه می‌دهد تا اجزای شناسایی شده را با آسیب‌پذیری‌های (Vulnerabilities) شناخته شده (CVEs) و اکسپلویت‌های (Exploits) عمومی مطابقت دهد و سطح حمله (Attack surface) را به‌طور قابل توجهی محدود کند. اطلاعات معمولاً از هدرهای پاسخ HTTP (HTTP response headers)، مسیرهای فایل منحصر‌به‌فرد، نام کوکی‌ها (Cookie names)، متادیتاهای (Metadata) سورس‌کد HTML و هش‌های رمزنگاری (Cryptographic hashes) دارایی‌های ثابت (Static assets) جمع‌آوری می‌شوند. با تعیین دقیق پشته تکنولوژی، نفوذگر می‌تواند از اسکن خودکار (Automated scanning) عمومی به سمت بهره‌جویی (Exploitation) هدفمند از نقاط ضعف مختص نسخه حرکت کند.

### آیا هدرهای پاسخ HTTP پشته تکنولوژی یا شماره نسخه‌ها را فاش می‌کنند؟
- [ ] خیر — هدرهای حساس حذف شده‌اند یا حاوی مقادیر عمومی هستند  
- [ ] بله — هدرهای پیش‌فرض مانند `X-Powered-By`، `Server` یا `X-AspNet-Version` حضور دارند  
- [ ] بله — هدرها نسخه‌های خاص و قدیمی وب‌سرور یا فریم‌ورک اپلیکیشن را فاش می‌کنند  

### آیا فریم‌ورک اپلیکیشن یا CMS از طریق مسیرهای فایل منحصر‌به‌فرد یا قراردادهای نام‌گذاری قابل شناسایی است؟
- [ ] خیر — مسیرهای فایل مبهم‌سازی (Obfuscated) شده‌اند و تکنولوژی زیربنایی را آشکار نمی‌کنند  
- [ ] بله — ساختارهای دایرکتوری رایج (مانند `/wp-content/` ،`/_next/` ،`/node_modules/`) قابل مشاهده هستند  
- [ ] بله — فایل‌های منحصر‌به‌فرد مانند `robots.txt` ،`humans.txt` یا `sitemap.xml` حاوی متادیتای مختص تکنولوژی هستند  

### آیا شماره نسخه‌های خاص در سورس‌کد HTML یا دارایی‌های ثابت افشا شده‌اند؟
- [ ] خیر — هیچ اطلاعاتی از نسخه در کامنت‌ها یا پارامترهای دارایی‌ها یافت نشد  
- [ ] بله — کامنت‌های HTML یا تگ‌های متا (مانند `<meta name="generator" content="...">`) وجود دارند  
- [ ] بله — رشته‌های مربوط به نسخه به نام فایل‌های CSS/JS پیوست شده‌اند (مانند `jquery.js?v=1.12.4`) و غیرفعال‌سازی آن‌ها امکان‌پذیر نیست  

### آیا صفحات خطای پیش‌فرض یا رابط‌های مدیریتی، محیط نرم‌افزاری را آشکار می‌کنند؟
- [ ] خیر — اپلیکیشن برای تمامی کدهای وضعیت، از صفحات خطای سفارشی و عمومی استفاده می‌کند  
- [ ] بله — صفحات خطای پیش‌فرض برای وب‌سرور یا فریم‌ورک فعال هستند و داده‌های نسخه را نشت می‌دهند  
- [ ] بله — پورتال‌های ورود مدیریتی (مانند `/wp-admin` ،`/phpmyadmin`) در دسترس و قابل شناسایی هستند  

### آیا کوکی‌ها فریم‌ورک مدیریت نشست را فاش می‌کنند؟
- [ ] خیر — کوکی‌های نشست (Session Cookies) از نام‌های عمومی یا سفارشی استفاده می‌کنند  
- [ ] بله — نام کوکی‌های مختص تکنولوژی (مانند `PHPSESSID` ،`JSESSIONID` ،`ASPSESSIONID`) استفاده شده‌اند
