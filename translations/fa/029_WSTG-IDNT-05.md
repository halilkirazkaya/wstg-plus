## WSTG-IDNT-05 — تست سیاست‌های ضعیف یا اجرانشده برای نام کاربری (Testing for Weak or Unenforced Username Policy)

تست سیاست‌های ضعیف یا اجرانشده برای نام کاربری بر قوانینی تمرکز دارد که بر ایجاد شناسه حساب کاربری نظارت می‌کنند و میزان آسیب‌پذیری آن‌ها را در برابر شمارش (Enumeration) یا جعل (Spoofing) بررسی می‌کند. سیاست‌های ضعیف اغلب اجازه استفاده از توالی‌های قابل پیش‌بینی، کلمات متداول لغت‌نامه یا شناسه‌هایی را می‌دهند که مشابه شناسه‌های داخلی کارکنان است؛ این امر فضای جستجو را برای حملات بروت فورس (Brute Force) و پر کردن اعتبارنامه‌ها (Credential Stuffing) به‌طور قابل توجهی کاهش می‌دهد. از دیدگاه مهاجم، شناسایی این الگوها اولین گام در کشف حساب‌های کاربری در مقیاس وسیع است که اغلب از طریق تحلیل پیام‌های خطای ثبت‌نام، رفتار بازنشانی رمز عبور یا متاداده‌ها در پروفایل‌های عمومی به دست می‌آید. حصول اطمینان از یک سیاست قدرتمند، از نقشه‌برداری سیستماتیک پایگاه کاربران توسط مهاجمان جلوگیری کرده و دفاع در برابر فیشینگ (Phishing) هدفمند و تلاش‌های خودکار برای دور زدن احراز هویت (Authentication Bypass) را تسهیل می‌کند.


| فیلد | مقدار |
|---|---|
| **شناسه WSTG** | WSTG-IDNT-05 |
| **CWE** | CWE-521 |
| **وضعیت تست** | انجام نشده |
| **شدت** | کم / متوسط |

**منابع:**

**ابزارها:** `Burp Suite (Intruder)`, `ffuf`, `Custom Python Scripts`, `theHarvester`

### آیا نام‌های کاربری بر اساس الگوهای بسیار قابل پیش‌بینی یا ترتیبی هستند؟
- [ ] خیر — نام‌های کاربری رشته‌هایی تصادفی، تعریف شده توسط کاربر یا با آنتروپی بالا (High-Entropy) هستند.  
- [ ] بله — نام‌های کاربری از یک فرمت قابل پیش‌بینی پیروی می‌کنند (مانند `user1001` ،`user1002`) اما احراز هویت قدرتمند است.  
- [ ] بله — نام‌های کاربری **کاملاً ترتیبی** هستند یا از یک فرمت سازمانی شناخته شده پیروی می‌کنند که شمارش را آسان می‌سازد.  

### آیا اپلیکیشن الزامات مربوط به حداقل طول و پیچیدگی را برای نام‌های کاربری اعمال می‌کند؟
- [ ] بله — سیاست‌های سخت‌گیرانه برای طول و مجموعه کاراکترها در طول ثبت‌نام **اعمال می‌شوند**.  
- [ ] بله — سیاست‌هایی وجود دارند اما اجازه استفاده از نام‌های کاربری بسیار کوتاه (مثلاً ۱-۲ کاراکتر) یا بیش از حد ساده را می‌دهند.  
- [ ] خیر — **هیچ سیاستی** در رابطه با طول نام کاربری یا انواع کاراکترها اعمال نمی‌شود.  

### آیا می‌توان نام‌های کاربری معتبر را از طریق پاسخ‌های اپلیکیشن شمارش کرد؟
- [ ] خیر — اپلیکیشن پیام‌های خطای عمومی بازمی‌گرداند و برای تمام تلاش‌ها زمان‌بندی (Timing) یکسانی از خود نشان می‌دهد.  
- [ ] بله — اپلیکیشن پیام‌های خطای متمایزی بازمی‌گرداند (مانند "نام کاربری قبلاً انتخاب شده است") اما محدودسازی نرخ درخواست (Rate Limiting) **اعمال شده است**.  
- [ ] بله — اپلیکیشن نام‌های کاربری معتبر را از طریق خطاهای ثبت‌نام، بازنشانی رمز عبور یا تفاوت‌های زمان‌بندی **نشت (Leak)** می‌دهد.  

### آیا این سیاست از ثبت‌نام با نام‌های کاربری مدیریتی رایج یا رزرو شده جلوگیری می‌کند؟
- [ ] بله — اپلیکیشن نام‌های رایج (مانند `admin` ،`root` ،`support`) و اصطلاحات رزرو شده سیستمی را مسدود می‌کند.  
- [ ] خیر — کاربران **می‌توانند** حساب‌هایی با نام‌های حساس یا مدیریتی ثبت کنند.  

### آیا سیاست نام کاربری در تمام نقاط ورود (API، موبایل، وب) یکسان است؟
- [ ] بله — سیاست‌ها به‌طور یکسان در تمام رابط‌ها و نسخه‌ها **اعمال می‌شوند**.  
- [ ] خیر — نقاط انتهایی قدیمی (Legacy Endpoints) یا نسخه‌های خاص API همان محدودیت‌های نام کاربری را **اعمال نمی‌کنند**.
