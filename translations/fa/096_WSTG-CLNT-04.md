## WSTG-CLNT-04 — تست بازنشانی یو‌آر‌ال سمت کلاینت (Client-Side URL Redirect)

بازنشانی آدرس یو‌آر‌ال سمت کلاینت (Client-Side URL Redirect) زمانی رخ می‌دهد که یک اپلیکیشن تحت وب، ورودی‌های کنترل‌شده توسط کاربر را — معمولاً از طریق پارامترهای URL یا هش فرگمنت‌ها (Hash fragments) — دریافت کرده و از آن‌ها در یک سینک (Sink) بازنشانی سمت کلاینت مانند `window.location` یا `document.location.href` استفاده می‌کند. این آسیب‌پذیری عمدتاً توسط مهاجمان برای اجرای کمپین‌های فیشینگ (Phishing) پیشرفته مورد استفاده قرار می‌گیرد، زیرا قربانی در ابتدا به دامنه اصلی و قانونی اعتماد می‌کند و سپس به صورت بی‌صدا به یک سایت مخرب هدایت می‌شود. فراتر از فیشینگ، بازنشانی‌های سمت کلاینت اغلب می‌توانند برای استخراج غیرمجاز داده‌های حساس مانند توکن‌های دسترسی OAuth یا شناسه‌های نشست (Session identifiers) به زنجیره حملات متصل شوند؛ به‌ویژه اگر بازنشانی در زمانی رخ دهد که این مقادیر در URL یا هدر `Referer` حضور دارند. در اپلیکیشن‌های تک‌صفحه‌ای (SPA) مدرن، این آسیب‌پذیری به وفور در منطق مسیریابی (Routing logic)، پارامترهای "return to" پس از احراز هویت (Authentication)، یا عملکردهای تغییر زبان مشاهده می‌شود.

| **وضعیت تست** | انجام نشده |

### آیا اپلیکیشن برای مدیریت بازنشانی‌ها بر اساس ورودی کاربر از اسکریپت‌های سمت کلاینت استفاده می‌کند؟
- [ ] خیر — بازنشانی منحصراً در سمت سرور و از طریق کدهای وضعیت `3xx` HTTP مدیریت می‌شود.
- [ ] بله — اپلیکیشن از سینک‌های جاوا اسکریپت مانند `window.location` برای پیمایش بر اساس پارامترهای URL استفاده می‌کند.
- [ ] بله — اپلیکیشن از یک روتر فریم‌ورک سمت کلاینت (مانند React Router یا Vue Router) استفاده می‌کند که مسیرهای ارائه شده توسط کاربر را پردازش می‌کند.

### آیا کنترل‌های اعتبارسنجی ورودی یا لیست سفید (Whitelisting) برای URL مقصد پیاده‌سازی شده است؟
- [ ] بله — یک **لیست سفید (Whitelist)** سختگیرانه از دامنه‌ها/مسیرهای مجاز اعمال شده و دور زدن آن **امکان‌پذیر نیست**.
- [ ] بله — اعتبارسنجی وجود دارد اما به عبارات باقاعده (Regex) ضعیف یا لیست‌های سیاه (Blacklists) متکی است و دور زدن آن **امکان‌پذیر است**.
- [ ] خیر — اپلیکیشن هر رشته دلخواهی را به عنوان مقصد بازنشانی می‌پذیرد.

### آیا منطق بازنشانی را می‌توان با استفاده از تکنیک‌های متداول مبهم‌سازی (Obfuscation) دور زد؟
- [ ] خیر — کنترل‌ها به درستی کاراکترهای کدگذاری شده، URLهای مبتنی بر پروتکل (`//`) و بک‌اسلش‌ها را مدیریت می‌کنند.
- [ ] بله — دور زدن با استفاده از کدگذاری URL (URL encoding) یا کدگذاری دوگانه (Double-encoding) **امکان‌پذیر است**.
- [ ] بله — دور زدن با استفاده از URLهای مبتنی بر پروتکل یا کاراکتر `@` (مانند `https://expected.com@attacker.com`) **امکان‌پذیر است**.
- [ ] بله — دور زدن با استفاده از رفتارهای خاص مرورگر (Browser quirks) یا تزریق بایت پوچ (Null byte injection) **امکان‌پذیر است**.

### آیا فرآیند بازنشانی باعث افشای داده‌های حساس به سایت مقصد می‌شود؟
- [ ] خیر — هیچ داده حساسی در URL وجود ندارد و هیچ داده‌ای از طریق هدرها در حین بازنشانی منتقل نمی‌شود.
- [ ] بله — داده‌های حساس (مانند توکن‌های نشست یا کلیدهای API) از طریق هدر `Referer` به سایت خارجی **نشت می‌کنند**.
- [ ] بله — داده‌های حساس موجود در بخش فرگمنت یو‌آر‌ال (`#`) برای اسکریپت‌های سایت مقصد **قابل دسترسی هستند**.

### آیا امکان اجرای جاوا اسکریپت از طریق سینک بازنشانی (DOM-based XSS) وجود دارد؟
- [ ] خیر — سینک فقط اجازه پیمایش به پروتکل‌های `http` یا `https` را می‌دهد.
- [ ] بله — سینک بازنشانی اجازه استفاده از URIهای `:javascript` یا `:data` را می‌دهد که اجرای **DOM-based XSS** را **امکان‌پذیر می‌سازد**.
