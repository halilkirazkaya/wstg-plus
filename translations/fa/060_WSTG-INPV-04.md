## WSTG-INPV-04 — تست آلودگی پارامترهای HTTP (HTTP Parameter Pollution - HPP)

آلودگی پارامترهای HTTP (HPP) زمانی رخ می‌دهد که یک برنامه چندین پارامتر HTTP با نام یکسان دریافت کرده و آن‌ها را به روشی ناسازگار یا ناامن پردازش کند. با ارائه پارامترهای تکراری، مهاجم می‌تواند منطق داخلی برنامه را دستکاری کرده و به طور بالقوه فایروال‌های برنامه وب (Web Application Firewall - WAF)، فیلترهای اعتبارسنجی ورودی (Input Validation) یا مکانیزم‌های کنترل دسترسی (Access Control Mechanisms) را دور بزند. این آسیب‌پذیری به شدت به وب‌سرور یا فریم‌ورک (Framework) برنامه زیربنایی وابسته است که ممکن است اولین، آخرین یا نسخه‌ای الحاق شده (Concatenated) از پارامترهای تکراری را انتخاب کند. بهره‌برداری (Exploitation) معمولاً نقاط انتهایی (Endpoints) را هدف قرار می‌دهد که پارامترها را به APIهای سمت سرور (Back-end)، پرس‌وجوهای پایگاه داده یا مکانیزم‌های تغییر مسیر URL ارسال می‌کنند؛ این امر منجر به پیامدهایی از اسکریپت‌نویسی بین‌سایتی (Cross-Site Scripting - XSS) تا استخراج غیرمجاز داده‌ها (Data Exfiltration) می‌شود.

### برنامه چگونه با چندین پارامتر HTTP با نام یکسان برخورد می‌کند؟
- [ ] خیر — پارامترهای تکراری توسط سرور رد (Rejected) یا نادیده گرفته می‌شوند  
- [ ] بله — سرور به طور مداوم تنها اولین یا آخرین مورد را انتخاب می‌کند  
- [ ] بله — سرور مقادیر را الحاق (Concatenate) می‌کند که به طور بالقوه امکان تزریق (Injection) را فراهم می‌سازد  

### آیا می‌توان از HPP برای دور زدن کنترل‌های امنیتی مانند WAF یا فیلتر ورودی استفاده کرد؟
- [ ] خیر — کنترل‌های امنیتی به درستی تمام موارد تکرار شده یک پارامتر را بررسی می‌کنند  
- [ ] بله — یک WAF یا فیلتر فقط اولین مورد را بررسی می‌کند و اجازه می‌دهد مورد دوم مخرب عبور کند  
- [ ] بله — الحاق به مهاجم اجازه می‌دهد تا یک محموله مخرب (Payload) را میان چندین پارامتر تقسیم کند تا از شناسایی فرار کند  

### آیا برنامه پارامترهای آلوده را بدون پاک‌سازی مناسب در پاسخ منعکس می‌کند؟
- [ ] خیر — پارامترها قبل از انعکاس در رابط کاربری (UI)، پاک‌سازی (Sanitized) یا کدگذاری می‌شوند  
- [ ] بله — آلودگی منجر به محتوای منعکس شده (Reflected) می‌شود، اما پاک‌سازی اعمال می‌گردد  
- [ ] بله — آلودگی منجر به محتوای منعکس شده می‌شود و پاک‌سازی اعمال نمی‌گردد که به XSS یا خطاهای منطقی ختم می‌شود  

### آیا پارامترهای ارسال شده به فراخوانی‌های API داخلی یا سیستم‌های Back-end در برابر آلودگی آسیب‌پذیر هستند؟
- [ ] خیر — درخواست‌های Back-end از روش‌های ساخت ایمن و غیر داینامیک استفاده می‌کنند  
- [ ] بله — HPP به مهاجم اجازه می‌دهد تا پارامترها را در ساختار درخواست Back-end تزریق (Inject) یا بازنویسی (Overwrite) کند  

### آیا برنامه هنگام آلوده شدن پارامترها در درخواست‌های GET در مقابل POST رفتار متفاوتی نشان می‌دهد؟
- [ ] خیر — رفتار در تمامی متدهای HTTP یکسان است  
- [ ] بله — فقط پارامترهای GET در برابر آلودگی آسیب‌پذیر هستند  
- [ ] بله — فقط پارامترهای POST (Body) در برابر آلودگی آسیب‌پذیر هستند
