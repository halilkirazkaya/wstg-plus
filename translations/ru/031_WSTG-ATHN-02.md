## WSTG-ATHN-02 — Тестирование на наличие учетных данных по умолчанию (Default Credentials)

Тестирование на наличие учетных данных по умолчанию включает в себя выявление административных интерфейсов, сервисов или компонентов приложения, которые все еще используют имена пользователей и пароли, установленные на заводе или предоставленные вендором. Эта уязвимость является высокоприоритетной целью для злоумышленников, так как она часто открывает прямой путь к полной компрометации системы, административному доступу или удаленному выполнению кода (Remote Code Execution, RCE) с минимальными усилиями. Обычно это встречается в готовом программном обеспечении (off-the-shelf software), CMS-платформах, инструментах управления базами данных и интегрированном оборудовании, таком как устройства интернета вещей (IoT) или сетевые устройства. Эксплуатация обычно осуществляется путем сопоставления выявленных версий программного обеспечения с публичными базами данных паролей по умолчанию или с использованием автоматизированных инструментов перебора учетных данных (Credential Stuffing) на этапах разведки и эксплуатации.

### Доступны ли административные интерфейсы или интерфейсы управления из интернета или недоверенных сегментов сети?
- [ ] Нет — интерфейсы управления недоступны  
- [ ] Да — интерфейсы присутствуют, но доступ ограничен сетевыми средствами контроля (например, VPN, список разрешенных IP-адресов/IP Allowlist)  
- [ ] Да — интерфейсы общедоступны и требуют аутентификации  

### Были ли изменены учетные данные по умолчанию, предоставленные вендором, для всех выявленных компонентов?
- [ ] Да — все учетные данные по умолчанию **были изменены** во всех компонентах *(Наиболее безопасно)*  
- [ ] Да — учетные данные **были изменены** для основного приложения, но второстепенные компоненты (например, CMS, БД) остались с настройками по умолчанию  
- [ ] Нет — учетные данные по умолчанию **активны** на одном или нескольких идентифицированных интерфейсах *(Критично)*  

### Требует ли приложение принудительной смены пароля при первом входе для новых или административных учетных записей?
- [ ] Да — смена пароля **обязательна** перед выполнением любых административных действий  
- [ ] Нет — приложение **позволяет** использовать учетные данные по умолчанию неограниченное время  

### Активны ли механизмы блокировки или средства ограничения скорости (Rate Limiting) для предотвращения автоматизированного тестирования учетных данных по умолчанию?
- [ ] Да — применяется строгое ограничение скорости или блокировка учетной записи (Account Lockout)  
- [ ] Да — средства контроля присутствуют, но возможен их обход через манипуляцию заголовками (Header Manipulation) или ротацию IP-адресов  
- [ ] Нет — механизмы ограничения скорости или блокировки **не включены**  

### Используют ли сторонние плагины, промежуточное ПО (Middleware) или административные инструменты (например, phpMyAdmin, Tomcat Manager) уникальные учетные данные?
- [ ] Нет — сторонние компоненты отсутствуют в среде  
- [ ] Да — все сторонние компоненты используют уникальные, отличные от стандартных учетные данные  
- [ ] Нет — по крайней мере один сторонний компонент **доступен** через известные учетные данные по умолчанию
