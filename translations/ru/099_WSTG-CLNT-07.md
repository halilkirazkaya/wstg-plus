## WSTG-CLNT-07 — Совместное использование ресурсов из разных источников (Cross-Origin Resource Sharing, CORS)

Совместное использование ресурсов из разных источников (Cross-Origin Resource Sharing, CORS) — это браузерный механизм, который позволяет серверу явно разрешать доступ к своим ресурсам из различных источников, фактически ослабляя политику одного источника (Same-Origin Policy, SOP). Ошибки конфигурации возникают, когда приложение чрезмерно доверяет произвольным источникам, часто путем отражения заголовка `Origin` в ответном заголовке `Access-Control-Allow-Origin` или использования некорректно реализованных белых списков (Whitelists) на основе регулярных выражений. С точки зрения злоумышленника, разрешительная политика CORS может быть эксплуатирована путем размещения вредоносного скрипта на подконтрольном домене, который выполняет аутентифицированные запросы к уязвимому приложению. Это позволяет злоумышленнику эксфильтровать (Exfiltrate) конфиденциальные данные, такие как CSRF-токены или персональные данные (Personally Identifiable Information, PII), непосредственно из тела ответа. Данная уязвимость наиболее критична для конечных точек API (API endpoints), которые обрабатывают аутентифицированные сессии и возвращают непубличную информацию.

### Реализует ли приложение заголовки CORS на аутентифицированных конечных точках?
- [ ] Нет — CORS **не включен**; браузер по умолчанию применяет строгую политику Same-Origin Policy  
- [ ] Да — заголовки CORS присутствуют и ограничены строгим **белым списком**  
- [ ] Да — заголовки CORS присутствуют, но настроены **избыточно разрешающе**  

### Как сервер обрабатывает заголовок `Access-Control-Allow-Origin` (ACAO)?
- [ ] ACAO настроен на конкретный **доверенный** статический домен  
- [ ] ACAO настроен на `*` (wildcard), и передача учетных данных (Credentials) **запрещена**  
- [ ] ACAO настроен на `*` (wildcard), и передача учетных данных **разрешена** *(Примечание: Большинство браузеров блокируют такую конфигурацию)*  
- [ ] ACAO **динамически отражает** значение заголовка `Origin` из запроса  

### Установлен ли заголовок `Access-Control-Allow-Credentials` (ACAC) в значение `true`?
- [ ] Нет — ACAC **отсутствует** или установлен в `false`  
- [ ] Да — ACAC установлен в `true`, но ACAO **ограничен** доверенными источниками  
- [ ] Да — ACAC установлен в `true`, и ACAO **отражает** произвольные источники *(Критично)*  

### Можно ли обойти белый список источников через манипуляции с поддоменами или значением null?
- [ ] Нет — белый список строго соблюдается и **не может** быть обойден  
- [ ] Да — белый список принимает **любой** поддомен цели (например, `attacker.target.com`)  
- [ ] Да — белый список принимает значение `null` через песочницу iframe  
- [ ] Да — белый список уязвим для обхода регулярных выражений (например, `target.com.attacker.com` или `target.com-safe.com`)  

### Позволяет ли конфигурация CORS эксфильтрацию конфиденциальных данных?
- [ ] Нет — открытые конечные точки **не содержат** конфиденциальных данных или данных, специфичных для сессии  
- [ ] Да — аутентифицированные конечные точки возвращают PII, учетные данные или CSRF-токены, которые **могут** быть прочитаны неавторизованным источником
