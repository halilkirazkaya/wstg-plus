## WSTG-INPV-19 — Тестирование на подделку запросов на стороне сервера (Server-Side Request Forgery, SSRF)

Подделка запроса на стороне сервера (Server-Side Request Forgery, SSRF) возникает, когда злоумышленник может заставить веб-приложение отправлять несанкционированные запросы с сервера к внутренним или внешним ресурсам. Манипулируя параметрами, принимающими URL, имена хостов или IP-адреса, атакующий может обходить средства сетевого контроля, такие как межсетевые экраны (firewalls) и списки контроля доступа (ACL), для сканирования сегментов внутренней сети, доступа к сервисам метаданных облака (IMDS) или взаимодействия с уязвимыми внутренними API и базами данных. Эта уязвимость обычно проявляется в функциях загрузки изображений, генерации PDF, вебхуках (webhooks) или загрузке файлов по URL. С точки зрения злоумышленника, SSRF является мощным примитивом, используемым для продвижения в изолированные среды (pivot), извлечения конфиденциальных данных конфигурации или потенциального выполнения удаленного кода (Remote Code Execution, RCE) путем взаимодействия с внутренними сервисами, такими как `Redis` или `Memcached`.

### Обрабатывает ли приложение URL-адреса или имена хостов, предоставленные пользователем?
- [ ] Нет — функции не принимают внешние URL или имена хостов  
- [ ] Да — функция существует, но входные данные строго проверяются по белому списку (allow-list)  
- [ ] Да — функция существует и принимает предоставленные пользователем URL  

### Применяются ли средства валидации или черные списки к запрашиваемому адресу?
- [ ] Да — применяется строгий белый список (allow-list) доверенных доменов/IP  
- [ ] Да — используется черный список (deny-list) (например, блокировка `127.0.0.1`, `169.254.169.254`)  
- [ ] Нет — к входным данным не применяется никакая валидация или фильтрация  

### Возможно ли обойти фильтры с помощью кодирования, перенаправлений или DNS rebinding?
- [ ] Нет — фильтры надежны и безопасно обрабатывают перенаправления/DNS-разрешение  
- [ ] Да — обход возможен с использованием URL-кодирования (URL encoding) или альтернативных форматов IP (hex, octal)  
- [ ] Да — обход возможен через HTTP-перенаправления (3xx redirects) на внутренние цели  
- [ ] Да — обход возможен с помощью атак DNS rebinding для разрешения имен во внутренние IP  

### Может ли приложение получить доступ к внутренним сервисам метаданных или локальным интерфейсам?
- [ ] Нет — сегменты локальной сети и эндпоинты облачных метаданных недоступны  
- [ ] Да — доступ к localhost (`127.0.0.1`) или loopback-интерфейсам возможен  
- [ ] Да — доступ к сервисам облачных метаданных (например, AWS/GCP/Azure IMDS) возможен *(Критично)*  

### Каков характер ответа SSRF?
- [ ] Слепая (Blind) — пользователю не возвращается ответ или метаданные  
- [ ] Частичная (Partial) — пользователю возвращаются метаданные ответа (заголовки, размер, время отклика)  
- [ ] Полная (Full) — полное тело ответа от внутреннего ресурса отображается пользователю
