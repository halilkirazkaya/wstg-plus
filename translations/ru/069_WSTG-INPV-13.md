## WSTG-INPV-13 — Внедрение строк форматирования (Format String Injection)

Внедрение строк форматирования (Format String Injection) происходит, когда веб-приложение передает контролируемые пользователем входные данные напрямую в аргумент строки форматирования вариативной функции, такой как семейство `printf` в языке C или низкоуровневые обертки для логирования. Хотя эта уязвимость реже встречается в современных веб-фреймворках с управляемым кодом, она остается критической в устаревших CGI-приложениях, нативных расширениях или граничных системах логирования, взаимодействующих с низкоуровневыми системными библиотеками. Злоумышленники эксплуатируют это, предоставляя спецификаторы формата, такие как `%x` или `%p`, для утечки конфиденциальных адресов памяти и данных из стека, или спецификатор `%n` для выполнения произвольной записи в память. Успешная эксплуатация обычно приводит к полной компрометации системы через удаленное выполнение кода (Remote Code Execution, RCE) или, как минимум, к серьезному отказу в обслуживании (Denial of Service, DoS) из-за сбоев процессов.

### Обрабатывает ли приложение входные данные пользователя через нативный код или устаревшие CGI-интерфейсы?
- [ ] Нет — приложение полностью построено на фреймворках с управляемым кодом (например, JVM, CLR)  
- [ ] Да — приложение использует JNI, нативные расширения C/C++ или устаревшие бинарные CGI, где **могут** существовать уязвимости строк форматирования  

### Передаются ли вводимые пользователем данные в качестве основного аргумента функциям, чувствительным к формату?
- [ ] Нет — входные данные передаются как аргументы данных, а строки форматирования являются **статическими** и **жестко заданными**  
- [ ] Да — пользовательский ввод напрямую конкатенируется в аргумент строки форматирования, но **применяется** очистка (sanitization)  
- [ ] Да — пользовательский ввод напрямую используется в качестве строки форматирования, и очистка **не применяется**  

### Возможна ли утечка содержимого памяти с помощью спецификаторов формата?
- [ ] Нет — входные данные валидируются, и такие спецификаторы, как `%p`, `%x` или `%s`, **не обрабатываются**  
- [ ] Да — передача спецификаторов `%p` или `%x` приводит к отображению адресов памяти стека или кучи  
- [ ] Да — конфиденциальные данные (например, указатели, canary-значения стека) **могут** быть извлечены с помощью спецификаторов формата  

### Можно ли вызвать сбой приложения или манипулировать им с помощью спецификатора `%n`?
- [ ] Нет — спецификаторы, разрешающие запись в память, **отключены** или заблокированы компилятором/средой  
- [ ] Да — приложение завершается со сбоем (DoS), когда передаются строки `%s%s%s%s` или аналогичные  
- [ ] Да — **возможна** произвольная запись в память через `%n`, что потенциально ведет к удаленному выполнению кода (RCE)  

### Можно ли обойти современные механизмы защиты бинарных файлов (ASLR, DEP, Stack Canaries) с помощью этой инъекции?
- [ ] Нет — среда защищена, и утечка памяти **не может** быть использована для обхода защитных механизмов  
- [ ] Да — внедрение строк форматирования **может** быть использовано для утечки базовых адресов, что делает обход ASLR/DEP **возможным**
