## WSTG-CONF-14 — Тестирование других неправильных конфигураций заголовков безопасности HTTP

Тестирование неправильных конфигураций заголовков безопасности HTTP (HTTP security header misconfigurations) включает проверку наличия и корректности реализации защитных заголовков, обеспечивающих базовую защиту от распространенных векторов веб-атак. Хотя эти заголовки не устраняют уязвимости в самом коде, их отсутствие значительно повышает риск атак типа «человек посередине» (Man-in-the-middle, MITM), эксплуатации MIME-сниффинга (MIME-sniffing) и непреднамеренной утечки данных между разными источниками (cross-origin) через рефереры (referrers). С точки зрения злоумышленника, отсутствие заголовков безопасности является явным индикатором плохо защищенной среды, что часто облегчает реализацию более сложных цепочек атак, таких как перехват сеанса (session hijacking) или эксфильтрация информации. Эти конфигурации обычно оцениваются на уровне сервера и должны последовательно применяться ко всем типам ответов, включая конечные точки API (API endpoints) и статические ресурсы.

### Аудит наличия заголовков безопасности
| Заголовок | Присутствует | Отсутствует | Рекомендуемая конфигурация |
|--------|:-------:|:-------:|---------------------------|
| `Strict-Transport-Security` |  |  | `max-age=63072000; includeSubDomains; preload` |
| `X-Content-Type-Options` |  |  | `nosniff` |
| `Referrer-Policy` |  |  | `no-referrer` или `strict-origin-when-cross-origin` |
| `Permissions-Policy` |  |  | (Зависит от функций, например, `geolocation=()`) |
| `Cross-Origin-Opener-Policy` |  |  | `same-origin` |

### Каким образом обеспечивается применение заголовков безопасности в рамках всего приложения?
- [ ] Да — заголовки применяются глобально через центральный балансировщик нагрузки или обратный прокси-сервер (reverse proxy) и **не могут** быть переопределены.  
- [ ] Да — заголовки применяются к ответам основного документа, но **отсутствуют** в ответах API или статических ресурсах.  
- [ ] Нет — заголовки применяются непоследовательно или **отсутствуют** во всем домене приложения.  

### Правильно ли настроен заголовок Strict-Transport-Security (HSTS)?
- [ ] Да — параметр `max-age` установлен на длительный период (например, 2 года) и опция `includeSubDomains` **включена**.  
- [ ] Да — параметр `max-age` установлен, но опция `includeSubDomains` **выключена**, что оставляет поддомены уязвимыми.  
- [ ] Нет — заголовок HSTS **отсутствует** или `max-age` установлен в 0, что допускает атаки с понижением версии протокола (protocol downgrade attacks).  

### Предотвращает ли Referrer-Policy утечку конфиденциальных параметров URL?
- [ ] Да — политика установлена на `no-referrer` или `same-origin` *(Наиболее безопасно)*.  
- [ ] Да — политика установлена на `strict-origin-when-cross-origin`.  
- [ ] Нет — политика **не установлена** или установлена на `unsafe-url`, что потенциально приводит к утечке токенов или конфиденциальных персональных данных (PII) в заголовках referer.  

### Предотвращает ли заголовок X-Content-Type-Options MIME-сниффинг?
- [ ] Да — директива `nosniff` **присутствует** и правильно реализована.  
- [ ] Нет — заголовок **отсутствует**, что потенциально позволяет браузеру исполнять неисполняемые файлы (например, изображения) как скрипты.
