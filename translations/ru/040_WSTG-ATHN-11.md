## WSTG-ATHN-11 — Тестирование многофакторной аутентификации (MFA)

Тестирование многофакторной аутентификации (Multi-Factor Authentication, MFA) оценивает надежность второго уровня безопасности, предназначенного для предотвращения несанкционированного доступа даже в случае компрометации основных учетных данных. Злоумышленники часто пытаются обойти MFA, выявляя эндпоинты, где она применяется непоследовательно, такие как устаревшие (legacy) версии API, мобильные бэкенды или процессы сброса пароля. Эксплуатация часто включает в себя манипулирование ответами сервера, перебор (Brute Force) краткосрочных кодов (OTP), или использование состояния гонки (Race Conditions) и недостатков управления сессиями, которые позволяют пользователю перейти в состояние аутентификации без предоставления второго фактора.

### Обеспечивается ли MFA согласованно на всех порталах аутентификации?
- [ ] Да — MFA требуется для всех попыток входа через веб-интерфейс, мобильные приложения и API  
- [ ] Да — но MFA **не применяется** на определенных эндпоинтах (например, устаревшие `/api/v1/login` или порталы для мобильных устройств)  
- [ ] Нет — MFA **не реализована** в приложении *(Критично)*  

### Можно ли обойти этап проверки MFA путем прямой навигации по URL или манипуляции ответами сервера?
- [ ] Нет — прямая навигация или подмена параметров **не позволяют** обойти проверку  
- [ ] Да — переход напрямую к URL-адресам внутренних панелей управления **возможен** без прохождения проверки MFA  
- [ ] Да — манипуляция ответом сервера (например, замена `401 Unauthorized` на `200 OK`) **возможна** для получения доступа  

### Защищен ли процесс проверки кода MFA от атак методом перебора (Brute Force)?
- [ ] Да — после нескольких неудачных попыток ввода одноразового пароля (OTP) **применяется** строгое ограничение частоты запросов (Rate Limiting) или блокировка учетной записи  
- [ ] Да — ограничение частоты запросов существует, но его **возможно** обойти с помощью ротации IP-адресов или манипуляции заголовками  
- [ ] Нет — ограничение частоты запросов не применяется, и **возможен** автоматизированный перебор кодов  

### Сохраняет ли приложение безопасное состояние сессии во время перехода к MFA?
- [ ] Да — токены высокопривилегированных сессий выдаются **только после** успешного прохождения MFA  
- [ ] Нет — полнофункциональный сессионный файл cookie выдается **до** завершения MFA, что позволяет получить доступ к некоторым функциям  
- [ ] Нет — приложение использует статический идентификатор или предсказуемый переход сессии, который **может** быть перехвачен  

### Уязвимы ли альтернативные факторы MFA (SMS, Email, резервные коды) для эксплуатации?
- [ ] Нет — все факторы используют безопасные, непредсказуемые и ограниченные по времени значения  
- [ ] Да — резервные коды предсказуемы или **могут** быть перечислены (fuzzing)  
- [ ] Да — функциональность «Повторно отправить код» может быть использована для проведения SMS/Email-флудинга или раскрытия частичных контактных данных
