## WSTG-ATHZ-03 — Тестирование на повышение привилегий

Повышение привилегий (Privilege Escalation) происходит, когда злоумышленник использует уязвимости для получения доступа к ресурсам или функционалу, зарезервированному для пользователей с более высоким уровнем авторизации или другими учетными записями. При вертикальном повышении (Vertical Escalation) обычный пользователь пытается получить доступ к административным функциям, в то время как горизонтальное повышение (Horizontal Escalation) предполагает доступ к данным другого пользователя с тем же уровнем привилегий. Эти недостатки обычно проявляются в плохо реализованных списках управления доступом (ACL), небезопасных прямых ссылках на объекты (IDOR) или манипуляциях параметрами (Parameter Tampering) в токенах сессий или идентификации. С точки зрения злоумышленника, это часто достигается путем манипулирования числовыми идентификаторами (Numeric IDs), изменения параметров на основе ролей в куках (Cookies) или JWT, либо путем прямого вызова скрытых конечных точек API (API Endpoints), в которых отсутствуют проверки авторизации на стороне сервера.

### Реализованы ли в приложении несколько уровней привилегий или мультиарендность (Multi-tenancy)?
- [ ] Нет — приложение является системой с одним пользователем или одной ролью  
- [ ] Да — определены несколько ролей или арендаторов, требующих тестирования авторизации  

### Возможно ли горизонтальное повышение привилегий (доступ на том же уровне)?
- [ ] Нет — проверки авторизации применяются ко всем идентификаторам ресурсов и владельцам сессий  
- [ ] Да — доступ к данным других пользователей возможен через IDOR или манипуляцию параметрами  
- [ ] Да — возможна утечка данных, но изменение данных других пользователей невозможно  

### Возможно ли вертикальное повышение привилегий (от низкого к высокому)?
- [ ] Нет — административные конечные точки строго соблюдают управление доступом на основе ролей (RBAC)  
- [ ] Да — низкопривилегированные пользователи могут получить доступ к административным функциям через прямой доступ по URL  
- [ ] Да — доступ к административным функциям возможен путем манипулирования заголовками ролей, куками или утверждениями (Claims) JWT  

### Можно ли изменить роли или разрешения пользователя с помощью массового назначения (Mass Assignment) или загрязнения параметров (Parameter Pollution)?
- [ ] Нет — поля на основе ролей доступны только для чтения и не могут быть изменены пользователями  
- [ ] Да — привилегии пользователя могут быть повышены путем включения скрытых параметров (например, `role=admin`) при обновлении профиля или регистрации  

### Применяются ли проверки авторизации последовательно во всех версиях API и методах HTTP?
- [ ] Да — средства контроля применяются последовательно во всех версиях и методах  
- [ ] Нет — устаревшие версии API (например, `/v1/`) или определенные методы HTTP (например, `PUT`, `DELETE`, `PATCH`) позволяют обойти авторизацию  
- [ ] Нет — административные функции скрыты в пользовательском интерфейсе, но активны на уровне API для всех пользователей
