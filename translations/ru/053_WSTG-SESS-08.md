## WSTG-SESS-08 — Путаница сессий (Session Puzzling)

Путаница сессий (Session Puzzling), также известная как перегрузка переменных сессии (Session Variable Overloading), возникает, когда приложение использует одну и ту же переменную сессии для нескольких целей в разных модулях или не может должным образом очистить состояния сессии при переходах между рабочими процессами. Злоумышленники эксплуатируют это поведение, заполняя переменные сессии в одном контексте — например, при неаутентифицированном предварительном просмотре профиля или многоэтапной регистрации — а затем переходят в защищенную область, где приложение ошибочно доверяет этим ранее созданным переменным. Это может привести к критическим последствиям, включая обход аутентификации (Authentication Bypass), повышение привилегий (Privilege Escalation) или несанкционированный доступ к конфиденциальным данным путем обмана приложения, заставляя его считать, что сессия находится в состоянии с более высокими привилегиями, чем на самом деле. Уязвимость наиболее распространена в сложных приложениях с сохранением состояния (stateful), где логика управления сессиями (Session Management) непоследовательно применяется в различных функциональных компонентах.

### Использует ли приложение одни и те же имена переменных сессии для разных целей в различных модулях?
- [ ] Нет — имена переменных уникальны или области видимости изолированы  
- [ ] Да — существуют общие имена переменных, но состояние очищается между модулями  
- [ ] Да — существуют общие имена переменных, и состояние **не** очищается  

### Могут ли неаутентифицированные действия влиять на переменные сессии, используемые в аутентифицированных контекстах?
- [ ] Нет — переменные сессии инициализируются только **после** успешной аутентификации  
- [ ] Да — переменные сессии могут быть установлены до входа в систему, но **не** влияют на состояние после входа  
- [ ] Да — переменным сессии, установленным до аутентификации, **доверяют** после аутентификации *(Критическая)*  

### Переинициализирует ли приложение идентификатор сессии и связанные с ним переменные при изменении уровня привилегий?
- [ ] Да — сессия полностью сбрасывается, и **выдается** новый ID  
- [ ] Частично — выдается новый ID, но некоторые переменные сессии **сохраняются**  
- [ ] Нет — ID сессии и все переменные **остаются** без изменений  

### Можно ли получить административные привилегии путем манипулирования переменными сессии через учетную запись без привилегий?
- [ ] Нет — переменные привилегий **не могут** быть изменены пользователями  
- [ ] Да — переменные могут быть изменены, но приложение **проверяет** их в базе данных бэкенда  
- [ ] Да — переменные могут быть изменены, и приложение неявно **доверяет** состоянию сессии *(Критическая)*  

### Очищается ли состояние сессии сразу после завершения определенного рабочего процесса (например, сброс пароля, оформление заказа)?
- [ ] Да — переменные сессии для рабочего процесса уничтожаются по его завершении  
- [ ] Нет — переменные рабочего процесса **сохраняются** и могут быть повторно использованы в других областях приложения
