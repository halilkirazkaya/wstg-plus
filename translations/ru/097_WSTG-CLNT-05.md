## WSTG-CLNT-05 — Тестирование внедрения CSS (CSS Injection)

Внедрение CSS (CSS Injection) происходит, когда приложение позволяет входным данным пользователя влиять на каскадные таблицы стилей (CSS) веб-страницы без надлежащей очистки или экранирования. Хотя это часто воспринимается как косметическая проблема, злоумышленники могут использовать внедрение CSS для эксфильтрации конфиденциальных данных, таких как токены CSRF или идентификаторы сессий (Session IDs), используя селекторы атрибутов и свойства `background-image` для инициирования внешних запросов. Эта уязвимость обычно возникает в эндпоинтах (endpoints), где пользовательские настройки (например, кастомные темы, цвета шрифтов) отражаются внутри блоков `<style>` или встроенных атрибутов `style`. С точки зрения злоумышленника, успешная эксплуатация может привести к подмене интерфейса (UI redressing), фишингу через несанкционированное изменение макета или скрытой эксфильтрации данных в средах, где строгие политики безопасности контента (Content Security Policies, CSP) в противном случае могли бы блокировать атаки на основе Cross-Site Scripting (XSS).


| Поле | Значение |
|---|---|
| **WSTG ID** | WSTG-CLNT-05 |
| **CWE** | CWE-74 |
| **Статус тестирования** | Не выполнялось |
| **Критичность** | Средняя / Высокая* |

> *Критичность становится высокой, если точка внедрения позволяет эксфильтрацию конфиденциальных атрибутов, таких как токены CSRF, или если она может быть использована для подмены интерфейса (UI redressing) в критически важных рабочих процессах.

### Отражает ли приложение контролируемые пользователем входные данные в контексте CSS?
- [ ] Нет — входные данные не отражаются в тегах style, атрибутах или CSS-файлах  
- [ ] Да — данные отражаются, но строго ограничены безопасными буквенно-цифровыми значениями  
- [ ] Да — данные отражаются внутри атрибута `style` или блока `<style>`  

### Выполняется ли надлежащая очистка специфичных для CSS метасимволов и функций?
- [ ] Да — такие символы, как `{`, `}`, `:`, и функции вроде `url()` экранируются должным образом  
- [ ] Да — очистка выполняется, но возможен обход (bypass) через кодирование или CSS-комментарии  
- [ ] Нет — очистка к метасимволам CSS не применяется  

### Возможна ли эксфильтрация данных с использованием CSS-селекторов?
- [ ] Нет — селекторы не могут инициировать внешние запросы или привести к утечке данных  
- [ ] Да — возможна частичная эксфильтрация данных через селекторы атрибутов и `background-image`  
- [ ] Да — возможна полная эксфильтрация конфиденциальных токенов с использованием автоматизированных методов перебора (Brute Force)  

### Смягчает ли политика безопасности контента (CSP) риск внедрения CSS?
- [ ] Да — `style-src` ограничен значением 'self', а `img-src` или `connect-src` ограничены  
- [ ] Да — CSP присутствует, но использует 'unsafe-inline' или разрешает внешние домены через подстановочные знаки  
- [ ] Нет — CSP для предотвращения загрузки внешних ресурсов через CSS не реализована  

### Можно ли использовать уязвимость для подмены интерфейса (UI redressing) или фишинга?
- [ ] Нет — область внедрения слишком ограничена для существенного изменения макета  
- [ ] Да — модификация интерфейса (UI modification) возможна, что позволяет накладывать вредоносные элементы  
- [ ] Да — возможен полный контроль над макетом страницы, что способствует проведению крайне убедительных фишинговых атак
