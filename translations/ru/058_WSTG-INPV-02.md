## WSTG-INPV-02 — Хранимый межсайтовый скриптинг (Stored Cross Site Scripting, XSS)

Хранимый межсайтовый скриптинг (Stored Cross Site Scripting, XSS), также известный как персистентный XSS (Persistent XSS), возникает, когда приложение получает данные от пользователя и сохраняет их в постоянной базе данных, файловой системе или другом хранилище без надлежащей валидации (Validation) или кодирования (Encoding). Когда жертва впоследствии переходит на страницу, которая извлекает и отображает эти неочищенные данные, вредоносный скрипт выполняется в контексте её браузера в рамках источника (Origin) приложения. Эта уязвимость особенно опасна, так как она не требует от жертвы перехода по специально созданной ссылке; любой пользователь, просматривающий затронутую страницу, становится целью, что может привести к перехвату сессии (Session Hijacking), несанкционированным действиям или сбору учетных данных (Credential Harvesting). Злоумышленники обычно выбирают целями разделы комментариев, профили пользователей, доски сообщений и журналы администратора, где ввод данных постоянно отображается для других пользователей или администраторов.

### Существуют ли точки входа, которые сохраняют пользовательский ввод для последующего отображения другим пользователям?
- [ ] Нет — приложение не сохраняет контролируемый пользователем ввод для последующей отрисовки  
- [ ] Да — ввод сохраняется (например, профили, комментарии), но **не** отрисовывается в контексте HTML  
- [ ] Да — ввод сохраняется и отрисовывается для других пользователей или администраторов  

### Применяется ли кодирование выходных данных или санитайзинг при отображении сохраненных данных?
- [ ] Да — контекстное кодирование выходных данных (Output Encoding) **применяется**, и обход **невозможен** *(Наиболее безопасно)*  
- [ ] Да — санитайзинг (Sanitization) (например, `DOMPurify`) **применяется**, но обход **возможен** из-за ошибок конфигурации  
- [ ] Нет — данные отрисовываются непосредственно в DOM **без** кодирования или санитайзинга *(Высокий риск)*  

### Можно ли обойти валидацию входных данных или санитайзинг с помощью альтернативных пейлоадов?
- [ ] Нет — фильтры безопасно обрабатывают различные кодировки, нестандартные теги и обработчики событий  
- [ ] Да — обход **возможен** с использованием кодирования HTML-сущностей или специфических тегов (например, `<svg>`, `<img>`)  
- [ ] Да — обход **возможен** с помощью полиглот-пейлоадов (Polyglot Payloads) или XSS на основе мутации (mXSS)  

### Обеспечивает ли политика безопасности контента (Content Security Policy, CSP) дополнительный уровень защиты?
- [ ] Да — строгая CSP **включена** и предотвращает выполнение встроенных (inline) скриптов и загрузку из неавторизованных источников  
- [ ] Да — CSP **включена**, но настроена неправильно (например, разрешен `unsafe-inline` или слишком широкий `script-src`)  
- [ ] Нет — CSP для приложения **не включена**  

### Возможно ли реализовать цепочку "Stored XSS в RCE" или другие сценарии с высоким уровнем воздействия (Blind XSS)?
- [ ] Нет — воздействие ограничено контекстом браузера на стороне клиента  
- [ ] Да — хранимый XSS **может** быть использован для атаки на администраторов ("Слепой" XSS / Blind XSS) во внутренних панелях управления  
- [ ] Да — хранимый XSS **может** быть связан с другими уязвимостями (например, CSRF, эксфильтрация конфиденциальных данных)
