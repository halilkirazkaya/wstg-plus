## WSTG-CLNT-04 — Тестирование клиентского перенаправления URL (Client-Side URL Redirect)

Клиентское перенаправление URL (Client-side URL Redirect) возникает, когда веб-приложение принимает контролируемые пользователем входные данные — обычно через параметры URL или фрагменты хеша — и использует их в клиентской sink-функции (Sink), такой как `window.location` или `document.location.href`. Эту уязвимость злоумышленники используют в первую очередь для проведения сложных фишинговых кампаний (Phishing), так как жертва изначально доверяет легитимному домену, прежде чем будет незаметно перенаправлена на вредоносный сайт. Помимо фишинга, клиентские перенаправления часто могут быть объединены в цепочку для эксфильтрации конфиденциальных данных, таких как токены доступа OAuth (OAuth access tokens) или идентификаторы сессий (Session identifiers), если перенаправление происходит в тот момент, когда эти значения присутствуют в URL или заголовке `Referer`. В современных одностраничных приложениях (SPA) эта уязвимость часто встречается в логике маршрутизации, параметрах возврата («return to») после аутентификации или в функциях переключения языка.

### Использует ли приложение клиентские скрипты для обработки перенаправлений на основе пользовательского ввода?
- [ ] Нет — перенаправление обрабатывается исключительно на стороне сервера с помощью HTTP-кодов состояния `3xx`  
- [ ] Да — приложение использует JavaScript-функции (Sinks), такие как `window.location`, для навигации на основе параметров URL  
- [ ] Да — приложение использует роутер клиентского фреймворка (например, React Router, Vue Router), который обрабатывает пути, предоставленные пользователем  

### Реализованы ли проверка входных данных или контроль на основе белых списков (Whitelisting) для целевого URL?
- [ ] Да — применяется строгий **белый список** разрешенных доменов/путей, и обход (Bypass) **невозможен**  
- [ ] Да — проверка присутствует, но полагается на слабые регулярные выражения или черные списки, и обход **возможен**  
- [ ] Нет — приложение принимает любую произвольную строку в качестве цели перенаправления  

### Можно ли обойти логику перенаправления с помощью распространенных методов обфускации?
- [ ] Нет — механизмы контроля корректно обрабатывают закодированные символы, протокольно-относительные URL (Protocol-relative URL) (`//`) и обратные косые черты  
- [ ] Да — обход **возможен** с использованием URL-кодирования или двойного кодирования  
- [ ] Да — обход **возможен** с использованием протокольно-относительных URL или символов `@` (например, `https://expected.com@attacker.com`)  
- [ ] Да — обход **возможен** с использованием специфических особенностей браузеров или инъекции нулевого байта (Null byte injection)  

### Раскрывает ли процесс перенаправления конфиденциальные данные целевому сайту?
- [ ] Нет — конфиденциальные данные отсутствуют в URL и не передаются через заголовки во время перенаправления  
- [ ] Да — конфиденциальные данные (например, сессионные токены, API-ключи) **утекают** через заголовок `Referer` на внешний сайт  
- [ ] Да — конфиденциальные данные, присутствующие во фрагменте URL (`#`), **доступны** скриптам целевого сайта  

### Возможна ли передача JavaScript через sink-функцию перенаправления (DOM-based XSS)?
- [ ] Нет — sink-функция позволяет выполнять навигацию только по протоколам `http` или `https`  
- [ ] Да — sink-функция перенаправления разрешает использование URI-схем `javascript:` или `data:`, что делает **возможным** выполнение DOM-ориентированного межсайтового скриптинга (DOM-based XSS)
