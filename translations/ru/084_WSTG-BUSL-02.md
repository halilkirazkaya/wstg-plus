## WSTG-BUSL-02 — Тестирование возможности подделки запросов (Test Ability to Forge Requests)

Подделка запросов в контексте бизнес-логики (Business Logic) подразумевает манипулирование определенными приложением параметрами для выполнения действий вне рамок предусмотренного рабочего процесса или уровня авторизации (Authorization). Злоумышленники нацеливаются на многошаговые процессы, такие как системы оформления заказа или регистрация учетных записей, где состояние поддерживается с помощью клиентских параметров, которые сервер не подвергает повторной валидации (Re-validation) на основе доверенного источника данных на стороне бэкенда (Backend). Изменяя скрытые поля, значения JSON или параметры REST, такие как цены, количество или внутренние флаги статуса, злоумышленник может обойти требования по оплате, осуществить повышение привилегий (Privilege Escalation) или вызвать непреднамеренные изменения состояния. Эта уязвимость обычно проявляется в веб-формах с сохранением состояния (Stateful) или API, где сервер неявно доверяет представлению бизнес-состояния со стороны клиента во время транзакции.


| Поле | Значение |
|---|---|
| **WSTG ID** | WSTG-BUSL-02 |
| **CWE** | CWE-602 |
| **Статус теста** | Не выполнено |
| **Уровень риска** | Высокий / Критический* |

> *Уровень риска становится критическим, если поддельный запрос позволяет совершить финансовые потери, несанкционированные административные действия или полный захват учетной записи (Account Takeover).

### Проверяет ли приложение транзакционные параметры на соответствие серверному «источнику истины» (Source of Truth)?
- [ ] Да — все критичные параметры (цена, роль, ID) проверяются по базе данных, и обход (Bypass) невозможен  
- [ ] Да — валидация применяется, но ее можно обойти с помощью загрязнения параметров (Parameter Pollution) или альтернативного кодирования  
- [ ] Нет — приложение доверяет клиентским значениям для определения стоимости или результата транзакции *(Критично)*  

### Могут ли критически важные для бизнеса значения (например, количества, суммы) быть изменены на несанкционированные?
- [ ] Нет — отрицательные, нулевые или чрезмерно высокие значения отклоняются сервером  
- [ ] Да — сервер принимает измененные значения, но только в ограниченном диапазоне  
- [ ] Да — могут быть отправлены отрицательные или нулевые значения, что ведет к финансовым потерям или обходу логики  

### Используются ли скрытые поля или параметры API для сохранения состояния в многошаговых процессах?
- [ ] Нет — состояние безопасно поддерживается через серверные сессии или подписанные токены  
- [ ] Да — состояние передается через клиент, но проверки целостности (HMAC/подписи) включены и надежны  
- [ ] Да — состояние передается через клиент, а проверки целостности отключены или могут быть удалены  

### Возможно ли подделать запросы для пропуска обязательных промежуточных шагов в рабочем процессе (Workflow)?
- [ ] Нет — сервер обеспечивает строгую последовательность операций для каждой транзакции  
- [ ] Да — шаги могут быть пропущены, но финальное действие все равно требует валидных данных из предыдущих шагов  
- [ ] Да — финальный запрос на «отправку» или «завершение» может быть подделан напрямую для обхода авторизации или этапов оплаты
