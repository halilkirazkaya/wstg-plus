## WSTG-APIT-01 — Разведка API (API Reconnaissance)

Разведка API (API Reconnaissance) — это систематический процесс идентификации и сопоставления поверхности атаки API для обнаружения эндпоинтов (endpoints), поддерживаемых методов и базовых структур данных. Злоумышленники выполняют этот этап для поиска недокументированных («теневых») API (shadow APIs), устаревших версий с унаследованными уязвимостями и открытых файлов документации, таких как определения Swagger или OpenAPI, которые раскрывают внутреннюю бизнес-логику. Анализируя предсказуемые шаблоны URL-адресов, клиентские файлы JavaScript и результаты брутфорса директорий (directory brute-force), злоумышленник может построить подробную карту API для выявления целей атак типа «Нарушение авторизации на уровне объекта» (Broken Object Level Authorization, BOLA) или «Массовое назначение» (Mass Assignment). Эта разведка обычно нацелена на стандартные пути, такие как `/api/v1/`, `/swagger.json` или `/graphql`, для получения плана серверной функциональности приложения.

### Доступны ли публично файлы документации API (Swagger, OpenAPI, WSDL)?
- [ ] Нет — документация API недоступна или ограничена аутентификацией  
- [ ] Да — документация доступна, но требует действительных учетных данных  
- [ ] Да — документация API (например, `/swagger-ui.html`) общедоступна без аутентификации  

### Можно ли обнаружить недокументированные или «теневые» эндпоинты API с помощью фаззинга (fuzzing)?
- [ ] Нет — фаззинг директорий и эндпоинтов не выявил недокументированных путей  
- [ ] Да — недокументированные эндпоинты были найдены, но к ним невозможно получить доступ без авторизации  
- [ ] Да — недокументированные эндпоинты были найдены и к ним можно получить доступ без действительной авторизации  

### Раскрывает ли приложение несколько версий API (например, /v1/, /v2/, /beta/)?
- [ ] Нет — доступна только текущая защищенная версия API  
- [ ] Да — существуют устаревшие версии, но в них реализованы те же механизмы безопасности, что и в текущей версии  
- [ ] Да — устаревшие версии (например, `/v1/`) доступны и в них отсутствуют механизмы безопасности более новых версий  

### Содержат ли клиентские ресурсы (JavaScript/мобильные приложения) утечки структур эндпоинтов или ключей API?
- [ ] Нет — клиентский код не содержит жестко закодированных (hardcoded) путей API или конфиденциальных ключей  
- [ ] Да — клиентский код содержит карты эндпоинтов API, но не содержит конфиденциальных ключей  
- [ ] Да — ключи API, секреты или URL-адреса эндпоинтов только для внутреннего использования жестко закодированы в клиентских ресурсах  

### Можно ли обнаружить скрытые параметры или заголовки на известных эндпоинтах?
- [ ] Нет — фаззинг параметров (например, с помощью `Arjun`) не выявил скрытых входных данных  
- [ ] Да — скрытые параметры (например, `debug=true`, `admin=1`) были обнаружены, но они не функционируют  
- [ ] Да — скрытые параметры были обнаружены и могут быть использованы для изменения поведения приложения
