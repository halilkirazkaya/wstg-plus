## WSTG-APIT-02 — API Broken Object Level Authorization

Нарушение авторизации на уровне объектов (Broken Object Level Authorization, BOLA), также известное как небезопасная прямая ссылка на объект (Insecure Direct Object Reference, IDOR), возникает, когда API не в состоянии проверить, имеет ли пользователь соответствующие разрешения для доступа к конкретному ресурсу или манипулирования им, идентифицируемому по ID. Злоумышленники эксплуатируют это путем систематического перечисления или угадывания идентификаторов — таких как числовые ID или UUID — в путях запросов, параметрах запросов или телах JSON для доступа к данным, принадлежащим другим пользователям. Эта уязвимость является наиболее распространенной и значимой проблемой в безопасности API, потенциально приводящей к массовой эксфильтрации данных (Data Exfiltration), несанкционированному изменению или полному захвату учетной записи (Account Takeover). С точки зрения злоумышленника цель состоит в том, чтобы выявить эндпоинты (Endpoints), принимающие идентификаторы объектов, и проверить, отсутствует ли серверная логика авторизации или можно ли ее обойти путем манипулирования этими ID.

### Являются ли идентификаторы объектов предсказуемыми или перечисляемыми в запросах API?
- [ ] Нет — идентификаторы длинные, случайные и криптографически стойкие (например, UUIDv4)  
- [ ] Да — идентификаторы представляют собой последовательные целые числа (например, `101`, `102`)  
- [ ] Да — идентификаторы следуют предсказуемому шаблону или производны от общедоступной информации  

### Выполняет ли API проверку владения объектом на стороне сервера для каждого запроса?
- [ ] Да — проверки авторизации применяются для каждого запроса, и их обход **невозможен**  
- [ ] Да — проверки применяются, но обход **возможен** через Parameter Pollution или туннелирование методов (Method Tunneling)  
- [ ] Нет — приложение полагается исключительно на аутентификацию пользователя без проверки владения конкретным ресурсом  

### Возможно ли получить доступ к ресурсу другого пользователя или изменить его путем смены идентификатора?
- [ ] Нет — несанкционированный доступ к ресурсам других пользователей **невозможен**  
- [ ] Да — несанкционированный доступ на **чтение** (Horizontal BOLA) **возможен**  
- [ ] Да — несанкционированное **изменение** или **удаление** (Horizontal BOLA) **возможно**  

### Позволяет ли API получать доступ к административным объектам или объектам системного уровня путем замены ID?
- [ ] Нет — административные ресурсы защищены дополнительными уровнями авторизации  
- [ ] Да — доступ к ресурсам системного уровня или их изменение (Vertical BOLA) **возможны**  

### Можно ли обойти проверку авторизации, переместив идентификатор в другую часть запроса?
- [ ] Нет — логика авторизации последовательна независимо от расположения параметра  
- [ ] Да — обход **возможен**, когда ID перемещается из пути URL в тело JSON или заголовки  
- [ ] Да — обход **возможен**, когда предоставляется несколько ID и сервер обрабатывает тот, на который отсутствуют права доступа
