## WSTG-CLNT-11 — Тестирование Web Messaging

Web Messaging (Веб-сообщения), или `postMessage`, обеспечивает кросс-доменное взаимодействие (cross-origin communication) между объектами window, такими как родительская страница и дочернее всплывающее окно или встроенный iframe. Этот механизм имеет решающее значение для функциональности современных веб-приложений, но создает значительные риски, если принимающее приложение не проверяет личность отправителя или неправильно обрабатывает полезную нагрузку (Payload) сообщения. Злоумышленники эксплуатируют эти уязвимости, размещая вредоносный сайт, который встраивает целевое приложение и отправляет специально сформированные сообщения для инициирования непреднамеренных действий, эксфильтрации (извлечения) конфиденциальных данных или выполнения межсайтового скриптинга на основе DOM (DOM-based XSS). Успешная эксплуатация происходит, когда прослушиватели сообщений (message listeners) строго не проверяют свойство `origin` или передают `message.data` напрямую в опасные функции-поглотители (execution sinks).

### Использует ли приложение API `postMessage` для междокументного взаимодействия?
- [ ] Нет — прослушиватели `postMessage` **отсутствуют** в клиентском коде  
- [ ] Да — `postMessage` используется для взаимодействия внутренних компонентов  
- [ ] Да — `postMessage` используется для взаимодействия со сторонними доменами или виджетами  

### Проверяется ли строго источник (origin) входящих сообщений?
- [ ] Да — источник проверяется по строгому белому списку (whitelist) с использованием `===` или идентичного сравнения *(Наиболее безопасно)*  
- [ ] Да — источник проверяется с помощью регулярного выражения (regex), но логика **допускает** обход (bypass) (например, `^https://trusted.com`)  
- [ ] Нет — свойство `origin` **не проверяется**, что позволяет любому домену отправлять сообщения  
- [ ] Нет — в `targetOrigin` отправителя используется подстановочный знак (wildcard) `*`, что потенциально ведет к утечке данных вредоносным прослушивателям  

### Очищается (sanitized) ли полезная нагрузка сообщения перед обработкой приложением?
- [ ] Да — данные обрабатываются как обычный текст, и опасные функции-поглотители (sinks) не используются  
- [ ] Да — данные парсятся (например, `JSON.parse`) и проверяются перед использованием, обход **невозможен**  
- [ ] Нет — данные сообщения передаются напрямую в опасные функции-поглотители, такие как `innerHTML`, `eval()` или `setTimeout()`  
- [ ] Нет — данные сообщения используются для навигации в окне или изменения `location.href` без предварительной проверки  

### Может ли злоумышленник инициировать несанкционированные действия или извлечь данные с помощью вредоносного сообщения?
- [ ] Нет — даже при подмене сообщения доступ к конфиденциальным действиям или данным невозможен  
- [ ] Да — специально сформированное сообщение **может** инициировать изменение критических состояний (например, смену пароля, обновление профиля)  
- [ ] Да — специально сформированное сообщение **может** привести к DOM-based XSS или эксфильтрации CSRF-токенов/сессионных данных
