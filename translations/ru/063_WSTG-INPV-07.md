## WSTG-INPV-07 — XML-инъекция (XML Injection)

XML-инъекция (XML Injection) возникает, когда приложение включает предоставленные пользователем данные в XML-документ или поток без надлежащей проверки (validation), очистки (sanitization) или экранирования (escaping) метасимволов XML. Путем внедрения определенных XML-тегов или структурных элементов злоумышленник может манипулировать логикой документа, обходить механизмы аутентификации или нарушать целостность данных, обрабатываемых серверной частью. Эта уязвимость часто встречается в веб-сервисах на базе SOAP, REST API, принимающих XML, SAML-утверждениях (SAML assertions) и приложениях, которые динамически генерируют конфигурационные файлы XML или отчеты. С точки зрения злоумышленника, цель состоит в том, чтобы выйти за пределы предполагаемого контекста данных для изменения структуры документа, что потенциально может привести к несанкционированному повышению привилегий (privilege escalation) или выполнению непредусмотренной бизнес-логики.

### Принимает ли и обрабатывает ли приложение входные данные в формате XML от пользователя?
- [ ] Нет — приложение **не** обрабатывает входные данные XML  
- [ ] Да — входные данные XML обрабатываются через API (SOAP/REST)  
- [ ] Да — XML обрабатывается через загрузку файлов (SVG, DOCX и т. д.)  

### Экранируются или очищаются ли метасимволы XML (например, `<`, `>`, `&`, `'`, `"`) надлежащим образом?
- [ ] Да — все метасимволы **надлежащим образом** экранируются или очищаются *(Наиболее безопасно)*  
- [ ] Да — применяется некоторая фильтрация, но обход **возможен** с использованием кодирования (encoding)  
- [ ] Нет — необработанные входные данные вставляются непосредственно в структуры XML **без** очистки  

### Применяется ли проверка схемы (schema validation) на стороне сервера (XSD/DTD) для всех входных данных XML?
- [ ] Да — строгая проверка схемы **включена** и предотвращает структурные изменения  
- [ ] Да — проверка **включена**, но **не** предотвращает инъекцию тегов (tag injection) в поля данных  
- [ ] Нет — проверка схемы **отключена** или не реализована  

### Может ли злоумышленник внедрить новые XML-теги для изменения структуры документа или логики?
- [ ] Нет — структура остается неизменной независимо от входных данных  
- [ ] Да — теги могут быть внедрены, но они **не могут** изменить бизнес-логику  
- [ ] Да — внедрение тегов **возможно** и позволяет обойти логику или изменить данные *(Критично)*  

### Можно ли манипулировать XML-парсером (XML parser) с помощью разделов CDATA или комментариев XML?
- [ ] Нет — парсер правильно обрабатывает или отклоняет маркеры CDATA/комментариев  
- [ ] Да — внедрение `<![CDATA[...]]>` или `<!-- ... -->` **возможно** для сокрытия или обхода фильтров
