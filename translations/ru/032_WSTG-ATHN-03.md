## WSTG-ATHN-03 — Тестирование на наличие слабых механизмов блокировки учетных записей

Механизм блокировки учетной записи (Account lockout mechanism) является критически важной мерой эшелонированной защиты (defense-in-depth), предназначенной для противодействия атакам методом перебора паролей (Brute Force) и подстановки учетных данных (Credential Stuffing) путем деактивации учетной записи после определенного количества неудачных попыток аутентификации. Без надежной политики блокировки злоумышленники могут использовать автоматизированные инструменты для систематического подбора паролей к множеству учетных записей (Password Spraying) или атаковать одну конкретную высокозначимую учетную запись до достижения успеха. Эта уязвимость обычно проявляется на порталах входа, формах сброса пароля и конечных точках API (API), где отсутствует ограничение скорости запросов (Rate Limiting) или защита на уровне учетной записи. С точки зрения атакующего, слабый или отсутствующий механизм блокировки позволяет совершать бесконечное количество попыток подбора пароля, в то время как чрезмерно агрессивный механизм может быть использован для реализации распределенной атаки типа «отказ в обслуживании» (DoS) против легитимных пользователей путем преднамеренной блокировки их учетных записей.

| Поле | Значение |
|---|---|
| **WSTG ID** | WSTG-ATHN-03 |
| **CWE** | CWE-307 |
| **Статус теста** | Не выполнено |
| **Уровень риска** | Средний / Высокий* |

### Обеспечивается ли принудительная блокировка учетной записи после определенного количества неудачных попыток?
- [ ] Да — учетная запись блокируется после достижения определенного разумного порога (например, 5–10 попыток).
- [ ] Да — учетная запись блокируется, но только после чрезмерно большого количества попыток.
- [ ] Нет — учетная запись **не может** быть заблокирована, что позволяет проводить неограниченный перебор паролей.

### Можно ли обойти механизм блокировки с помощью распространенных методов обхода (evasion techniques)?
- [ ] Нет — блокировка применяется на стороне сервера (server-side) и **не зависит** от изменений на стороне клиента.
- [ ] Да — блокировку **можно** обойти путем ротации IP-адресов или использования пула прокси-серверов.
- [ ] Да — блокировку **можно** обойти путем манипулирования HTTP-заголовками, такими как `X-Forwarded-For` или `User-Agent`.
- [ ] Да — блокировку **можно** обойти путем очистки файлов cookie или запуска новых сессий.

### Предусматривает ли механизм блокировки способ восстановления учетной записи или истечение срока блокировки?
- [ ] Да — учетная запись разблокируется автоматически через определенное время или с помощью подтверждения по электронной почте.
- [ ] Да — для разблокировки учетной записи требуется вмешательство администратора.
- [ ] Нет — блокировка является постоянной без четкого пути восстановления, что повышает риск возникновения DoS.

### Различает ли приложение существующие и несуществующие имена пользователей при блокировке?
- [ ] Нет — ответ приложения идентичен как для существующих, так и для несуществующих пользователей.
- [ ] Да — приложение сообщает о том, что учетная запись заблокирована, что позволяет проводить перечисление имен пользователей (Username Enumeration).

### Уязвим ли механизм блокировки к атакам типа «отказ в обслуживании» (DoS)?
- [ ] Нет — такие меры контроля, как CAPTCHA или ограничение скорости по IP-адресу, предотвращают массовую блокировку учетных записей.
- [ ] Да — злоумышленник **может** систематически блокировать всех известных пользователей, вводя неверные пароли.
