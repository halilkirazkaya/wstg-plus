## WSTG-ATHN-01 — Тестирование передачи учетных данных по зашифрованному каналу (Testing for Credentials Transported over an Encrypted Channel)

Тестирование передачи учетных данных по зашифрованному каналу гарантирует, что конфиденциальные данные аутентификации, такие как имена пользователей, пароли и токены сессий (session tokens), защищены от перехвата во время транзита. Злоумышленники, находящиеся в сети — например, реализующие атаку «человек посередине» (Man-in-the-Middle, MitM) в публичных сетях Wi-Fi или скомпрометированных внутренних сетях — могут использовать инструменты сниффинга пакетов (packet sniffing) для захвата учетных данных в открытом виде, если TLS/SSL не применяется должным образом. Данная уязвимость обычно возникает в точках входа (login endpoints), формах сброса пароля и заголовках аутентификации API, где приложение не требует обязательного использования HTTPS или некорректно выполняет перенаправление с HTTP. Успешная эксплуатация (Exploit) предоставляет злоумышленнику полный доступ к учетным записям пользователей, что ведет к полной компрометации пользовательских данных и потенциальному горизонтальному перемещению (lateral movement) внутри приложения.

### Принудительно ли используется HTTPS для всего процесса аутентификации?
- [ ] Да — HSTS **включен** и весь трафик строго принудительно направляется через HTTPS  
- [ ] Да — перенаправление на HTTPS происходит, но HSTS **не включен**  
- [ ] Нет — учетные данные **могут** быть отправлены через незашифрованный протокол HTTP  

### Передаются ли страницы входа и формы по зашифрованному каналу?
- [ ] Да — страница, содержащая форму входа, доставляется через HTTPS  
- [ ] Нет — страница входа передается через HTTP, что позволяет осуществить подмену действия формы (form-action hijacking) или перехват учетных данных  

### Применяется ли флаг `Secure` ко всем конфиденциальным куки-файлам (cookies) сессии?
- [ ] Да — флаг `Secure` **применен** ко всем куки-файлам аутентификации и сессии  
- [ ] Да — флаг `Secure` **применен** только к некоторым куки-файлам  
- [ ] Нет — флаг `Secure` **не применен**, что позволяет извлечь куки через незашифрованные запросы  

### Поддерживает ли сервер слабые версии TLS или небезопасные наборы шифров (cipher suites)?
- [ ] Нет — **разрешены** только современные версии TLS (1.2+) и стойкие шифры  
- [ ] Да — **поддерживаются** устаревшие версии (TLS 1.0/1.1) или слабые шифры (RC4, DES, CBC)  

### Предотвращает ли приложение передачу учетных данных на сторонние домены по незашифрованным каналам?
- [ ] Да — конфиденциальные данные не отправляются на внешние домены, либо все внешние вызовы принудительно выполняются через HTTPS  
- [ ] Нет — заголовки аутентификации или учетные данные отправляются на сторонние эндпоинты (например, аналитика или SSO) через HTTP
