## WSTG-ATHN-06 — Тестирование на наличие уязвимости кэширования в браузере (Testing for Browser Cache Weakness)

Уязвимость кэширования в браузере (Browser Cache Weakness) возникает, когда конфиденциальная информация сохраняется в локальном кэше браузера и может быть извлечена неавторизованным пользователем, имеющим доступ к той же физической машине. Отсутствие надлежащих заголовков управления кэшированием (Cache-Control headers) позволяет потенциально конфиденциальным данным, таким как личная информация, данные учетной записи или идентификаторы сессии (Session identifiers), сохраняться после того, как пользователь вышел из системы или завершил сеанс. Злоумышленники или последующие пользователи общих или публичных терминалов могут использовать это, перемещаясь по истории браузера, используя кнопку «Назад» или проверяя файлы локального кэша для извлечения кэшированных ответов. Обеспечение внедрения строгих директив, таких как `Cache-Control: no-store` и `Pragma: no-cache`, имеет решающее значение для любого приложения, обрабатывающего аутентифицированный контент, чтобы гарантировать, что конфиденциальные данные никогда не будут записаны на диск.

### Присутствуют ли соответствующие заголовки управления кэшированием на страницах с аутентификацией или конфиденциальной информацией?
- [ ] Да — заголовки `Cache-Control: no-store, no-cache` и `Pragma: no-cache` **присутствуют** и **правильно настроены**  
- [ ] Да — некоторые заголовки присутствуют, но `no-store` **отсутствует**, что позволяет выполнять кэширование на диск  
- [ ] Нет — заголовки, связанные с кэшированием, **не применяются**, и приложение полагается на поведение браузера по умолчанию  

### Использует ли приложение директиву `private` для данных конкретного пользователя?
- [ ] Да — директива `Cache-Control: private` **включена** для всего персонализированного контента для предотвращения кэширования на промежуточных прокси-серверах (Intermediate proxies)  
- [ ] Нет — контент помечен как `public` или директива `private` отсутствует, что делает его **доступным для кэширования** промежуточными прокси-серверами  

### Доступна ли конфиденциальная информация через кнопку браузера «Назад» после успешного выхода из системы?
- [ ] Нет — браузер запрашивает повторную аутентификацию или страница **не загружается** из локального кэша  
- [ ] Да — конфиденциальная информация **по-прежнему видна** при нажатии кнопки «Назад» после завершения сессии  

### Защищены ли от кэширования конфиденциальные не-HTML файлы (например, PDF, выгрузки CSV, ответы JSON API)?
- [ ] Нет — приложение не генерирует и не обрабатывает конфиденциальные не-HTML файлы  
- [ ] Да — строгие заголовки управления кэшированием **применяются** ко всем загрузкам конфиденциальных файлов и конечным точкам API (API endpoints)  
- [ ] Нет — конфиденциальные загрузки или ответы API **сохраняются** в локальном кэше браузера или во временных директориях  

### Использует ли приложение заголовок `Expires: 0` или дату в прошлом для предотвращения использования устаревших данных?
- [ ] Да — заголовок `Expires` установлен в `0` или в прошедшую дату, что **гарантирует**, что браузер будет считать контент немедленно устаревшим  
- [ ] Нет — заголовок `Expires` **отсутствует** или для конфиденциальных страниц установлена дата в будущем
