## WSTG-ERRH-01 — Тестирование на наличие ненадлежащей обработки ошибок

Ненадлежащая обработка ошибок (Improper Error Handling) возникает, когда веб-приложение раскрывает конфиденциальные технические данные — такие как трассировки стека (stack traces), информация о схеме базы данных или внутренние пути к файлам — через свои сообщения об ошибках. Злоумышленники вызывают эти ошибки, отправляя некорректные входные данные (malformed input), запрашивая несуществующие ресурсы или провоцируя исключения на стороне сервера (server-side exceptions), чтобы составить карту базовой архитектуры приложения и выявить потенциальные векторы для дальнейшей эксплуатации (exploit). Такая утечка информации (information leakage) часто служит предвестником более серьезных атак, включая SQL-инъекции (SQL Injection) или обход пути (path traversal), предоставляя атакующему точные спецификации среды. Безопасная реализация должна возвращать общие, понятные пользователю сообщения, в то время как подробные диагностические данные должны сохраняться только в защищенных серверных логах.

### Использует ли приложение общий глобальный обработчик ошибок для необработанных исключений?
- [ ] Да — общие страницы ошибок **включены** и **не раскрывают** технических деталей  
- [ ] Да — используются кастомные страницы ошибок, но утечка **возможна** через специфические заголовки  
- [ ] Нет — серверные страницы ошибок по умолчанию (например, Tomcat, IIS, Nginx) **видны**  

### Можно ли перечислить техническую информацию с помощью некорректных входных данных или граничных условий?
- [ ] Нет — ошибки обрабатываются корректно с использованием уникальных идентификаторов (reference IDs) для логирования  
- [ ] Да — трассировки стека (stack traces) или SQL-запросы к базе данных **раскрываются** в ответах  
- [ ] Да — внутренние пути к файлам, переменные окружения или строки с версиями сервера **раскрываются**  

### Раскрывают ли ответы API подробные объекты ошибок или отладочную информацию?
- [ ] Нет — API возвращают стандартизированные коды ошибок и очищенные JSON-сообщения  
- [ ] Да — API возвращают подробные объекты отладки (debug objects) или полные детали исключений в теле ответа  
- [ ] Да — трассировки стека включены в поля `details` или `exception` JSON-ответа  

### Ведет ли себя приложение по-разному (на основе времени или содержимого) при возникновении ошибок?
- [ ] Нет — сигнатуры ответов и время обработки (timing) остаются согласованными независимо от типа ошибки  
- [ ] Да — дифференциальные ответы **могут** использоваться для перечисления (enumeration) валидных и невалидных состояний (например, перечисление пользователей (User Enumeration))
