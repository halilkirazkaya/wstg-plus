## WSTG-CRYP-02 — Тестирование на наличие Padding Oracle

Уязвимости типа Padding Oracle (Padding Oracle vulnerabilities) возникают, когда процесс дешифрования в приложении раскрывает информацию о том, является ли дополнение (Padding) расшифрованного шифротекста корректным или нет. Наблюдая за ответами по побочным каналам (Side-channel responses) — такими как различные коды состояния HTTP, специфические сообщения об ошибках или временные задержки ответов — злоумышленник может расшифровать данные, не зная ключа шифрования, и потенциально зашифровать произвольный открытый текст. Эта уязвимость обычно проявляется в зашифрованных сессионных файлах cookie, токенах аутентификации или параметрах URL, использующих блочные шифры в режиме сцепления блоков шифротекста (Cipher Block Chaining, CBC) с дополнением PKCS#7 (PKCS#7 padding). Эксплуатация позволяет полностью нарушить конфиденциальность и может привести к компрометации целостности путем создания поддельных зашифрованных блоков.

|---|---|

| **Статус теста** | Не выполнялся |

### Используется ли блочный шифр в режиме CBC для конфиденциальных данных?
- [ ] Нет — приложение использует аутентифицированное шифрование (Authenticated Encryption), например AES-GCM или ChaCha20-Poly1305.
- [ ] Да — приложение использует блочные шифры, но дополнение (Padding) **не требуется** (например, потоковые шифры или режим CTR).
- [ ] Да — приложение использует режим CBC, и дополнение **применяется**.

### Раскрывает ли сервер валидность дополнения через побочные каналы?
- [ ] Нет — сервер возвращает идентичные сообщения об ошибках и демонстрирует одинаковое время ответа.
- [ ] Да — сервер возвращает разные коды состояния HTTP (например, 200 против 500) при ошибках дополнения.
- [ ] Да — сервер возвращает специфические сообщения об ошибках (например, "Invalid Padding", "Decryption failed").
- [ ] Да — сервер демонстрирует измеряемые временные различия при сбое дешифрования.

### Возможна ли автоматизированная расшифровка шифротекста?
- [ ] Нет — побочные каналы **недостаточно** стабильны для эксплуатации.
- [ ] Да — шифротекст **может** быть частично расшифрован (последние несколько блоков).
- [ ] Да — полное восстановление открытого текста **возможно** с использованием таких инструментов, как `PadBuster`.

### Может ли злоумышленник выполнить атаку Bit-flipping или создать валидные шифротексты?
- [ ] Нет — проверки целостности (HMAC) проверяются **перед** дешифрованием.
- [ ] Да — проверка целостности отсутствует, но создание полезной нагрузки (Payload) **неосуществимо**.
- [ ] Да — создание произвольного шифротекста **возможно**, что позволяет выполнить повышение привилегий (Privilege Escalation) или инъекцию данных (Data Injection).
