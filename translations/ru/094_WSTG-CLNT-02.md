## WSTG-CLNT-02 — Тестирование выполнения JavaScript

Тестирование выполнения JavaScript (JavaScript Execution) ориентировано на выявление случаев, когда приложение неправильно обрабатывает данные, предоставленные пользователем, что приводит к выполнению несанкционированных сценариев в контексте браузера жертвы. Данная уязвимость обычно приводит к межсайтовому скриптингу (Cross-Site Scripting, XSS), который позволяет злоумышленникам извлекать сессионные куки (session cookies), манипулировать объектной моделью документа (Document Object Model, DOM) или выполнять действия от имени пользователя. Специалисты по тестированию на проникновение проверяют «приемники» (sinks), такие как `innerHTML`, `document.write()` и `eval()`, где могут обрабатываться неочищенные данные из таких «источников» (sources), как фрагменты URL, `localStorage` или `window.name`. Успешная эксплуатация (Exploit) нарушает целостность и конфиденциальность пользовательской сессии и может служить отправной точкой для более сложных атак на стороне клиента (client-side attacks).

### Обрабатывает ли приложение данные пользователя в опасных JavaScript-приемниках?
- [ ] Нет — все данные очищаются или используются в безопасных приемниках, таких как `textContent`  
- [ ] Да — данные обрабатываются в опасных приемниках, но применяется очистка (sanitization) или кодирование (encoding)  
- [ ] Да — данные обрабатываются в опасных приемниках, и очистка **не применяется**  

### Применяется ли политика безопасности контента (Content Security Policy, CSP) для предотвращения выполнения сценариев?
- [ ] Да — строгая CSP **включена**, и обход (bypass) **невозможен**  
- [ ] Да — CSP **включена**, но возможен обход через `unsafe-inline` или небезопасные CDN  
- [ ] Нет — CSP **не включена**  

### Может ли JavaScript выполняться через параметры или фрагменты URL (DOM-based XSS)?
- [ ] Нет — входные данные кодируются должным образом, и выполнение сценариев **невозможно**  
- [ ] Да — входные данные отражаются в DOM, но выполнение **предотвращается** встроенными механизмами защиты современных фреймворков  
- [ ] Да — входные данные напрямую выполняются в браузере, и эксплуатация **возможна**  

### Уязвимы ли обработчики событий или схемы URI для инъекции сценариев?
- [ ] Нет — входные данные очищаются для удаления атрибутов событий и схем `javascript:`  
- [ ] Да — обработчики событий (event handlers) **могут** быть внедрены, но фильтры ограничивают сложность полезной нагрузки (payload)  
- [ ] Да — возможно выполнение произвольного JavaScript через атрибуты `onerror`, `onload` или `href`
