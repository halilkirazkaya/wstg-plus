## WSTG-INPV-17 — Тестирование на инъекцию заголовка Host (Host Header Injection)

Инъекция заголовка Host (Host Header Injection) возникает в тех случаях, когда приложение доверяет предоставленному пользователем заголовку `Host` и включает его в серверную логику, такую как генерация ссылок или перенаправления, без надлежащей валидации. Злоумышленники манипулируют этим заголовком для реализации отравления веб-кеша (Web Cache Poisoning), отравления функции сброса пароля путем перенаправления конфиденциальных токенов на внешний домен или обхода механизмов контроля безопасности, основанных на маршрутизации по заголовкам. Успешная эксплуатация позволяет злоумышленнику эксфильтровать данные сессии, выполнять захват учетных записей (Account Takeover) или перенаправлять ничего не подозревающих пользователей на вредоносную инфраструктуру. Эта уязвимость чаще всего встречается в средах с балансировкой нагрузки или в приложениях, которые динамически генерируют абсолютные URL-адреса на основе контекста входящего запроса.


| Поле | Значение |
|---|---|
| **WSTG ID** | WSTG-INPV-17 |
| **CWE** | CWE-20 |
| **Статус теста** | Не выполнялся |
| **Уровень серьезности** | Средний / Высокий* |

> *Уровень серьезности становится высоким, если заголовок Host используется для генерации конфиденциальных ссылок в электронных письмах, например, для сброса пароля.

### Отображает ли приложение значение заголовка `Host` в ссылках, перенаправлениях или скриптах?
- [ ] Нет — приложение использует жестко заданные (hardcoded) базовые URL-адреса или строго валидированную конфигурацию  
- [ ] Да — заголовок `Host` отображается, но строго проверяется по белому списку (Whitelist)  
- [ ] Да — заголовок `Host` отображается, и обход **возможен** с помощью некорректно сформированных значений (например, `expected.com:attacker.com`)  
- [ ] Да — заголовок `Host` отображается **без** какой-либо валидации  

### Обрабатываются ли приложением или инфраструктурой альтернативные заголовки, связанные с хостом?
- [ ] Нет — такие заголовки, как `X-Forwarded-Host`, `X-Host` или `Forwarded`, игнорируются  
- [ ] Да — альтернативные заголовки обрабатываются, но **не** переопределяют внутреннюю логику  
- [ ] Да — альтернативные заголовки **могут** быть использованы для переопределения основного значения заголовка `Host`  

### Можно ли манипулировать заголовком `Host` для отравления системных электронных писем (например, при сбросе пароля)?
- [ ] Нет — ссылки в письмах используют статический доверенный домен из конфигурации сервера  
- [ ] Да — заголовок `Host` влияет на ссылки в письмах, но валидацию **невозможно** обойти  
- [ ] Да — заголовком `Host` **можно** манипулировать для перенаправления токенов сброса пароля на домен, контролируемый злоумышленником *(Critical)*  

### Подвержено ли приложение отравлению веб-кеша (Web Cache Poisoning) через заголовок `Host`?
- [ ] Нет — механизм кеширования отсутствует или заголовок `Host` является частью ключа кеширования (Cache key)  
- [ ] Да — кеш присутствует, но он строго валидирует или игнорирует заголовок `Host` для параметров, не входящих в ключ кеширования (Unkeyed inputs)  
- [ ] Да — кеш существует и **может** быть отравлен путем инъекции заголовка `Host` для доставки вредоносного контента другим пользователям  

### Принимает ли сервер произвольные заголовки `Host` для маршрутизации виртуальных хостов?
- [ ] Нет — сервер возвращает ошибку или страницу по умолчанию для нераспознанных заголовков `Host`  
- [ ] Да — сервер принимает любой заголовок `Host`, что **полезно** для разведки (Reconnaissance) или перечисления внутренних сервисов (Internal service enumeration)
