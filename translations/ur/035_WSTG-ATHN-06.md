## WSTG-ATHN-06 — براؤزر کیشے کی کمزوری کی جانچ (Testing for Browser Cache Weakness)

براؤزر کیشے کی کمزوری (Browser cache weakness) اس وقت پیدا ہوتی ہے جب حساس معلومات (Sensitive Information) مقامی براؤزر کیشے میں محفوظ ہو جاتی ہیں اور اسی جسمانی مشین تک رسائی رکھنے والا کوئی غیر مجاز صارف (Unauthorized User) انہیں دوبارہ حاصل کر سکتا ہے۔ مناسب کیشے کنٹرول ہیڈرز (Cache Control Headers) کی کمی کی وجہ سے ممکنہ طور پر حساس ڈیٹا، جیسے کہ ذاتی معلومات، اکاؤنٹ کی تفصیلات، یا سیشن آئیڈینٹی فائرز (Session Identifiers)، صارف کے لاگ آؤٹ (Logout) کرنے یا سیشن بند کرنے کے بعد بھی برقرار رہ سکتے ہیں۔ حملہ آور یا مشترکہ یا عوامی ٹرمینلز پر آنے والے بعد کے صارفین براؤزر کی ہسٹری میں جا کر، "Back" بٹن استعمال کر کے، یا مقامی کیشے فائلوں کا معائنہ کر کے کیشے شدہ جوابات سے ڈیٹا نکال کر اس کا فائدہ اٹھا سکتے ہیں۔ تصدیق شدہ مواد (Authenticated Content) کو سنبھالنے والی کسی بھی ایپلی کیشن کے لیے `Cache-Control: no-store` اور `Pragma: no-cache` جیسی سخت ہدایات پر عمل درآمد کرنا انتہائی اہم ہے تاکہ یہ یقینی بنایا جا سکے کہ حساس ڈیٹا کبھی بھی ڈسک پر نہ لکھا جائے۔


| فیلڈ | قدر |
|---|---|
| **WSTG ID** | WSTG-ATHN-06 |
| **CWE** | CWE-525 |
| **ٹیسٹ کی صورتحال** | انجام نہیں دیا گیا (Not Performed) |
| **شدت** | کم / درمیانی* |

> *شدت (Severity) اس صورت میں 'درمیانی' ہو جاتی ہے اگر ایپلی کیشن کا استعمال کثرت سے عوامی ٹرمینلز (Public Kiosks) سے کیا جائے یا اس میں انتہائی حساس PII/PHI ڈیٹا موجود ہو۔

### کیا تصدیق شدہ یا حساس صفحات پر مناسب کیشے کنٹرول ہیڈرز موجود ہیں؟
- [ ] ہاں — `Cache-Control: no-store, no-cache` اور `Pragma: no-cache` **موجود** ہیں اور **صحیح طریقے سے کنفیگر** کیے گئے ہیں  
- [ ] ہاں — کچھ ہیڈرز موجود ہیں لیکن `no-store` **مفقود** ہے، جس سے ڈسک کیشنگ (Disk Caching) کی اجازت مل رہی ہے  
- [ ] نہیں — کیشے سے متعلق کوئی ہیڈرز **نہیں لگائے گئے**، اور براؤزر کے پہلے سے طے شدہ رویے پر انحصار کیا گیا ہے  

### کیا ایپلی کیشن صارف کے مخصوص ڈیٹا کے لیے `private` ڈائریکٹو استعمال کرتی ہے؟
- [ ] ہاں — تمام ذاتی نوعیت کے مواد کے لیے `Cache-Control: private` **فعال** ہے تاکہ پراکسی کیشنگ (Proxy Caching) کو روکا جا سکے  
- [ ] نہیں — مواد کو `public` قرار دیا گیا ہے یا اس میں `private` ڈائریکٹو کی کمی ہے، جس کی وجہ سے درمیانی پراکسیز اسے **کیشے** کر سکتی ہیں  

### کیا کامیاب لاگ آؤٹ کے بعد براؤزر کے "Back" بٹن کے ذریعے حساس معلومات تک رسائی ممکن ہے؟
- [ ] نہیں — براؤزر دوبارہ تصدیق (Re-authentication) کا مطالبہ کرتا ہے یا صفحہ مقامی کیشے سے **لوڈ نہیں ہوتا**  
- [ ] ہاں — سیشن ختم ہونے کے بعد بھی "Back" بٹن کے ذریعے حساس معلومات **اب بھی دیکھی جا سکتی ہیں**  

### کیا حساس غیر ایچ ٹی ایم ایل فائلیں (مثلاً PDFs، CSV ایکسپورٹس، JSON API رسپانسز) کیشنگ سے محفوظ ہیں؟
- [ ] نہیں — ایپلی کیشن کے ذریعے کوئی حساس غیر ایچ ٹی ایم ایل (Non-HTML) فائلیں تیار یا ہینڈل نہیں کی جاتی ہیں  
- [ ] ہاں — تمام حساس فائل ڈاؤن لوڈز اور اے پی آئی (API) اینڈ پوائنٹس پر سخت کیشے کنٹرول ہیڈرز **لاگو** کیے گئے ہیں  
- [ ] نہیں — حساس ڈاؤن لوڈز یا API رسپانسز مقامی براؤزر کیشے یا عارضی ڈائریکٹریز میں **محفوظ** ہو رہے ہیں  

### کیا ایپلی کیشن پرانے ڈیٹا کے استعمال کو روکنے کے لیے `Expires: 0` یا ماضی کی کوئی تاریخ استعمال کرتی ہے؟
- [ ] ہاں — `Expires` ہیڈر کو `0` یا کسی پرانی تاریخ پر سیٹ کیا گیا ہے، جو یہ **یقینی** بناتا ہے کہ براؤزر مواد کو فوری طور پر پرانا (Stale) تصور کرے  
- [ ] نہیں — حساس صفحات کے لیے `Expires` ہیڈر **موجود نہیں** ہے یا مستقبل کی تاریخ پر سیٹ ہے
