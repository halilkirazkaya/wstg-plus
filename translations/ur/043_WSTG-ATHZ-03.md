## WSTG-ATHZ-03 — مراعات میں اضافے کی جانچ (Testing for Privilege Escalation)

مراعات میں اضافہ (Privilege Escalation) اس وقت ہوتا ہے جب کوئی حملہ آور ایسی کمزوریوں کا فائدہ اٹھاتا ہے جن کے ذریعے وہ ان وسائل یا افعال تک رسائی حاصل کر لیتا ہے جو عام طور پر اعلیٰ سطح کے اختیارات یا مختلف شناختوں والے صارفین کے لیے مخصوص ہوتے ہیں۔ عمودی اضافے (Vertical Escalation) میں، ایک عام صارف انتظامی افعال (Administrative functions) تک رسائی حاصل کرنے کی کوشش کرتا ہے، جبکہ افقی اضافے (Horizontal Escalation) میں اسی مراعات کی سطح رکھنے والے کسی دوسرے صارف کے ڈیٹا تک رسائی حاصل کرنا شامل ہے۔ یہ خامیاں عام طور پر ناقص طور پر لاگو کی گئی رسائی کے کنٹرول کی فہرستوں (ACLs)، غیر محفوظ براہ راست آبجیکٹ حوالہ جات (IDOR)، یا سیشن یا شناختی ٹوکنز کے اندر پیرامیٹر میں چھیڑ چھاڑ (Parameter Tampering) کی وجہ سے پیدا ہوتی ہیں۔ حملہ آور کے نقطہ نظر سے، یہ اکثر عددی آئی ڈیز (Numeric IDs) میں ہیرا پھیری، کوکیز یا جے ڈبلیو ٹی (JWT) میں کردار پر مبنی پیرامیٹرز کی تبدیلی، یا ایسے پوشیدہ اے پی آئی اینڈ پوائنٹس (API endpoints) کو براہ راست کال کرنے سے حاصل کیا جاتا ہے جن میں سرور سائڈ پر اجازت دہی کی جانچ (Authorization checks) موجود نہ ہو۔

### کیا ایپلی کیشن متعدد مراعات کی سطحیں یا ملٹی ٹیننسی (Multi-tenancy) نافذ کرتی ہے؟
- [ ] نہیں — ایپلی کیشن ایک واحد صارف یا واحد کردار کا نظام ہے  
- [ ] ہاں — متعدد کرداروں یا ٹیننٹس (Tenants) کی تعریف کی گئی ہے اور ان کے لیے اجازت دہی کی جانچ درکار ہے  

### کیا افقی طور پر مراعات میں اضافہ (Horizontal privilege escalation) ممکن ہے؟
- [ ] نہیں — تمام وسائل کے شناخت کنندگان اور سیشن مالکان پر اجازت دہی کی جانچ (Authorization checks) لاگو کی جاتی ہے  
- [ ] ہاں — IDOR یا پیرامیٹر میں چھیڑ چھاڑ کے ذریعے دوسرے صارفین کے ڈیٹا تک رسائی ممکن ہے  
- [ ] ہاں — ڈیٹا کا اخراج (Data leakage) ممکن ہے لیکن دوسرے صارفین کے ڈیٹا میں ترمیم ممکن نہیں ہے  

### کیا عمودی طور پر مراعات میں اضافہ (Vertical privilege escalation) ممکن ہے؟
- [ ] نہیں — انتظامی اینڈ پوائنٹس (Administrative endpoints) سختی سے کردار پر مبنی رسائی کے کنٹرول (RBAC) کو نافذ کرتے ہیں  
- [ ] ہاں — کم مراعات یافتہ صارفین براہ راست یو آر ایل (URL) تک رسائی کے ذریعے انتظامی افعال تک رسائی حاصل کر سکتے ہیں  
- [ ] ہاں — کردار سے متعلق ہیڈرز، کوکیز، یا JWT کلیمز میں ہیرا پھیری کر کے انتظامی افعال تک رسائی حاصل کی جا سکتی ہے  

### کیا صارف کے کردار یا اجازتوں کو ماس اسائنمنٹ (Mass Assignment) یا پیرامیٹر پولیوشن (Parameter Pollution) کے ذریعے تبدیل کیا جا سکتا ہے؟
- [ ] نہیں — کردار پر مبنی فیلڈز صرف پڑھنے کے لیے (Read-only) ہیں اور صارفین ان میں ترمیم نہیں کر سکتے  
- [ ] ہاں — پروفائل اپ ڈیٹس یا رجسٹریشن کے دوران پوشیدہ پیرامیٹرز (مثلاً `role=admin`) شامل کر کے صارف کے اختیارات میں اضافہ کیا جا سکتا ہے  

### کیا تمام API ورژنز اور HTTP میتھڈز میں اجازت دہی کی جانچ مستقل مزاجی سے نافذ ہے؟
- [ ] ہاں — تمام ورژنز اور میتھڈز میں کنٹرولز مستقل طور پر لاگو ہیں  
- [ ] نہیں — پرانے API ورژنز (مثلاً `/v1/`) یا مخصوص HTTP میتھڈز (مثلاً `PUT`, `DELETE`, `PATCH`) اجازت دہی کو نظر انداز (Bypass) کرتے ہیں  
- [ ] نہیں — انتظامی افعال UI میں پوشیدہ ہیں لیکن API کی سطح پر تمام صارفین کے لیے فعال (Enabled) ہیں
