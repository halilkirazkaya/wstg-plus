## WSTG-APIT-02 — API Broken Object Level Authorization

ناقص آبجیکٹ لیول آتھورائزیشن (Broken Object Level Authorization - BOLA)، جسے غیر محفوظ براہ راست آبجیکٹ حوالہ (Insecure Direct Object Reference - IDOR) بھی کہا جاتا ہے، اس وقت پیش آتی ہے جب ایک اے پی آئی (API) اس بات کی توثیق (Validation) کرنے میں ناکام رہتی ہے کہ آیا صارف کے پاس کسی آئی ڈی (ID) کے ذریعے شناخت شدہ مخصوص ریسورس (Resource) تک رسائی حاصل کرنے یا اس میں ہیرا پھیری کرنے کے مناسب اختیارات موجود ہیں یا نہیں۔ حملہ آور ریکویسٹ پاتھس (Request paths)، کوئری پیرامیٹرز (Query parameters)، یا JSON bodies میں موجود آئیڈینٹیفائرز (Identifiers) — جیسے عددی آئی ڈیز (Numeric IDs) یا UUIDs — کو منظم طریقے سے شمار (Enumerate) کر کے یا اندازہ لگا کر دوسرے صارفین کے ڈیٹا تک رسائی حاصل کرنے کے لیے اس کمزوری کا فائدہ اٹھاتے ہیں۔ یہ سیکیورٹی رسک جدید API سیکیورٹی میں سب سے عام اور اثر انگیز مسئلہ ہے، جو بڑے پیمانے پر ڈیٹا کے اخراج (Data exfiltration)، غیر مجاز ترمیم، یا مکمل اکاؤنٹ ٹیک اوور (Account takeover) کا باعث بن سکتا ہے۔ حملہ آور کا مقصد ایسے اینڈ پوائنٹس (Endpoints) کی نشاندہی کرنا ہوتا ہے جو آبجیکٹ آئیڈینٹیفائرز کو قبول کرتے ہیں اور یہ جانچنا ہوتا ہے کہ آیا سرور سائیڈ آتھورائزیشن لاجک (Server-side authorization logic) موجود نہیں ہے یا ان آئی ڈیز میں ہیرا پھیری کر کے اسے بائی پاس (Bypass) کیا جا سکتا ہے۔

### کیا اے پی آئی ریکویسٹ کے اندر آبجیکٹ آئیڈینٹیفائرز (Object identifiers) قابل قیاس یا شمار کے قابل (Enumerable) ہیں؟
- [ ] نہیں — آئیڈینٹیفائرز طویل، بے ترتیب (Random)، اور کرپٹوگرافک طور پر محفوظ ہیں (مثلاً، UUIDv4)  
- [ ] ہاں — آئیڈینٹیفائرز مسلسل اعداد کی صورت میں ہیں (مثلاً، `101`, `102`)  
- [ ] ہاں — آئیڈینٹیفائرز ایک قابل قیاس پیٹرن کی پیروی کرتے ہیں یا عوامی معلومات سے اخذ کیے گئے ہیں  

### کیا اے پی آئی ہر ریکویسٹ کے لیے آبجیکٹ کی ملکیت کی سرور سائیڈ توثیق (Server-side validation) کرتی ہے؟
- [ ] ہاں — آتھورائزیشن چیکس (Authorization checks) ہر ریکویسٹ کے لیے لاگو ہیں اور بائی پاس ممکن نہیں ہے  
- [ ] ہاں — چیکس لاگو تو ہیں لیکن پیرامیٹر پولوشن (Parameter pollution) یا میتھڈ ٹنلنگ (Method tunneling) کے ذریعے بائی پاس **ممکن ہے**  
- [ ] نہیں — ایپلی کیشن مخصوص ریسورس کی ملکیت چیک کیے بغیر صرف صارف کے تصدیق شدہ (Authenticated) ہونے پر بھروسہ کرتی ہے  

### کیا آئیڈینٹیفائر تبدیل کر کے کسی دوسرے صارف کے ریسورس تک رسائی حاصل کرنا یا اس میں ترمیم کرنا ممکن ہے؟
- [ ] نہیں — دوسرے صارفین کے ریسورسز تک غیر مجاز رسائی **ممکن نہیں ہے**  
- [ ] ہاں — غیر مجاز **پڑھنے** کی رسائی (Horizontal BOLA) **ممکن ہے**  
- [ ] ہاں — غیر مجاز **ترمیم** یا **حذف** کرنے کی رسائی (Horizontal BOLA) **ممکن ہے**  

### کیا اے پی آئی آئی ڈیز کو تبدیل کر کے ایڈمنسٹریٹو یا سسٹم لیول کے آبجیکٹس تک رسائی کی اجازت دیتی ہے؟
- [ ] نہیں — ایڈمنسٹریٹو ریسورسز سیکنڈری آتھورائزیشن لیئرز کے ذریعے محفوظ ہیں  
- [ ] ہاں — سسٹم لیول کے ریسورسز تک رسائی یا ترمیم (Vertical BOLA) **ممکن ہے**  

### کیا آئیڈینٹیفائر کو ریکویسٹ کے کسی دوسرے حصے میں منتقل کر کے آتھورائزیشن چیک کو بائی پاس کیا جا سکتا ہے؟
- [ ] نہیں — پیرامیٹر کی جگہ سے قطع نظر آتھورائزیشن لاجک مستقل (Consistent) ہے  
- [ ] ہاں — بائی پاس **ممکن ہے** جب آئی ڈی کو URL path سے JSON body یا ہیڈرز میں منتقل کیا جائے  
- [ ] ہاں — بائی پاس **ممکن ہے** جب ایک سے زیادہ آئی ڈیز فراہم کی جائیں اور سرور غیر مجاز آئی ڈی پر کارروائی کرے
