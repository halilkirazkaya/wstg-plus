## WSTG-CONF-06 — HTTP طریقوں کا ٹیسٹ (Test HTTP Methods)

ایچ ٹی ٹی پی طریقوں (HTTP Methods) کے ٹیسٹ میں یہ شناخت کرنا شامل ہے کہ ویب سرور (Web Server) اور ایپلی کیشن کن افعال (Verbs) کو سپورٹ کرتے ہیں تاکہ اس بات کو یقینی بنایا جا سکے کہ صرف ضروری فنکشنلٹی ہی کھلی ہے۔ معیاری `GET` اور `POST` کے علاوہ، سرورز نادانستہ طور پر خطرناک طریقے جیسے کہ `PUT` ، `DELETE` ، یا `TRACE` کو فعال کر سکتے ہیں، جو حملہ آوروں کو نقصان دہ فائلیں اپ لوڈ کرنے، موجودہ مواد کو حذف کرنے، یا کراس سائٹ ٹریسنگ (Cross-Site Tracing - XST) کے ذریعے سیشن کوکیز (Session Cookies) نکالنے کی اجازت دے سکتے ہیں۔ پینٹیسٹرز (Pentesters) رسپانس ہیڈرز (Response Headers) جیسے کہ `Allow` یا `Public` کا معائنہ کرتے ہیں اور غیر مجاز فنکشنلٹی تک رسائی حاصل کرنے کے لیے میتھڈ اوور رائڈنگ (Method Overriding) تکنیکوں یا ورب ٹیمپرنگ (Verb Tampering) کا استعمال کرتے ہوئے پابندیوں کو بائی پاس کرنے کی کوشش کرتے ہیں۔ یہ ٹیسٹ اٹیک سرفیس (Attack Surface) کو مضبوط بنانے اور ان غلط کنفیگریشنز (Misconfigurations) کو روکنے کے لیے انتہائی اہم ہے جو سرور کے کمپرومائز (Server Compromise) یا غیر مجاز ڈیٹا ہیرا پھیری (Unauthorized Data Manipulation) کا باعث بن سکتے ہیں۔


| فیلڈ | ویلیو |
|---|---|
| **WSTG ID** | WSTG-CONF-06 |
| **CWE** | CWE-650 |
| **ٹیسٹ کی صورتحال** | انجام نہیں دیا گیا |
| **سنگینی** | کم / درمیانی* |

> *اگر `PUT` یا `DELETE` غیر مجاز فائل ہیرا پھیری کی اجازت دیں یا اگر `TRACE` سیشن ٹوکن (Session Token) نکالنے میں سہولت فراہم کرے تو سنگینی زیادہ (High) ہو جاتی ہے۔

### کیا پوری ایپلی کیشن میں `PUT` یا `DELETE` جیسے خطرناک HTTP طریقے غیر فعال ہیں؟
- [ ] جی ہاں — صرف محفوظ طریقے (GET, POST, HEAD) **فعال ہیں**  
- [ ] نہیں — `PUT` یا `DELETE` **فعال ہیں** لیکن ان کے لیے درست تصدیق (Authentication) درکار ہے  
- [ ] نہیں — `PUT` یا `DELETE` **فعال ہیں** اور تصدیق کے بغیر قابل رسائی ہیں *(انتہائی تشویشناک)*  

### کیا کراس سائٹ ٹریسنگ (XST) کو روکنے کے لیے `TRACE` طریقہ غیر فعال ہے؟
- [ ] جی ہاں — `TRACE` طریقہ **غیر فعال ہے** یا 405 Method Not Allowed واپس کرتا ہے  
- [ ] نہیں — `TRACE` طریقہ **فعال ہے** لیکن حساس ہیڈرز کو ظاہر (Reflect) نہیں کرتا  
- [ ] نہیں — `TRACE` طریقہ **فعال ہے** اور `Cookie` یا `Authorization` ہیڈرز کو ظاہر کرتا ہے *(درمیانی)*  

### کیا سرور کے ذریعے HTTP میتھڈ اوور رائڈنگ (مثلاً `X-HTTP-Method-Override`) سپورٹ کی جاتی ہے؟
- [ ] نہیں — سرور میتھڈ اوور رائڈ ہیڈرز پر **کارروائی نہیں کرتا** ہے  
- [ ] جی ہاں — سرور اوور رائڈ ہیڈرز پر کارروائی کرتا ہے لیکن ایکسیس کنٹرولز (Access Controls) **نافذ ہیں**  
- [ ] جی ہاں — سرور اوور رائڈ ہیڈرز پر کارروائی کرتا ہے اور ایکسیس کنٹرول بائی پاس **ممکن ہے**  

### کیا سرور من مانے یا غلط شکل والے (Malformed) HTTP طریقوں کا محفوظ طریقے سے جواب دیتا ہے؟
- [ ] جی ہاں — سرور 405 Method Not Allowed یا 501 Not Implemented واپس کرتا ہے  
- [ ] نہیں — سرور `JEFF` یا `TEST` جیسے من مانے طریقوں کے لیے 200 OK یا 500 Error واپس کرتا ہے  
- [ ] نہیں — من مانے طریقوں کا استعمال تصدیق یا اجازت (Authorization) کے فلٹرز کو بائی پاس کرنے کی اجازت دیتا ہے  

### کیا معلومات کے افشاء (Information Disclosure) کو کم کرنے کے لیے `OPTIONS` طریقے کو محدود یا کنفیگر کیا گیا ہے؟
- [ ] جی ہاں — `OPTIONS` غیر فعال ہے یا ایک معمولی `Allow` ہیڈر واپس کرتا ہے  
- [ ] نہیں — `OPTIONS` فعال طریقوں کی ایک وسیع رینج ظاہر کرتا ہے، بشمول DAV یا ڈیبگنگ وربز (Debugging Verbs)
