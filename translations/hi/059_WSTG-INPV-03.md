## WSTG-INPV-03 — HTTP वर्ब टैम्परिंग के लिए परीक्षण (Testing for HTTP Verb Tampering)

HTTP वर्ब टैम्परिंग (HTTP Verb Tampering) उन कमजोरियों का फायदा उठाती है जो वेब सर्वर (Web Server) और एप्लीकेशन फ्रेमवर्क (Application Framework) द्वारा उपयोग किए गए HTTP मेथड (HTTP Method) के आधार पर विशिष्ट संसाधनों (Resources) तक पहुंच को अधिकृत करने के तरीके में होती हैं। हमलावर मानक मेथड जैसे `GET` या `POST` को `HEAD`, `PUT`, `OPTIONS`, या यहां तक कि मनमाने, गैर-मानक स्ट्रिंग्स (Strings) से बदलकर सुरक्षा प्रतिबंधों (Security Constraints) को बायपास (Bypass) करने का प्रयास करते हैं, जिन्हें बैकएंड (Backend) असंगत रूप से संसाधित कर सकता है। यह भेद्यता (Vulnerability) आमतौर पर तब होती है जब सुरक्षा कॉन्फ़िगरेशन (Configuration)—जैसे Java EE web.xml फ़िल्टर या .NET ऑथराइजेशन नियम—स्पष्ट रूप से अनुमत मेथड की सूची तो देते हैं लेकिन अन्य पर विचार करने में विफल रहते हैं, या जब "डिनाय-बाय-मेथड" (deny-by-method) दृष्टिकोण का उपयोग "डिनाय-ऑल" (deny-all) के बजाय किया जाता है। सफल एक्सप्लॉइट (Exploit) के परिणामस्वरूप प्रशासनिक कार्यों तक अनधिकृत पहुंच, डेटा संशोधन, या सर्वर के आंतरिक कॉन्फ़िगरेशन के संबंध में सूचना प्रकटीकरण (Information Disclosure) हो सकता है।

### क्या एप्लीकेशन गैर-मानक या मनमाने HTTP मेथड पर प्रतिक्रिया देती है?
- [ ] नहीं — एप्लीकेशन स्पष्ट रूप से आवश्यक मेथड को छोड़कर सभी को अस्वीकार करती है
- [ ] हाँ — एप्लीकेशन मानक मेथड (OPTIONS, TRACE) पर प्रतिक्रिया देती है लेकिन सुरक्षा को बायपास नहीं कर सकती
- [ ] हाँ — एप्लीकेशन मनमाने स्ट्रिंग्स (जैसे, FOO, TEST) को स्वीकार करती है और उन्हें `GET` या `POST` अनुरोधों के रूप में संसाधित करती है

### क्या HTTP मेथड बदलकर प्रमाणीकरण/प्राधिकरण (Authentication/Authorization) को बायपास किया जा सकता है?
- [ ] नहीं — सुरक्षा नियंत्रण HTTP वर्ब की परवाह किए बिना वैश्विक स्तर पर लागू किए जाते हैं
- [ ] हाँ — नियंत्रण लागू हैं लेकिन `HEAD` मेथड का उपयोग करके बायपास करना संभव है
- [ ] हाँ — सुरक्षा नियंत्रण वैकल्पिक मेथड पर लागू नहीं होते हैं, जिससे प्रतिबंधित एंडपॉइंट्स (Endpoints) तक अनधिकृत पहुंच की अनुमति मिलती है

### क्या सर्वर पर `PUT` या `DELETE` जैसे खतरनाक मेथड सक्षम हैं?
- [ ] नहीं — खतरनाक मेथड अक्षम (Disabled) हैं या 405 Method Not Allowed लौटाते हैं
- [ ] हाँ — मेथड सक्षम हैं लेकिन वैध, उच्च-विशेषाधिकार प्राप्त प्रमाणीकरण की आवश्यकता होती है
- [ ] हाँ — मेथड सक्षम हैं और अनधिकृत फ़ाइल अपलोड या संसाधन हटाने की अनुमति देते हैं *(गंभीर)*

### क्या `OPTIONS` मेथड अनुमत वर्ब के बारे में संवेदनशील जानकारी प्रकट करता है?
- [ ] नहीं — `OPTIONS` अक्षम है या जेनेरिक प्रतिक्रिया लौटाता है
- [ ] हाँ — `OPTIONS` `Allow` हेडर लौटाता है लेकिन प्रतिबंधित आंतरिक मेथड को उजागर नहीं करता है
- [ ] हाँ — `OPTIONS` केवल-आंतरिक या प्रशासनिक मेथड को प्रकट करता है जो आगे के एक्सप्लॉइट (Exploit) में सहायता करते हैं

### क्या `TRACE` मेथड सक्षम है, जो संभावित रूप से क्रॉस-साइट ट्रेसिंग (Cross-Site Tracing - XST) की अनुमति देता है?
- [ ] नहीं — `TRACE` और `TRACK` मेथड अक्षम हैं
- [ ] हाँ — `TRACE` सक्षम है और संवेदनशील कुकीज़/टोकन सहित HTTP हेडर को वापस दर्शाता है
