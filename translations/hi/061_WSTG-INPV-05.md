## WSTG-INPV-05 — SQL इंजेक्शन (SQL Injection)

SQL इंजेक्शन (SQL Injection) तब होता है जब उपयोगकर्ता द्वारा दिए गए इनपुट को उचित पैरामीटराइजेशन (Parameterization) या सैनिटाइजेशन (Sanitization) के बिना SQL क्वेरी में शामिल किया जाता है, जिससे हमलावरों को क्वेरी लॉजिक में हेरफेर करने की अनुमति मिलती है। सफल एक्सप्लॉइटेशन (Exploitation) के कारण अनधिकृत डेटा एक्सफिल्ट्रेशन (Data exfiltration), ऑथेंटिकेशन बाईपास (Authentication bypass), डेटा संशोधन और कुछ मामलों में `xp_cmdshell` या `LOAD_FILE()` जैसी डेटाबेस सुविधाओं के माध्यम से रिमोट कोड एक्जीक्यूशन (Remote code execution) हो सकता है। यह वल्नरेबिलिटी (Vulnerability) आमतौर पर लॉगिन फॉर्म, सर्च फंक्शनलिटी, REST एपीआई (API) पैरामीटर्स, HTTP हेडर्स और किसी भी ऐसे एंडपॉइंट को प्रभावित करती है जो उपयोगकर्ता-नियंत्रित इनपुट से डायनेमिक SQL क्वेरी बनाता है। एक हमलावर के दृष्टिकोण से, इंजेक्शन पॉइंट की पहचान करना डेटाबेस को पूरी तरह से कॉम्प्रोमाइज करने और बुनियादी ढांचे के भीतर संभावित लेटरल मूवमेंट के लिए प्राथमिक प्रवेश द्वार है।

### क्या यूज़र-कंट्रोलेबल पैरामीटर्स को सुरक्षित डेटाबेस एक्सेस विधियों का उपयोग करके प्रोसेस किया जाता है?
- [ ] हाँ — एप्लिकेशन विशेष रूप से ORM या पैरामीटराइज्ड क्वेरी का उपयोग करता है और बाईपास **संभव नहीं है**  
- [ ] हाँ — पैरामीटराइज्ड क्वेरी का उपयोग किया जाता है लेकिन एज केसेस (जैसे, `ORDER BY` क्लॉज) के माध्यम से बाईपास **संभव है**  
- [ ] नहीं — पैरामीटराइजेशन के **बिना** SQL क्वेरी बनाने के लिए रॉ स्ट्रिंग कॉनकैटिनेशन का उपयोग किया जाता है  

### क्या SQL इंजेक्शन के माध्यम से ऑथेंटिकेशन तंत्र को दरकिनार किया जा सकता है?
- [ ] नहीं — लॉगिन पैरामीटर्स इंजेक्शन के प्रति **संवेदनशील नहीं** हैं  
- [ ] हाँ — टॉटोलॉजी-आधारित पेलोड (Payload) (जैसे, `' OR 1=1 --`) का उपयोग करके ऑथेंटिकेशन बाईपास **संभव है**  

### क्या इन-बैंड (In-band) या आउट-ऑफ़-बैंड (Out-of-band) तकनीकों के माध्यम से डेटा एक्सफिल्ट्रेशन संभव है?
- [ ] नहीं — किसी डेटा एक्सफिल्ट्रेशन की पहचान नहीं हुई  
- [ ] हाँ — UNION-आधारित या एरर-आधारित एक्सफिल्ट्रेशन **संभव है**  
- [ ] हाँ — ब्लाइंड (बूलियन या टाइम-आधारित) एक्सफिल्ट्रेशन **संभव है**  
- [ ] हाँ — आउट-ऑफ़-बैंड (DNS/HTTP) एक्सफिल्ट्रेशन **संभव है**  

### क्या एप्लिकेशन फंक्शन सेकंड-ऑर्डर SQL इंजेक्शन (Second-order SQL injection) वल्नरेबिलिटी प्रदर्शित करते हैं?
- [ ] नहीं — बाद की क्वेरी के लिए प्राप्त किए जाने पर स्टोर किए गए डेटा को सुरक्षित रूप से संभाला जाता है  
- [ ] हाँ — उपयोगकर्ता इनपुट स्टोर किया जाता है और बाद में बिना वैलिडेशन के असुरक्षित SQL क्वेरी में उपयोग किया जाता है  

### क्या डेटाबेस सर्विस अकाउंट के विशेषाधिकार आवश्यक न्यूनतम तक सीमित हैं?
- [ ] हाँ — सर्विस अकाउंट के पास **न्यूनतम विशेषाधिकार (Least privilege)** हैं (जैसे, विशिष्ट टेबल/कार्यों तक सीमित)  
- [ ] नहीं — सर्विस अकाउंट के पास उन्नत विशेषाधिकार हैं (जैसे, `DROP TABLE`, `FILE` विशेषाधिकार, या `xp_cmdshell` **सक्षम** है)
