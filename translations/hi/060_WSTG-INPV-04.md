## WSTG-INPV-04 — HTTP पैरामीटर पलूशन (HTTP Parameter Pollution - HPP) के लिए टेस्टिंग

HTTP पैरामीटर पलूशन (HTTP Parameter Pollution - HPP) तब होता है जब एक एप्लीकेशन एक ही नाम वाले कई HTTP पैरामीटर प्राप्त करती है और उन्हें असंगत या असुरक्षित तरीके से प्रोसेस करती है। डुप्लिकेट पैरामीटर प्रदान करके, एक हमलावर एप्लीकेशन के इंटरनल लॉजिक में हेरफेर कर सकता है, जिससे संभावित रूप से वेब एप्लीकेशन फ़ायरवॉल (Web Application Firewalls - WAF), इनपुट वैलिडेशन (Input Validation) फिल्टर, या एक्सेस कंट्रोल मैकेनिज्म (Access Control Mechanisms) को बायपास किया जा सकता है। यह भेद्यता (Vulnerability) काफी हद तक अंतर्निहित वेब सर्वर (Web Server) या एप्लीकेशन फ्रेमवर्क (Application Framework) पर निर्भर करती है, जो दोहराए गए पैरामीटर्स में से पहले, आखिरी या कॉनकेटनेटेड (Concatenated) संस्करण को चुन सकते हैं। इसका एक्सप्लोइटेशन (Exploitation) आमतौर पर उन एंडपॉइंट्स (Endpoints) को लक्षित करता है जो बैक-एंड एपीआई (Back-end APIs), डेटाबेस क्वेरी (Database queries), या URL रीडायरेक्शन (URL redirection) मैकेनिज्म को पैरामीटर पास करते हैं, जिसके परिणामस्वरूप क्रॉस-साइट स्क्रिप्टिंग (Cross-Site Scripting - XSS) से लेकर अनधिकृत डेटा एक्सफिल्ट्रेशन (Data exfiltration) तक के प्रभाव हो सकते हैं।

### एप्लीकेशन एक ही नाम के कई HTTP पैरामीटर्स को कैसे संभालती है?
- [ ] नहीं — डुप्लिकेट पैरामीटर्स को सर्वर द्वारा रिजेक्ट (Rejected) या इग्नोर कर दिया जाता है
- [ ] हाँ — सर्वर लगातार केवल पहले या आखिरी अकरेंस (Occurrence) को चुनता है
- [ ] हाँ — सर्वर वैल्यूज को कॉनकेटनेट (Concatenate) करता है, जो संभावित रूप से इंजेक्शन (Injection) की अनुमति देता है

### क्या HPP का उपयोग WAF या इनपुट फिल्टर जैसे सुरक्षा नियंत्रणों को बायपास करने के लिए किया जा सकता है?
- [ ] नहीं — सुरक्षा नियंत्रण पैरामीटर के सभी अकरेंस की सही ढंग से जांच करते हैं
- [ ] हाँ — एक WAF या फिल्टर केवल पहले अकरेंस की जांच करता है, जिससे दुर्भावनापूर्ण दूसरे अकरेंस को गुजरने की अनुमति मिल जाती है
- [ ] हाँ — कॉनकेटनेशन एक हमलावर को डिटेक्शन से बचने के लिए पेलोड (Payload) को कई पैरामीटर्स में विभाजित (Split) करने की अनुमति देता है

### क्या एप्लीकेशन उचित सैनिटाइजेशन (Sanitization) के बिना रिस्पॉन्स में पलूशन वाले पैरामीटर्स को रिफ्लेक्ट करती है?
- [ ] नहीं — UI में रिफ्लेक्ट (Reflect) होने से पहले पैरामीटर्स को सैनिटाइज या एनकोड किया जाता है
- [ ] हाँ — पलूशन के परिणामस्वरूप कंटेंट रिफ्लेक्ट होता है, लेकिन सैनिटाइजेशन लागू किया गया है
- [ ] हाँ — पलूशन के परिणामस्वरूप कंटेंट रिफ्लेक्ट होता है और सैनिटाइजेशन लागू नहीं है, जिससे XSS या लॉजिक एरर्स (Logic errors) होते हैं

### क्या इंटरनल API कॉल्स या बैक-एंड सिस्टम को पास किए गए पैरामीटर्स पलूशन के प्रति संवेदनशील हैं?
- [ ] नहीं — बैक-एंड रिक्वेस्ट सुरक्षित, गैर-डायनेमिक निर्माण विधियों का उपयोग करते हैं
- [ ] हाँ — HPP एक हमलावर को बैक-एंड रिक्वेस्ट स्ट्रक्चर में पैरामीटर्स को इंजेक्ट (Inject) या ओवरराइट (Overwrite) करने की अनुमति देता है

### क्या एप्लीकेशन अलग व्यवहार प्रदर्शित करती है जब पैरामीटर्स GET बनाम POST रिक्वेस्ट में पलूशन वाले होते हैं?
- [ ] नहीं — व्यवहार सभी HTTP मेथड्स (Methods) में सुसंगत है
- [ ] हाँ — केवल GET पैरामीटर्स पलूशन के प्रति संवेदनशील हैं
- [ ] हाँ — केवल POST (body) पैरामीटर्स पलूशन के प्रति संवेदनशील हैं
