## WSTG-INFO-05 — सूचना रिसाव (Information Leakage) के लिए वेबपेज सामग्री की समीक्षा करें

वेबपेज सामग्री की समीक्षा करने में एप्लिकेशन के जवाबों (responses) का मैन्युअल और स्वचालित निरीक्षण शामिल है, जिसमें HTML, JavaScript और CSS फ़ाइलें शामिल हैं, ताकि अनजाने में उपयोगकर्ताओं के सामने उजागर हुई संवेदनशील जानकारी की पहचान की जा सके। हमलावर क्लाइंट-साइड सोर्स कोड (source code) की बारीकी से जांच करते हैं ताकि डेवलपर कमेंट्स (developer comments), छिपे हुए फॉर्म फ़ील्ड्स (hidden form fields), आंतरिक सर्वर पथ (internal server paths), हार्डकोडेड API कीज़ (hardcoded API keys), और डीबगिंग जानकारी (debugging information) का पता लगाया जा सके, जो आगे के शोषण (exploitation) चरणों में सहायता कर सकती है। यह रिसाव अक्सर तब होता है जब डेवलपर्स उत्पादन (production) वातावरण में जाने से पहले मेटाडेटा (metadata) या डीबगिंग कोड को हटाना भूल जाते हैं, जिससे संभावित रूप से लॉजिक की कमियां या बैकएंड इन्फ्रास्ट्रक्चर विवरण प्रकट हो सकते हैं। एक हमलावर के दृष्टिकोण से, यह सुरक्षा अलर्ट ट्रिगर किए बिना या सर्वर के बैकएंड लॉजिक के साथ सीधे बातचीत किए बिना एप्लिकेशन की आंतरिक संरचना को मैप करने और अनदेखा किए गए प्रवेश बिंदुओं (entry points) को खोजने का एक कम-प्रयास वाला तरीका प्रदान करता है।

| फ़ील्ड | मान |
|---|---|
| **WSTG ID** | WSTG-INFO-05 |
| **CWE** | CWE-200 |
| **परीक्षण की स्थिति** | निष्पादित नहीं किया गया |
| **गंभीरता** | कम / मध्यम* |

> *यदि संवेदनशील हार्डकोडेड क्रेडेंशियल्स (hardcoded credentials), निजी कुंजियाँ (private keys), या आंतरिक वातावरण चर (internal environment variables) खोजे जाते हैं, तो गंभीरता 'उच्च' (High) हो जाती है।

### क्या सोर्स कोड में संवेदनशील जानकारी वाले डेवलपर कमेंट्स मौजूद हैं?
- [ ] नहीं — कोई कमेंट्स नहीं मिले या केवल सामान्य, गैर-संवेदनशील कमेंट्स मिले  
- [ ] हाँ — कमेंट्स मौजूद हैं लेकिन उनमें **कोई** संवेदनशील लॉजिक या डेटा नहीं है  
- [ ] हाँ — कमेंट्स आंतरिक पथों (internal paths), SQL प्रश्नों (SQL queries), या प्रशासनिक निर्देशों को **प्रकट** करते हैं  
- [ ] हाँ — कमेंट्स क्रेडेंशियल्स या संवेदनशील डेवलपर नोट्स को **प्रकट** करते हैं *(उच्च)*  

### क्या छिपे हुए HTML इनपुट फ़ील्ड्स में संवेदनशील डेटा या स्थिति (state) की जानकारी है?
- [ ] नहीं — कोई छिपे हुए फ़ील्ड नहीं मिले या उनमें केवल गैर-संवेदनशील टोकन (tokens) हैं  
- [ ] हाँ — छिपे हुए फ़ील्ड्स का उपयोग स्थिति के लिए किया जाता है लेकिन वे **एन्क्रिप्टेड** (encrypted) या **हस्ताक्षरित** (signed) हैं  
- [ ] हाँ — छिपे हुए फ़ील्ड्स में **प्लेनटेक्स्ट** (plaintext) पासवर्ड, उपयोगकर्ता भूमिकाएं, या आंतरिक आईडी (IDs) शामिल हैं  

### क्या JavaScript फ़ाइलों में हार्डकोडेड सीक्रेट्स, API कीज़, या आंतरिक IP पते मौजूद हैं?
- [ ] नहीं — JS फ़ाइलों में कोई संवेदनशील स्ट्रिंग्स या सीक्रेट्स नहीं मिले  
- [ ] हाँ — JS फ़ाइलों में **उचित** प्रतिबंधों के साथ सार्वजनिक API कीज़ मौजूद हैं  
- [ ] हाँ — JS फ़ाइलों में **असुरक्षित** API कीज़, आंतरिक IP, या निजी एंडपॉइंट्स (endpoints) मौजूद हैं  
- [ ] हाँ — JS फ़ाइलों में **हार्डकोडेड** प्रशासनिक क्रेडेंशियल्स या निजी कुंजियाँ मौजूद हैं *(गंभीर)*  

### क्या एप्लिकेशन सोर्स में फ्रेमवर्क वर्जन या आंतरिक फ़ाइल पथ प्रकट करता है?
- [ ] नहीं — फ्रेमवर्क की जानकारी और फ़ाइल पथ **मिनिफाइड** (minified) या **ऑब्फसकेटेड** (obfuscated) हैं  
- [ ] हाँ — फ्रेमवर्क के नाम और वर्जन **दृश्यमान** हैं लेकिन पैच किए गए हैं  
- [ ] हाँ — सोर्स कोड में आंतरिक फ़ाइल पथ या पूर्ण सर्वर पथ (absolute server paths) **उजागर** हैं  

### क्या मिनिफिकेशन या ऑब्फसकेशन की कमी के कारण सोर्स कोड में रिसाव की संभावना है?
- [ ] नहीं — प्रोडक्शन कोड **पूरी तरह से मिनिफाइड** है और कमेंट्स हटा दिए गए हैं  
- [ ] हाँ — कोड मिनिफाइड है लेकिन सोर्स में कमेंट्स **बचे हुए** हैं  
- [ ] हाँ — सोर्स मैप्स (`.map` फ़ाइलें) **सार्वजनिक रूप से सुलभ** हैं, जिससे सोर्स कोड के पूर्ण पुनर्निर्माण की अनुमति मिलती है
