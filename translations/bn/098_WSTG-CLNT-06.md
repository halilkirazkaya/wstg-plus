## WSTG-CLNT-06 — ক্লায়েন্ট-সাইড রিসোর্স ম্যানিপুলেশন (Client-Side Resource Manipulation) পরীক্ষা করা

ক্লায়েন্ট-সাইড রিসোর্স ম্যানিপুলেশন (Client-Side Resource Manipulation) তখন ঘটে যখন একটি অ্যাপ্লিকেশন ব্যবহারকারীর ইনপুটকে ব্রাউজার দ্বারা লোড করা রিসোর্সের (যেমন scripts, stylesheets, বা iframes) সোর্স (source) বা পাথ (path) প্রভাবিত করার অনুমতি দেয়। URI ফ্রেগমেন্ট (fragments), কুয়েরি প্যারামিটার (query parameters), বা অন্যান্য DOM সোর্স ম্যানিপুলেট করে, একজন আক্রমণকারী (attacker) অ্যাপ্লিকেশনটিকে ক্ষতিকারক এক্সটার্নাল কন্টেন্ট লোড করতে বা অননুমোদিত কোড এক্সিকিউট করতে বাধ্য করতে পারে। এই দুর্বলতাটি সাধারণত JavaScript লজিকে পাওয়া যায় যা বিশ্বস্ত অ্যালো-লিস্ট (allow-list) এর বিরুদ্ধে পর্যাপ্ত ভ্যালিডেশন ছাড়াই HTML এলিমেন্টের `src` বা `href` অ্যাট্রিবিউটগুলো ডাইনামিকভাবে পপুলেট করে। একজন আক্রমণকারীর দৃষ্টিকোণ থেকে, এটি এমন একটি URL তৈরি করার মাধ্যমে এক্সপ্লয়েট (exploit) করা হয় যা রিসোর্স লোডিংকে আক্রমণকারীর নিয়ন্ত্রিত সার্ভারে রিডাইরেক্ট করে, যার ফলে DOM-ভিত্তিক ক্রস-সাইট স্ক্রিপ্টিং (Cross-Site Scripting (XSS)), সেনসিটিভ ডেটা এক্সফিল্ট্রেশন (data exfiltration), বা UI রিড্রেসিং (UI redressing) হতে পারে।

### অ্যাপ্লিকেশনটি কি রিসোর্সগুলো ডাইনামিকভাবে লোড বা এমবেড করার জন্য ক্লায়েন্ট-সাইড সিঙ্ক (client-side sinks) ব্যবহার করে?
- [ ] না — রিসোর্সগুলো শুধুমাত্র স্ট্যাটিক HTML বা সার্ভার-সাইড লজিকের মাধ্যমে লোড করা হয়  
- [ ] হ্যাঁ — ক্লায়েন্ট-সাইড JavaScript ডাইনামিকভাবে `src`, `href`, বা `data` এর মতো রিসোর্স অ্যাট্রিবিউটগুলো পপুলেট করে  

### রিসোর্স URI সোর্সের জন্য কি ভ্যালিডেশন কন্ট্রোল বা অ্যালো-লিস্ট (allow-lists) ইমপ্লিমেন্ট করা হয়েছে?
- [ ] হ্যাঁ — সমস্ত ডাইনামিক রিসোর্সের জন্য কঠোর অ্যালো-লিস্ট এবং অরিজিন ভ্যালিডেশন (origin validation) **প্রয়োগ করা হয়েছে**  
- [ ] হ্যাঁ — ভ্যালিডেশন উপস্থিত আছে কিন্তু এটি দুর্বল ব্ল্যাকলিস্ট (blacklists) বা সহজেই বাইপাস করা যায় এমন regex-এর ওপর নির্ভরশীল  
- [ ] না — ব্যবহারকারী-নিয়ন্ত্রিত রিসোর্স পাথের ওপর কোনো ভ্যালিডেশন কন্ট্রোল প্রয়োগ করা হয়নি  

### বিকল্প প্রোটোকল বা এনকোডিং (encoding) ব্যবহার করে কি URI ভ্যালিডেশন বাইপাস (bypass) করা সম্ভব?
- [ ] না — রিসোর্স ফিল্টার বাইপাস করা **সম্ভব নয়**  
- [ ] হ্যাঁ — `data:`, `vbscript:`, বা `javascript:` এর মতো বিভিন্ন প্রোটোকল ব্যবহার করে বাইপাস **সম্ভব**  
- [ ] হ্যাঁ — সাধারণ স্ট্রিং ম্যাচ এড়ানোর জন্য এনকোড করা ক্যারেক্টার বা নাল বাইট (null bytes) ব্যবহার করে বাইপাস **সম্ভব**  

### একজন আক্রমণকারী কি রিসোর্স লোডিংকে কোনো এক্সটার্নাল, অবিশ্বস্ত ডোমেনে রিডাইরেক্ট করতে পারে?
- [ ] না — রিসোর্স পাথগুলো একই অরিজিন (same origin) বা একটি নির্দিষ্ট সাবডিরেক্টরির মধ্যে সীমাবদ্ধ  
- [ ] হ্যাঁ — অ্যাপ্লিকেশনটিকে যেকোনো এক্সটার্নাল ডোমেন থেকে স্ক্রিপ্ট বা স্টাইল লোড করতে বাধ্য করা **যেতে পারে**  

### রিসোর্স ম্যানিপুলেশনের সর্বোচ্চ প্রভাব (impact) কী হতে পারে?
- [ ] নিম্ন — ম্যানিপুলেশন শুধুমাত্র ইমেজ বা সেনসিটিভ নয় এমন টেক্সটের মতো নন-এক্সিকিউটেবল এলিমেন্টগুলোকে প্রভাবিত করে  
- [ ] মাঝারি — ম্যানিপুলেশন সিএসএস ইনজেকশন (CSS Injection) বা উল্লেখযোগ্য UI রিড্রেসিং (UI redressing) এর অনুমতি দেয়  
- [ ] উচ্চ — ম্যানিপুলেশন DOM-ভিত্তিক XSS বা আর্কিট্রারি জাভাস্ক্রিপ্ট (arbitrary JavaScript) এক্সিকিউশনের দিকে নিয়ে যায় *(Critical)*
