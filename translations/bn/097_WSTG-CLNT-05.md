## WSTG-CLNT-05 — CSS ইনজেকশন (CSS Injection) পরীক্ষা

CSS ইনজেকশন (CSS Injection) তখন ঘটে যখন একটি অ্যাপ্লিকেশন সঠিক স্যানিটাইজেশন (Sanitization) বা এস্কেপিং (Escaping) ছাড়াই ব্যবহারকারীর দেওয়া ইনপুটকে একটি ওয়েব পেজের ক্যাসকেডিং স্টাইল শিটস (Cascading Style Sheets - CSS) প্রভাবিত করার অনুমতি দেয়। যদিও একে প্রায়ই একটি বাহ্যিক বা সাজসজ্জা সংক্রান্ত সমস্যা হিসেবে দেখা হয়, তবে আক্রমণকারীরা অ্যাট্রিবিউট সিলেক্টর (Attribute selectors) এবং ব্যাকগ্রাউন্ড-ইমেজ প্রপার্টিজ (background-image properties) ব্যবহার করে বাহ্যিক রিকোয়েস্ট (External request) ট্রিগার করার মাধ্যমে সিএসআরএফ (CSRF) টোকেন বা সেশন আইডি (Session IDs)-এর মতো সংবেদনশীল তথ্য এক্সফিল্ট্রেট (Exfiltrate) করতে পারে। এই দুর্বলতা সাধারণত এমন এন্ডপয়েন্টগুলিতে (Endpoints) দেখা যায় যেখানে ব্যবহারকারীর পছন্দ (যেমন: কাস্টম থিম, ফন্ট কালার) `<style>` ব্লক বা ইনলাইন `style` অ্যাট্রিবিউটের ভেতরে প্রতিফলিত হয়। একজন আক্রমণকারীর দৃষ্টিকোণ থেকে, এর সফল এক্সপ্লয়েট (Exploit) ইউআই রিড্রেসিং (UI redressing), অননুমোদিত লেআউট পরিবর্তনের মাধ্যমে ফিশিং (Phishing), অথবা এমন পরিবেশে স্টিলথি ডেটা এক্সফিল্ট্রেশনের (Stealthy data exfiltration) দিকে নিয়ে যেতে পারে যেখানে কঠোর কন্টেন্ট সিকিউরিটি পলিসি (Content Security Policy - CSP) অন্যথায় জাভাস্ক্রিপ্ট-ভিত্তিক আক্রমণ (JavaScript-based attacks) প্রতিরোধ করতে পারত।

|---|---|

### অ্যাপ্লিকেশন কি CSS কনটেক্সটের মধ্যে ব্যবহারকারী-নিয়ন্ত্রিত ইনপুট প্রতিফলিত করে?
- [ ] না — ইনপুট স্টাইল ট্যাগ, অ্যাট্রিবিউট বা CSS ফাইলের মধ্যে প্রতিফলিত হয় না  
- [ ] হ্যাঁ — ইনপুট প্রতিফলিত হয় তবে কঠোরভাবে নিরাপদ আলফানিউমেরিক ভ্যালুর মধ্যে সীমাবদ্ধ  
- [ ] হ্যাঁ — ইনপুট একটি `style` অ্যাট্রিবিউট বা `<style>` ব্লকের মধ্যে প্রতিফলিত হয়  

### CSS-নির্দিষ্ট মেটাক্যারেক্টার এবং ফাংশনগুলো কি সঠিকভাবে স্যানিটাইজ করা হয়েছে?
- [ ] হ্যাঁ — `{`, `}`, `:` এর মতো ক্যারেক্টার এবং `url()` এর মতো ফাংশনগুলো সঠিকভাবে এস্কেপ (Properly escaped) করা হয়েছে  
- [ ] হ্যাঁ — স্যানিটাইজেশন বিদ্যমান তবে এনকোডিং বা CSS কমেন্টের মাধ্যমে তা বাইপাস (Bypass) করা সম্ভব  
- [ ] না — CSS মেটাক্যারেক্টারগুলোতে কোনো স্যানিটাইজেশন প্রয়োগ করা হয়নি  

### CSS সিলেক্টর ব্যবহার করে কি ডেটা এক্সফিল্ট্রেশন (Data exfiltration) সম্ভব?
- [ ] না — সিলেক্টরগুলো বাহ্যিক রিকোয়েস্ট ট্রিগার করতে বা ডেটা লিক করতে পারে না  
- [ ] হ্যাঁ — অ্যাট্রিবিউট সিলেক্টর এবং `background-image` এর মাধ্যমে আংশিক ডেটা এক্সফিল্ট্রেশন সম্ভব  
- [ ] হ্যাঁ — অটোমেটেড ব্রুট ফোর্স (Brute force) টেকনিক ব্যবহার করে সংবেদনশীল টোকেনের পূর্ণ এক্সফিল্ট্রেশন সম্ভব  

### একটি কন্টেন্ট সিকিউরিটি পলিসি (CSP) কি CSS ইনজেকশনের ঝুঁকি প্রশমন করে?
- [ ] হ্যাঁ — `style-src` শুধুমাত্র 'self' এ সীমাবদ্ধ এবং `img-src` বা `connect-src` সীমাবদ্ধ করা হয়েছে  
- [ ] হ্যাঁ — CSP বিদ্যমান তবে এটি 'unsafe-inline' ব্যবহার করে অথবা ওয়াইল্ডকার্ড (Wildcard) বাহ্যিক ডোমেইন অনুমোদন করে  
- [ ] না — CSS এর মাধ্যমে বাহ্যিক রিসোর্স লোডিং প্রতিরোধ করার জন্য কোনো CSP ইমপ্লিমেন্ট করা হয়নি  

### এই দুর্বলতা কি ইউআই রিড্রেসিং (UI redressing) বা ফিশিং (Phishing) করার জন্য ব্যবহার করা যেতে পারে?
- [ ] না — ইনজেকশনের পরিধি লেআউট উল্লেখযোগ্যভাবে পরিবর্তন করার জন্য অত্যন্ত সীমিত  
- [ ] হ্যাঁ — ইউআই (UI) পরিবর্তন সম্ভব, যা ক্ষতিকারক এলিমেন্ট ওভারলে করার অনুমতি দেয়  
- [ ] হ্যাঁ — সম্পূর্ণ পেজ লেআউট নিয়ন্ত্রণ সম্ভব, যা অত্যন্ত বিশ্বাসযোগ্য ফিশিং আক্রমণকে সহজতর করে
