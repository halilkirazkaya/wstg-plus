## WSTG-CLNT-04 — ক্লায়েন্ট-সাইড ইউআরএল রিডাইরেক্টের (Client-Side URL Redirect) জন্য পরীক্ষা

ক্লায়েন্ট-সাইড ইউআরএল রিডাইরেকশন (Client-side URL redirection) তখন ঘটে যখন একটি ওয়েব অ্যাপ্লিকেশন (Web application) ব্যবহারকারী-নিয়ন্ত্রিত ইনপুট—সাধারণত ইউআরএল প্যারামিটার (URL parameters) বা হ্যাশ ফ্র্যাগমেন্টের (Hash fragments) মাধ্যমে—গ্রহণ করে এবং এটি `window.location` বা `document.location.href` এর মতো একটি ক্লায়েন্ট-সাইড রিডাইরেকশন সিঙ্ক (Client-side redirection sink) এর মধ্যে ব্যবহার করে। এই দুর্বলতাটি মূলত আক্রমণকারীরা অত্যাধুনিক ফিশিং ক্যাম্পেইন (Phishing campaigns) পরিচালনা করতে ব্যবহার করে, কারণ ভিকটিম বা আক্রান্ত ব্যক্তি শুরুতে একটি বৈধ ডোমেইনকে (Legitimate domain) বিশ্বাস করে এবং পরে নীরবে একটি ক্ষতিকারক সাইটে (Malicious site) রিডাইরেক্ট হয়ে যায়। ফিশিং ছাড়াও, যদি ইউআরএল বা `Referer` হেডারে সংবেদনশীল তথ্য উপস্থিত থাকে এবং সেই অবস্থায় রিডাইরেকশন ঘটে, তবে ক্লায়েন্ট-সাইড রিডাইরেক্ট ব্যবহার করে ওঅথ অ্যাক্সেস টোকেন (OAuth access tokens) বা সেশন আইডেন্টিফায়ারের (Session identifiers) মতো সংবেদনশীল তথ্য হাতিয়ে নেওয়া (Exfiltrate) সম্ভব। আধুনিক সিঙ্গেল পেজ অ্যাপ্লিকেশনগুলোতে (Single Page Applications - SPAs), এই দুর্বলতা প্রায়শই রাউটিং লজিক (Routing logic), অথেন্টিকেশনের (Authentication) পর "return to" প্যারামিটার, বা ভাষা পরিবর্তনের কার্যকারিতায় (Language-switching functionalities) দেখা যায়।

### অ্যাপ্লিকেশন কি ব্যবহারকারীর ইনপুটের ভিত্তিতে রিডাইরেক্ট পরিচালনা করতে ক্লায়েন্ট-সাইড স্ক্রিপ্ট ব্যবহার করে?
- [ ] না — রিডাইরেকশন শুধুমাত্র সার্ভার-সাইডে `3xx` HTTP স্ট্যাটাস কোডের (HTTP status codes) মাধ্যমে পরিচালিত হয়  
- [ ] হ্যাঁ — অ্যাপ্লিকেশন ইউআরএল প্যারামিটারের ভিত্তিতে নেভিগেট করতে `window.location` এর মতো জাভাস্ক্রিপ্ট সিঙ্ক ব্যবহার করে  
- [ ] হ্যাঁ — অ্যাপ্লিকেশন একটি ক্লায়েন্ট-সাইড ফ্রেমওয়ার্ক রাউটার (যেমন- React Router, Vue Router) ব্যবহার করে যা ব্যবহারকারীর দেওয়া পাথ (Path) প্রসেস করে  

### গন্তব্য ইউআরএল-এর জন্য কি ইনপুট ভ্যালিডেশন (Input Validation) বা হোয়াইটলিস্টিং (Whitelisting) কন্ট্রোল প্রয়োগ করা হয়েছে?
- [ ] হ্যাঁ — অনুমোদিত ডোমেইন/পাথ-এর একটি কঠোর হোয়াইটলিস্ট (Whitelist) প্রয়োগ করা হয়েছে এবং এটি বাইপাস (Bypass) করা সম্ভব নয়  
- [ ] হ্যাঁ — ভ্যালিডেশন বিদ্যমান কিন্তু এটি দুর্বল রেজেক্স (Regex) বা ব্ল্যাকলিস্টের (Blacklists) ওপর নির্ভরশীল, এবং এটি বাইপাস করা সম্ভব  
- [ ] না — অ্যাপ্লিকেশন রিডাইরেকশন টার্গেট হিসেবে যেকোনো নির্বিচারে স্ট্রিং (Arbitrary string) গ্রহণ করে  

### সাধারণ অবফাসকেশন (Obfuscation) কৌশল ব্যবহার করে কি রিডাইরেক্ট লজিক বাইপাস করা সম্ভব?
- [ ] না — কন্ট্রোলগুলো এনকোড করা ক্যারেক্টার (Encoded characters), প্রোটোকল-রিলেটিভ ইউআরএল (Protocol-relative URLs - `//`), এবং ব্যাকস্ল্যাশ সঠিকভাবে হ্যান্ডেল করে  
- [ ] হ্যাঁ — ইউআরএল এনকোডিং (URL encoding) বা ডাবল-এনকোডিং (Double-encoding) ব্যবহার করে বাইপাস সম্ভব  
- [ ] হ্যাঁ — প্রোটোকল-রিলেটিভ ইউআরএল বা `@` ক্যারেক্টার (যেমন- `https://expected.com@attacker.com`) ব্যবহার করে বাইপাস সম্ভব  
- [ ] হ্যাঁ — নির্দিষ্ট ব্রাউজার কোয়ার্কস (Browser quirks) বা নাল বাইট ইনজেকশন (Null byte injection) ব্যবহার করে বাইপাস সম্ভব  

### রিডাইরেকশন প্রক্রিয়াটি কি গন্তব্য সাইটের কাছে কোনো সংবেদনশীল তথ্য প্রকাশ করে?
- [ ] না — ইউআরএল-এ কোনো সংবেদনশীল তথ্য নেই বা রিডাইরেক্টের সময় হেডারের মাধ্যমে কোনো তথ্য পাঠানো হয় না  
- [ ] হ্যাঁ — সংবেদনশীল তথ্য (যেমন- সেশন টোকেন, API কী) `Referer` হেডারের মাধ্যমে বহিরাগত সাইটে ফাঁস (Leak) হয়  
- [ ] হ্যাঁ — ইউআরএল ফ্র্যাগমেন্টে (`#`) থাকা সংবেদনশীল তথ্য গন্তব্য সাইটের স্ক্রিপ্ট দ্বারা অ্যাক্সেস করা সম্ভব  

### রিডাইরেক্ট সিঙ্ক ব্যবহারের মাধ্যমে কি জাভাস্ক্রিপ্ট এক্সিকিউট করা সম্ভব (DOM-based XSS)?
- [ ] না — সিঙ্ক শুধুমাত্র `http` বা `https` প্রোটোকলে নেভিগেশন করার অনুমতি দেয়  
- [ ] হ্যাঁ — রিডাইরেক্ট সিঙ্ক `javascript:` বা `data:` URI ব্যবহারের অনুমতি দেয়, যার ফলে DOM-based XSS সম্ভব
