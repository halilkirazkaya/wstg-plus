## WSTG-IDNT-04 — অ্যাকাউন্ট এন্যুমারেশন এবং অনুমানযোগ্য ব্যবহারকারী অ্যাকাউন্ট পরীক্ষা (Testing for Account Enumeration and Guessable User Account)

অ্যাকাউন্ট এন্যুমারেশন (Account Enumeration) তখনই ঘটে যখন একটি অ্যাপ্লিকেশন HTTP রেসপন্স (HTTP Response), এরর মেসেজ (Error Message), বা পরিমাপযোগ্য টাইমিং ডিফারেন্স (Timing Difference)-এর পার্থক্যের মাধ্যমে কোনো ব্যবহারকারী অ্যাকাউন্টের অস্তিত্ব বা অনস্তিত্ব প্রকাশ করে। আক্রমণকারীরা বৈধ অ্যাকাউন্টগুলো শনাক্ত করতে পদ্ধতিগতভাবে ইউজারনেমের তালিকা জমা দিয়ে এই আচরণের সুযোগ নেয়, যা পরবর্তীতে ক্রেডেনশিয়াল স্টাফিং (Credential Stuffing), ব্রুট-ফোর্স অ্যাটাক (Brute Force Attack), বা সোশ্যাল ইঞ্জিনিয়ারিং (Social Engineering)-এর জন্য লক্ষ্যবস্তু করা হয়। এই ভালনারেবিলিটি সাধারণত লগইন পোর্টাল, পাসওয়ার্ড রিসেট ফিচার, রেজিস্ট্রেশন ফর্ম এবং ব্যবহারকারী-নির্দিষ্ট আইডেন্টিফায়ার হ্যান্ডেল করে এমন API এন্ডপয়েন্ট (API Endpoint)-গুলোতে দেখা যায়। একজন আক্রমণকারীর দৃষ্টিকোণ থেকে, সফল এন্যুমারেশন একটি ব্লাইন্ড ব্রুট-ফোর্স প্রচেষ্টাকে পরিচিত এবং বৈধ আইডেন্টিটির ওপর লক্ষ্যবস্তু আক্রমণে রূপান্তরিত করার মাধ্যমে অ্যাটাক সারফেস (Attack Surface) উল্লেখযোগ্যভাবে সংকুচিত করে।

### সমস্ত ক্ষেত্রে অথেনটিকেশন এরর মেসেজগুলো কি অভিন্ন?
- [ ] হ্যাঁ — অবৈধ ইউজারনেম এবং অবৈধ পাসওয়ার্ড উভয়ের জন্যই জেনেরিক এরর মেসেজ ব্যবহার করা হয়  
- [ ] হ্যাঁ — মেসেজগুলো একই রকম, তবে বিরামচিহ্ন বা কেসিং-এ সামান্য পার্থক্য **বিদ্যমান**  
- [ ] না — এরর মেসেজে স্পষ্টভাবে "User not found" বা "Incorrect password" উল্লেখ থাকে *(ভালনারেবল)*  

### পাসওয়ার্ড রিসেট বা "পাসওয়ার্ড ভুলে গেছেন" (Forgot Password Flow) প্রক্রিয়াটি কি অ্যাকাউন্টের অস্তিত্ব প্রকাশ করে?
- [ ] না — ইমেল/ইউজারনেম বিদ্যমান থাকুক বা না থাকুক, অ্যাপ্লিকেশন একটি জেনেরিক মেসেজ প্রদান করে  
- [ ] হ্যাঁ — নিয়ন্ত্রণ ব্যবস্থা বিদ্যমান, তবে রেসপন্স লেন্থ (Response Length) বা স্ট্যাটাস কোডের মাধ্যমে তা বাইপাস করা **সম্ভব**  
- [ ] হ্যাঁ — অ্যাপ্লিকেশনটি স্পষ্টভাবে নিশ্চিত করে যে রিসেট ইমেল পাঠানো হয়েছে নাকি অ্যাকাউন্টটির **কোনো অস্তিত্ব নেই**  

### রেজিস্ট্রেশন প্রক্রিয়াটি কি ইউজার এন্যুমারেশন থেকে সুরক্ষিত?
- [ ] না — রেজিস্ট্রেশন কোনো তথ্য ফাঁস করে না অথবা বাল্ক চেক প্রতিরোধ করতে CAPTCHA/রেট-লিমিটিং (Rate Limiting) ব্যবহার করে  
- [ ] হ্যাঁ — নিয়ন্ত্রণ ব্যবস্থা বিদ্যমান, তবে টাইমিং বা API রেসপন্সের পার্থক্যের মাধ্যমে তা বাইপাস করা **সম্ভব**  
- [ ] হ্যাঁ — ইউজারনেম বা ইমেলটি **ইতিমধ্যেই ব্যবহার করা হয়েছে** কিনা অ্যাপ্লিকেশনটি ব্যবহারকারীকে সাথে সাথে জানিয়ে দেয়  

### বৈধ বনাম অবৈধ ইউজারনেম তুলনা করার সময় কি টাইমিং ডিফারেন্স পরিমাপযোগ্য?
- [ ] না — কনস্ট্যান্ট-টাইম কম্পারিশন (Constant-time comparisons) ব্যবহারের কারণে অ্যাকাউন্টের বৈধতা নির্বিশেষে রেসপন্স টাইম একই থাকে  
- [ ] হ্যাঁ — টাইমিং ডিফারেন্স বিদ্যমান কিন্তু নেটওয়ার্কের মাধ্যমে নির্ভরযোগ্যভাবে পরিমাপ করার জন্য তা অত্যন্ত নগণ্য  
- [ ] হ্যাঁ — পরিমাপযোগ্য টাইমিং ডিসক্রিপেন্সি (Timing Discrepancies) **বিদ্যমান** (যেমন, শুধুমাত্র বৈধ ব্যবহারকারীদের জন্য পাসওয়ার্ড হ্যাশিং কার্যকর হওয়ার কারণে)  

### অ্যাপ্লিকেশনটি কি অ্যাকাউন্টের জন্য অনুমানযোগ্য নাম ব্যবহারের প্রথা অনুসরণ করে?
- [ ] না — অ্যাকাউন্ট আইডেন্টিফায়ারগুলো র্যান্ডম (UUIDs) অথবা ব্যবহারকারী-নির্ধারিত এবং জটিল  
- [ ] হ্যাঁ — অ্যাকাউন্ট আইডেন্টিফায়ারগুলো একটি নির্দিষ্ট প্যাটার্ন অনুসরণ করে (যেমন, `emp001`, `emp002`) এবং এন্যুমারেশন **সম্ভব**  
- [ ] হ্যাঁ — আইডেন্টিফায়ারগুলো অনুক্রমিক পূর্ণসংখ্যা (Sequential Integers), যা সম্পূর্ণ ডাটাবেস এন্যুমারেশন **সম্ভব** করে তোলে
