## WSTG-ATHN-06 — ব্রাউজার ক্যাশ দুর্বলতা পরীক্ষা (Testing for Browser Cache Weakness)

ব্রাউজার ক্যাশ দুর্বলতা (Browser Cache Weakness) তখন ঘটে যখন সংবেদনশীল তথ্য (Sensitive Information) লোকাল ব্রাউজার ক্যাশে সংরক্ষিত হয় এবং একই ফিজিক্যাল মেশিনে অ্যাক্সেস থাকা কোনো অননুমোদিত ব্যবহারকারী (Unauthorized User) তা পুনরুদ্ধার করতে পারে। যথাযথ ক্যাশ কন্ট্রোল হেডারের অভাবের ফলে ব্যবহারকারী লগ আউট করার বা সেশন শেষ করার পরেও ব্যক্তিগত তথ্য, অ্যাকাউন্টের বিবরণ বা সেশন আইডেন্টিফায়ার (Session Identifiers)-এর মতো সম্ভাব্য সংবেদনশীল ডেটা থেকে যেতে পারে। শেয়ার্ড বা পাবলিক টার্মিনালের আক্রমণকারীরা বা পরবর্তী ব্যবহারকারীরা ব্রাউজার হিস্ট্রি নেভিগেট করে, "Back" বাটন ব্যবহার করে অথবা ক্যাশ করা রেসপন্সগুলো এক্সফিল্ট্রেট (Exfiltrate) করার জন্য লোকাল ক্যাশ ফাইলগুলো পরীক্ষা করে এর সুযোগ নিতে পারে। অথেন্টিকেটেড কন্টেন্ট (Authenticated Content) হ্যান্ডেল করে এমন যেকোনো অ্যাপ্লিকেশনের জন্য `Cache-Control: no-store` এবং `Pragma: no-cache`-এর মতো কঠোর নির্দেশাবলী বাস্তবায়ন করা অত্যন্ত গুরুত্বপূর্ণ যাতে সংবেদনশীল ডেটা কখনোই ডিস্কে লেখা না হয়।


| ফিল্ড | মান |
|---|---|
| **WSTG ID** | WSTG-ATHN-06 |
| **CWE** | CWE-525 |
| **টেস্ট স্ট্যাটাস** | সম্পন্ন হয়নি (Not Performed) |
| **গুরুত্বের মাত্রা (Severity)** | নিম্ন (Low) / মাঝারি (Medium)* |

> *যদি অ্যাপ্লিকেশনটি প্রায়শই শেয়ার্ড/পাবলিক কিয়স্ক (Public Kiosks) থেকে অ্যাক্সেস করা হয় বা এতে অত্যন্ত সংবেদনশীল PII/PHI ডেটা থাকে তবে গুরুত্বের মাত্রা 'মাঝারি' হিসেবে গণ্য হবে।

**টুলস:** `Burp Suite (Proxy/Repeater)`, `Browser Developer Tools (Network Tab)`, `Zed Attack Proxy (ZAP)`

### অথেন্টিকেটেড বা সংবেদনশীল পেজগুলোতে কি যথাযথ ক্যাশ-কন্ট্রোল (Cache-control) হেডার বিদ্যমান?
- [ ] হ্যাঁ — `Cache-Control: no-store, no-cache` এবং `Pragma: no-cache` **উপস্থিত** এবং **সঠিকভাবে কনফিগার করা** আছে  
- [ ] হ্যাঁ — কিছু হেডার উপস্থিত আছে কিন্তু `no-store` **অনুপস্থিত**, যা ডিস্ক ক্যাশিং (Disk Caching)-এর অনুমতি দিচ্ছে  
- [ ] না — কোনো ক্যাশ-সম্পর্কিত হেডার **প্রয়োগ করা হয়নি**, যা ডিফল্ট ব্রাউজার আচরণের ওপর নির্ভর করছে  

### অ্যাপ্লিকেশনটি কি ব্যবহারকারী-নির্দিষ্ট ডেটার জন্য `private` ডিরেক্টিভ ব্যবহার করে?
- [ ] হ্যাঁ — প্রক্সি ক্যাশিং রোধ করতে সমস্ত পার্সোনালাইজড কন্টেন্টের জন্য `Cache-Control: private` **সক্রিয়** করা আছে  
- [ ] না — কন্টেন্টকে `public` হিসেবে চিহ্নিত করা হয়েছে বা এতে `private` ডিরেক্টিভ নেই, যা এটিকে ইন্টারমিডিয়েট প্রক্সি (Intermediate Proxies) দ্বারা **ক্যাশযোগ্য (Cachable)** করে তোলে  

### সফল লগআউটের পর ব্রাউজারের "Back" বাটনের মাধ্যমে কি সংবেদনশীল তথ্য অ্যাক্সেস করা সম্ভব?
- [ ] না — ব্রাউজার পুনরায় অথেন্টিকেশনের জন্য অনুরোধ করে অথবা পেজটি লোকাল ক্যাশ থেকে **লোড হয় না**  
- [ ] হ্যাঁ — সেশন শেষ হওয়ার পরেও ব্যাক বাটনের মাধ্যমে সংবেদনশীল তথ্য **এখনও দৃশ্যমান**  

### সংবেদনশীল নন-HTML ফাইলগুলো (যেমন: PDFs, CSV এক্সপোর্ট, JSON API রেসপন্স) কি ক্যাশিং থেকে সুরক্ষিত?
- [ ] না — অ্যাপ্লিকেশন দ্বারা কোনো সংবেদনশীল নন-HTML ফাইল তৈরি বা হ্যান্ডেল করা হয় না  
- [ ] হ্যাঁ — সমস্ত সংবেদনশীল ফাইল ডাউনলোড এবং API এন্ডপয়েন্টে কঠোর ক্যাশ-কন্ট্রোল হেডার **প্রয়োগ করা হয়েছে**  
- [ ] না — সংবেদনশীল ডাউনলোড বা API রেসপন্সগুলো লোকাল ব্রাউজার ক্যাশ বা টেম্পোরারি ডিরেক্টরিতে **সংরক্ষিত হয়**  

### অ্যাপ্লিকেশনটি কি বাসি বা পুরোনো ডেটা (Stale Data) ব্যবহার রোধ করতে `Expires: 0` বা অতীতের কোনো তারিখ ব্যবহার করে?
- [ ] হ্যাঁ — `Expires` হেডার `0` বা একটি ঐতিহাসিক তারিখে সেট করা আছে, যা **নিশ্চিত করে** যে ব্রাউজার কন্টেন্টটিকে তাৎক্ষণিকভাবে পুরোনো হিসেবে গণ্য করবে  
- [ ] না — সংবেদনশীল পেজগুলোর জন্য `Expires` হেডার **অনুপস্থিত** বা ভবিষ্যতের কোনো তারিখে সেট করা আছে
