## WSTG-INPV-03 — HTTP ভার্ব ট্যাম্পারিং (HTTP Verb Tampering) এর জন্য পরীক্ষা

HTTP ভার্ব ট্যাম্পারিং (HTTP Verb Tampering) ওয়েব সার্ভার এবং অ্যাপ্লিকেশন ফ্রেমওয়ার্কগুলো কীভাবে ব্যবহৃত HTTP মেথড (HTTP Method) এর ওপর ভিত্তি করে নির্দিষ্ট রিসোর্সে অ্যাক্সেস অথোরাইজ (Authorize) করে, সেই দুর্বলতাকে কাজে লাগায়। আক্রমণকারীরা `GET` বা `POST` এর মতো সাধারণ মেথডগুলোর পরিবর্তে `HEAD`, `PUT`, `OPTIONS`, অথবা ব্যাকএন্ড অসামঞ্জস্যপূর্ণভাবে প্রসেস করতে পারে এমন কোনো যথেচ্ছ, অপ্রচলিত স্ট্রিং (String) ব্যবহার করে নিরাপত্তার সীমাবদ্ধতাগুলো এড়ানোর চেষ্টা করে। এই দুর্বলতা সাধারণত তখন ঘটে যখন সিকিউরিটি কনফিগারেশন (Security Configuration)—যেমন Java EE web.xml ফিল্টার বা .NET অথোরাইজেশন রুলস—সুনির্দিষ্টভাবে অনুমোদিত মেথডগুলোর তালিকা প্রদান করে কিন্তু অন্য মেথডগুলো বিবেচনা করতে ব্যর্থ হয়, অথবা যখন "deny-all" পদ্ধতির পরিবর্তে "deny-by-method" পদ্ধতি ব্যবহার করা হয়। সফলভাবে এই দুর্বলতা বা এক্সপ্লয়েট (Exploit) কাজে লাগানোর ফলে অ্যাডমিনিস্ট্রেটিভ ফাংশনে অননুমোদিত অ্যাক্সেস (Unauthorized Access), ডেটা পরিবর্তন বা সার্ভারের ইন্টারনাল কনফিগারেশন সম্পর্কিত তথ্য ফাঁস হতে পারে।


| ক্ষেত্র (Field) | মান (Value) |
|---|---|
| **WSTG ID** | WSTG-INPV-03 |
| **CWE** | CWE-288 |
| **টেস্ট স্ট্যাটাস (Test Status)** | সম্পন্ন করা হয়নি |
| **গুরুত্ব (Severity)** | মাঝারি / উচ্চ* |

> *যদি অথেনটিকেশন (Authentication) ছাড়াই অ্যাডমিনিস্ট্রেটিভ ফাংশন বা ডেটা পরিবর্তন (PUT/DELETE) করা সম্ভব হয়, তবে গুরুত্ব 'উচ্চ' (High) হিসেবে বিবেচিত হবে।

### অ্যাপ্লিকেশন কি অপ্রচলিত বা যথেচ্ছ HTTP মেথডে সাড়া দেয়?
- [ ] না — অ্যাপ্লিকেশন স্পষ্টভাবে প্রয়োজনীয় মেথডগুলো ছাড়া বাকি সব মেথড প্রত্যাখ্যান করে  
- [ ] হ্যাঁ — অ্যাপ্লিকেশন স্ট্যান্ডার্ড মেথডগুলোতে (OPTIONS, TRACE) সাড়া দেয় কিন্তু নিরাপত্তা ব্যবস্থা এড়ানো সম্ভব নয়  
- [ ] হ্যাঁ — অ্যাপ্লিকেশন যথেচ্ছ স্ট্রিং (যেমন: FOO, TEST) গ্রহণ করে এবং সেগুলোকে `GET` বা `POST` রিকোয়েস্ট হিসেবে প্রসেস করে  

### HTTP মেথড পরিবর্তনের মাধ্যমে কি অথেনটিকেশন বা অথোরাইজেশন এড়ানো সম্ভব?
- [ ] না — ব্যবহৃত HTTP ভার্ব নির্বিশেষে নিরাপত্তা নিয়ন্ত্রণগুলো সার্বজনীনভাবে প্রয়োগ করা হয়  
- [ ] হ্যাঁ — নিরাপত্তা নিয়ন্ত্রণগুলো বিদ্যমান থাকলেও `HEAD` মেথড ব্যবহার করে তা এড়ানো (Bypass) সম্ভব  
- [ ] হ্যাঁ — বিকল্প মেথডগুলোতে সিকিউরিটি কন্ট্রোল প্রয়োগ করা হয়নি, যা সীমাবদ্ধ এন্ডপয়েন্টগুলোতে অননুমোদিত অ্যাক্সেসের অনুমতি দেয়  

### সার্ভারে কি `PUT` বা `DELETE` এর মতো বিপজ্জনক মেথডগুলো সক্রিয় আছে?
- [ ] না — বিপজ্জনক মেথডগুলো নিষ্ক্রিয় করা আছে অথবা 405 Method Not Allowed কোড প্রদান করে  
- [ ] হ্যাঁ — মেথডগুলো সক্রিয় আছে কিন্তু তার জন্য বৈধ ও উচ্চ-সুবিধা সম্পন্ন অথেনটিকেশন প্রয়োজন  
- [ ] হ্যাঁ — মেথডগুলো সক্রিয় (Enabled) আছে এবং অননুমোদিত ফাইল আপলোড বা রিসোর্স মুছে ফেলার অনুমতি দেয় *(Critical)*  

### `OPTIONS` মেথড কি অনুমোদিত ভার্ব সম্পর্কে কোনো সংবেদনশীল তথ্য প্রকাশ করে?
- [ ] না — `OPTIONS` নিষ্ক্রিয় করা আছে অথবা একটি সাধারণ রেসপন্স প্রদান করে  
- [ ] হ্যাঁ — `OPTIONS` মেথড `Allow` হেডার প্রদান করে কিন্তু কোনো সীমাবদ্ধ ইন্টারনাল মেথড প্রকাশ করে না  
- [ ] হ্যাঁ — `OPTIONS` মেথড শুধুমাত্র অভ্যন্তরীণ বা অ্যাডমিনিস্ট্রেটিভ মেথডগুলো প্রকাশ করে যা পরবর্তী অ্যাটাকে সাহায্য করতে পারে  

### `TRACE` মেথড কি সক্রিয় আছে, যা সম্ভাব্য ক্রস-সাইট ট্রেসিং (Cross-Site Tracing - XST) ঘটাতে পারে?
- [ ] না — `TRACE` এবং `TRACK` মেথডগুলো নিষ্ক্রিয় করা আছে  
- [ ] হ্যাঁ — `TRACE` সক্রিয় আছে এবং সংবেদনশীল কুকি (Cookie) বা টোকেনসহ HTTP হেডারগুলো রিফ্লেক্ট করে
