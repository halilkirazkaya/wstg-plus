## WSTG-INPV-02 — هجوم البرمجة العابرة للمواقع المخزن (Stored Cross Site Scripting - XSS)

يحدث هجوم البرمجة العابرة للمواقع المخزن (Stored Cross Site Scripting - XSS)، والمعروف أيضاً باسم (Persistent XSS)، عندما يستقبل التطبيق بيانات من مستخدم ويقوم بتخزينها في قاعدة بيانات مستمرة، أو نظام ملفات، أو أي وسيط تخزين آخر دون إجراء عملية تحقق (Validation) أو ترميز (Encoding) كافية. عندما يتصفح الضحية لاحقاً صفحة تسترجع وتعرض هذه البيانات غير المفلترة، يتم تنفيذ البرنامج النصي الخبيث (Malicious script) داخل سياق المتصفح الخاص به وتحت مصدر التطبيق (Origin). تكمن خطورة هذه الثغرة بشكل خاص في أنها لا تتطلب من الضحية النقر على رابط معد خصيصاً؛ فكل مستخدم يستعرض الصفحة المتأثرة يصبح هدفاً، مما قد يؤدي إلى اختطاف الجلسة (Session hijacking)، أو القيام بإجراءات غير مصرح بها، أو حصاد بيانات الاعتماد (Credential harvesting). يستهدف المهاجمون عادةً أقسام التعليقات، وملفات تعريف المستخدمين، ولوحات الرسائل، وسجلات الإدارة حيث يتم عرض المدخلات باستمرار للمستخدمين الآخرين أو المسؤولين.

### هل توجد نقاط دخول تقوم بتخزين مدخلات المستخدم لعرضها لاحقاً لمستخدمين آخرين؟
- [ ] لا — التطبيق لا يقوم بتخزين المدخلات التي يمكن للمستخدم التحكم فيها لعرضها لاحقاً.
- [ ] نعم — يتم تخزين المدخلات (مثل الملفات الشخصية، التعليقات) ولكن **لا** يتم عرضها في سياق HTML.
- [ ] نعم — يتم تخزين المدخلات وعرضها لمستخدمين آخرين أو مسؤولين.

### هل يتم تطبيق ترميز المخرجات أو التطهير (Sanitization) عند عرض البيانات المخزنة؟
- [ ] نعم — يتم تطبيق ترميز المخرجات المعتمد على السياق (Context-aware) ولا يمكن تجاوزه *(الأكثر أماناً)*
- [ ] نعم — يتم تطبيق التطهير (مثل `DOMPurify`) ولكن التجاوز **ممكن** عبر أخطاء في التكوين.
- [ ] لا — يتم عرض البيانات مباشرة في DOM **بدون** ترميز أو تطهير *(خطورة عالية)*

### هل يمكن تجاوز عملية التحقق من المدخلات أو التطهير باستخدام حمولات (Payloads) بديلة؟
- [ ] لا — تتعامل الفلاتر مع مختلف أنواع الترميز، والوسوم غير القياسية، ومعالجات الأحداث (Event handlers) بشكل آمن.
- [ ] نعم — التجاوز **ممكن** باستخدام ترميز كيانات HTML أو وسوم معينة (مثل `<svg>`، `<img>`).
- [ ] نعم — التجاوز **ممكن** عبر حمولات (Polyglot payloads) أو هجوم XSS القائم على الطفرة (Mutation-based XSS - mXSS).

### هل توفر سياسة أمان المحتوى (Content Security Policy - CSP) طبقة دفاع ثانوية؟
- [ ] نعم — تم **تفعيل** سياسة CSP صارمة تمنع تنفيذ البرامج النصية المضمنة (Inline scripts) والمصادر غير المصرح بها.
- [ ] نعم — سياسة CSP **مفعلة** ولكنها سيئة التكوين (مثل استخدام `unsafe-inline` أو نطاق `script-src` واسع).
- [ ] لا — لا توجد سياسة CSP **مفعلة** للتطبيق.

### هل من الممكن تحقيق "Stored XSS to RCE" أو سلاسل هجمات أخرى عالية التأثير (Blind XSS)؟
- [ ] لا — يقتصر التأثير على سياق المتصفح من جهة العميل.
- [ ] نعم — يمكن استخدام Stored XSS لاستهداف المسؤولين (Blind XSS) في لوحات التحكم الداخلية.
- [ ] نعم — يمكن ربط Stored XSS بثغرات أخرى (مثل CSRF، أو تسريب البيانات الحساسة).
