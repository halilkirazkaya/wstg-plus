## WSTG-ATHN-06 — اختبار نقاط ضعف التخزين المؤقت للمتصفح (Browser Cache Weakness)

يحدث ضعف التخزين المؤقت للمتصفح (Browser Cache Weakness) عندما يتم تخزين معلومات حساسة في التخزين المؤقت المحلي للمتصفح، حيث يمكن استرجاعها من قبل مستخدم غير مصرح له لديه حق الوصول إلى نفس الجهاز الفعلي. يؤدي غياب ترويسات التحكم في التخزين المؤقت (Cache-Control Headers) المناسبة إلى السماح للبيانات الحساسة المحتملة، مثل المعلومات الشخصية، أو تفاصيل الحساب، أو معرفات الجلسة (Session Identifiers)، بالبقاء بعد تسجيل خروج المستخدم أو إغلاق الجلسة. يمكن للمهاجمين أو المستخدمين اللاحقين على الأجهزة المشتركة أو العامة استغلال ذلك من خلال التنقل في سجل المتصفح (Browser History)، أو استخدام زر "الرجوع" (Back)، أو فحص ملفات التخزين المؤقت المحلية لتسريب الاستجابات المخزنة. يعد ضمان تنفيذ توجيهات صارمة مثل `Cache-Control: no-store` و `Pragma: no-cache` أمراً بالغ الأهمية لأي تطبيق يتعامل مع محتوى يتطلب مصادقة لضمان عدم كتابة البيانات الحساسة على القرص الصلب مطلقاً.


| الحقل | القيمة |
|---|---|
| **معرف WSTG** | WSTG-ATHN-06 |
| **CWE** | CWE-525 |
| **حالة الاختبار** | لم يتم تنفيذه |
| **الخطورة** | منخفضة / متوسطة* |

> *تصبح الخطورة متوسطة إذا كان الوصول إلى التطبيق يتم بشكل متكرر من أكشاك الإنترنت المشتركة/العامة أو إذا كان يحتوي على بيانات حساسة للغاية من نوع معلومات الهوية الشخصية (PII) أو المعلومات الصحية المحمية (PHI).

**الأدوات:** `Burp Suite (Proxy/Repeater)`, `Browser Developer Tools (Network Tab)`, `Zed Attack Proxy (ZAP)`

### هل تتوفر ترويسات التحكم في التخزين المؤقت المناسبة في الصفحات الموثقة أو الحساسة؟
- [ ] نعم — ترويسات `Cache-Control: no-store, no-cache` و `Pragma: no-cache` **موجودة** و **مهيأة بشكل صحيح**  
- [ ] نعم — بعض الترويسات موجودة ولكن `no-store` **مفقودة**، مما يسمح بالتخزين المؤقت على القرص  
- [ ] لا — لم يتم **تطبيق** أي ترويسات متعلقة بالتخزين المؤقت، مما يعتمد على السلوك الافتراضي للمتصفح  

### هل يستخدم التطبيق توجيه `private` للبيانات الخاصة بالمستخدم؟
- [ ] نعم — تم **تفعيل** `Cache-Control: private` لجميع المحتويات المخصصة لمنع التخزين المؤقت في الخوادم الوكيلة (Proxy Caching)  
- [ ] لا — تم تحديد المحتوى كـ `public` أو يفتقر إلى توجيه `private` مما يجعله **قابلاً للتخزين** بواسطة الخوادم الوكيلة الوسيطة  

### هل يمكن الوصول إلى المعلومات الحساسة عبر زر "الرجوع" (Back) في المتصفح بعد تسجيل الخروج بنجاح؟
- [ ] لا — يطلب المتصفح إعادة المصادقة أو أن الصفحة **لا يتم تحميلها** من التخزين المؤقت المحلي  
- [ ] نعم — المعلومات الحساسة **لا تزال مرئية** عبر زر الرجوع بعد إنهاء الجلسة  

### هل الملفات الحساسة غير المتعلقة بـ HTML (مثل ملفات PDF، وتصدير CSV، واستجابات JSON API) محمية من التخزين المؤقت؟
- [ ] لا — لا يتم إنشاء أو التعامل مع أي ملفات حساسة غير متعلقة بـ HTML بواسطة التطبيق  
- [ ] نعم — تم **تطبيق** ترويسات تحكم صارمة في التخزين المؤقت على جميع تنزيلات الملفات الحساسة ونقاط نهاية واجهة برمجة التطبيقات (API Endpoints)  
- [ ] لا — يتم **تخزين** التنزيلات الحساسة أو استجابات API في التخزين المؤقت المحلي للمتصفح أو المجلدات المؤقتة  

### هل يستخدم التطبيق `Expires: 0` أو تاريخاً في الماضي لمنع استخدام البيانات القديمة؟
- [ ] نعم — تم ضبط ترويسة `Expires` على `0` أو تاريخ قديم، مما **يضمن** أن يعامل المتصفح المحتوى على أنه منتهي الصلاحية فوراً  
- [ ] لا — ترويسة `Expires` **مفقودة** أو تم ضبطها على تاريخ مستقبلي للصفحات الحساسة
