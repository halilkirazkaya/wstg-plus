## WSTG-ATHN-11 — اختبار المصادقة متعددة العوامل (MFA)

يقوم اختبار المصادقة متعددة العوامل (Multi-Factor Authentication - MFA) بتقييم قوة الطبقة الأمنية الثانوية المصممة لمنع الوصول غير المصرح به حتى في حالة اختراق بيانات الاعتماد الأساسية. يحاول المهاجمون بشكل متكرر تجاوز (Bypass) المصادقة متعددة العوامل من خلال تحديد نقاط النهاية (Endpoints) التي لا يتم فرضها فيها بشكل متسق، مثل إصدارات الواجهات البرمجية (API) القديمة، أو الواجهات الخلفية للهواتف المحمولة، أو سير عمل إعادة تعيين كلمة المرور. غالبًا ما يتضمن الاستغلال (Exploitation) التلاعب باستجابات الخادم، أو تنفيذ الهجوم الشامل (Brute Force) على الأكواد قصيرة العمر (One-Time Password - OTP)، أو استغلال حالات السباق (Race Conditions) وعيوب إدارة الجلسات (Session Management) التي تسمح للمستخدم بالانتقال إلى حالة مصادق عليها دون تقديم العامل الثاني.

|---|---|
| **معرف WSTG** | WSTG-ATHN-11 |
| **CWE** | CWE-287 |
| **حالة الاختبار** | لم يتم التنفيذ |

### هل يتم فرض المصادقة متعددة العوامل (MFA) بشكل متسق عبر جميع بوابات المصادقة؟
- [ ] نعم — المصادقة متعددة العوامل مطلوبة لجميع محاولات تسجيل الدخول عبر الويب، والهاتف المحمول، والواجهة البرمجية (API)  
- [ ] نعم — ولكن لا يتم فرض المصادقة متعددة العوامل على نقاط نهاية محددة (مثل `/api/v1/login` القديم أو البوابات الخاصة بالهاتف المحمول)  
- [ ] لا — لم يتم تنفيذ المصادقة متعددة العوامل في التطبيق *(حرج)*  

### هل يمكن تجاوز خطوة التحقق من المصادقة متعددة العوامل عبر التنقل المباشر لعنوان URL أو التلاعب بالاستجابة؟
- [ ] لا — لا يمكن للتنقل المباشر أو التلاعب بالمعاملات (Parameter Tampering) تجاوز التحدي الأمني  
- [ ] نعم — من الممكن الانتقال مباشرة إلى عناوين URL الخاصة بلوحة التحكم الداخلية دون إكمال تحدي المصادقة متعددة العوامل  
- [ ] نعم — من الممكن التلاعب باستجابة الخادم (على سبيل المثال، تغيير `401 Unauthorized` إلى `200 OK`) للحصول على حق الوصول  

### هل عملية التحقق من كود المصادقة متعددة العوامل محمية ضد هجمات القوة العمياء (Brute Force)؟
- [ ] نعم — يتم تطبيق تقييد صارم للمعدل (Rate Limiting) أو قفل الحساب بعد محاولات OTP فاشلة متعددة  
- [ ] نعم — يوجد تقييد للمعدل ولكن يمكن تجاوزه عبر تدوير عناوين IP أو التلاعب بالترويسات (Header Manipulation)  
- [ ] لا — لا يتم فرض تقييد للمعدل، ومن الممكن تنفيذ هجوم شامل آلي على الأكواد  

### هل يحافظ التطبيق على حالة جلسة آمنة أثناء الانتقال للمصادقة متعددة العوامل؟
- [ ] نعم — يتم إصدار رموز الجلسة (Session Tokens) ذات الامتيازات العالية فقط بعد إكمال المصادقة متعددة العوامل بنجاح  
- [ ] لا — يتم إصدار ملف تعريف ارتباط لجلسة (Session Cookie) يعمل بكامل طاقته قبل إكمال المصادقة متعددة العوامل، مما يسمح بالوصول إلى بعض الميزات  
- [ ] لا — يستخدم التطبيق معرفًا ثابتًا أو انتقالًا متوقعًا للجلسة يمكن اختطافه  

### هل عوامل المصادقة البديلة (SMS، البريد الإلكتروني، أكواد النسخ الاحتياطي) عرضة للاستغلال؟
- [ ] لا — تستخدم جميع العوامل قيمًا آمنة وغير متوقعة ومحددة بوقت  
- [ ] نعم — أكواد النسخ الاحتياطي متوقعة أو يمكن حصرها (Enumerated)  
- [ ] نعم — يمكن إساءة استخدام وظيفة "إعادة إرسال الكود" لتنفيذ إغراق (Flooding) رسائل SMS/البريد الإلكتروني أو الكشف عن تفاصيل الاتصال الجزئية
