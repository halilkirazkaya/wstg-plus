## WSTG-CONF-12 — سياسة أمان المحتوى (Content Security Policy - CSP)

تُعد سياسة أمان المحتوى (Content Security Policy - CSP) آلية دفاعية بالغة الأهمية ضمن مفهوم الدفاع العميق (Defense-in-depth)، حيث يتم تنفيذها عبر ترويسات استجابة HTTP للحد من مخاطر مثل هجمات البرمجة النصية عبر المواقع (Cross-Site Scripting - XSS)، واختطاف النقرات (Clickjacking)، وهجمات حقن البيانات. من خلال تحديد الموارد الديناميكية المسموح بتحميلها، تساهم سياسة CSP التي تم تكوينها بشكل جيد في تقييد قدرة المهاجم على تنفيذ برمجيات نصية (Scripts) غير مصرح بها أو استخراج البيانات الحساسة إلى نطاقات خارجية. يقوم مختبرو الاختراق (Pentesters) بتقييم السياسة للبحث عن التوجيهات (Directives) المتساهلة للغاية، واستخدام الكلمات المفتاحية غير الآمنة مثل `'unsafe-inline'`، والاعتماد على شبكات توصيل المحتوى (CDNs) غير الآمنة التي قد تسهل عملية التجاوز (Bypass). إن ضعف سياسة CSP أو فقدانها لا ينشئ ثغرة أمنية بشكل مباشر، ولكنه يزيد بشكل كبير من تأثير هجمات الحقن الناجحة بسبب الفشل في توفير طبقة حماية ثانوية.

### هل تتوفر ترويسة سياسة أمان المحتوى (CSP) في استجابات التطبيق؟
- [ ] نعم — ترويسة `Content-Security-Policy` **متوفرة** و **مُطبقة**  
- [ ] نعم — ترويسة `Content-Security-Policy-Report-Only` **متوفرة** لأغراض الاختبار  
- [ ] لا — لا توجد ترويسة CSP **متوفرة** في الاستجابات  

### هل تم تقييد توجيهات `script-src` أو `default-src` بشكل صحيح؟
- [ ] نعم — تستخدم التوجيهات قوائم بيضاء صارمة، أو رموزًا لمرة واحدة (Nonces)، أو تشفيرًا بصميًا (Hashes) والتجاوز **غير ممكن**  
- [ ] نعم — التوجيهات **مكونة بشكل صحيح** ولكن القائمة البيضاء تتضمن شبكات توصيل محتوى (CDNs) معروفة بإمكانية تجاوزها  
- [ ] لا — تستخدم التوجيهات رموزًا عامة (`*`) أو مخططات `data:`، مما يجعل التجاوز **ممكنًا**  

### هل تسمح السياسة بتنفيذ البرمجيات النصية المضمنة غير الآمنة أو دالة eval؟
- [ ] لا — الكلمات المفتاحية `'unsafe-inline'` و `'unsafe-eval'` **غير موجودة**  
- [ ] نعم — الكلمة المفتاحية `'unsafe-inline'` **موجودة** ولكنها محمية بواسطة `nonce` أو `hash`  
- [ ] نعم — الكلمات المفتاحية `'unsafe-inline'` أو `'unsafe-eval'` **مفعلة** دون حماية إضافية *(حرج)*  

### هل التطبيق محمي ضد هجمات اختطاف النقرات (Clickjacking) عبر توجيه `frame-ancestors`؟
- [ ] نعم — تم ضبط `frame-ancestors` على `'none'` أو `'self'`  
- [ ] نعم — توجيه `frame-ancestors` **مفعل** ولكنه يسمح بنطاقات خارجية محددة وموثوقة  
- [ ] لا — توجيه `frame-ancestors` **مفقود**، مع الاعتماد على الترويسة القديمة `X-Frame-Options` أو عدم وجود حماية  

### هل توجد آلية للإبلاغ عن انتهاكات سياسة CSP؟
- [ ] نعم — التوجيهات `report-uri` أو `report-to` **مكونة** ونشطة  
- [ ] لا — الإبلاغ عن الانتهاكات **معطل** أو **غير مكون**
