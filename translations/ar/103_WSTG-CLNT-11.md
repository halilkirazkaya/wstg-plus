## WSTG-CLNT-11 — اختبار مراسلات الويب (Web Messaging)

تتيح مراسلات الويب (Web Messaging)، أو `postMessage` التواصل عبر الأصول المختلفة (Cross-Origin Communication) بين كائنات النوافذ، مثل الصفحة الأبوية ونافذة منبثقة أو إطار (iframe) مضمن. تعد هذه الآلية بالغة الأهمية لوظائف الويب الحديثة، ولكنها تسبب مخاطر جسيمة إذا فشل التطبيق المستلم في التحقق من هوية المرسل أو تعامل مع حمولة (Payload) الرسالة بشكل غير صحيح. يستغل المهاجمون هذه الثغرات عبر استضافة موقع خبيث يضمن التطبيق المستهدف ويرسل رسائل مصممة خصيصاً لإثارة إجراءات غير مقصودة، أو استخراج بيانات حساسة، أو تنفيذ ثغرات برمجة المواقع عبر المواقع القائمة على نموذج كائن المستند (DOM-based Cross-Site Scripting - XSS). يحدث الاستغلال الناجح عندما لا يقوم مستمعو الرسائل (Message Listeners) بالتحقق بصرامة من خاصية الأصل (Origin) أو تمرير `message.data` مباشرة إلى أحواض تنفيذ خطرة (Dangerous Sinks).

### هل يستخدم التطبيق واجهة برمجة تطبيقات `postMessage` للتواصل بين المستندات؟
- [ ] لا — مستمعو `postMessage` **ليسوا** موجودين في الكود البرمجي من جهة العميل  
- [ ] نعم — يتم استخدام `postMessage` للتواصل بين المكونات الداخلية  
- [ ] نعم — يتم استخدام `postMessage` للتواصل مع نطاقات خارجية أو عناصر (Widgets) تابعة لجهات خارجية  

### هل يتم التحقق بصرامة من أصل (Origin) الرسائل الواردة؟
- [ ] نعم — يتم التحقق من الأصل مقابل قائمة بيضاء صارمة باستخدام `===` أو مقارنة مطابقة *(الأكثر أماناً)*  
- [ ] نعم — يتم التحقق من الأصل باستخدام تعبير نمطي (Regex)، ولكن المنطق **عرضة** للتجاوز (Bypass) (على سبيل المثال، `^https://trusted.com`)  
- [ ] لا — **لا يتم** التحقق من خاصية `origin` مما يسمح لأي نطاق بإرسال رسائل  
- [ ] لا — يتم استخدام حرف بدل (Wildcard) `*` في `targetOrigin` الخاص بالمرسل، مما قد يؤدي لتسريب البيانات إلى مستمعين خبيثين  

### هل يتم تطهير حمولة (Payload) الرسالة قبل معالجتها من قبل التطبيق؟
- [ ] نعم — يتم التعامل مع البيانات كنص عادي ولا يتم استخدام أحواض تنفيذ خطرة  
- [ ] نعم — يتم تحليل البيانات (مثل `JSON.parse`) والتحقق منها قبل الاستخدام، والتجاوز **غير ممكن**  
- [ ] لا — يتم تمرير بيانات الرسالة مباشرة إلى أحواض تنفيذ خطرة مثل `innerHTML` أو `eval()` أو `setTimeout()`  
- [ ] لا — يتم استخدام بيانات الرسالة للتنقل في النافذة أو تغيير `location.href` دون التحقق من صحتها  

### هل يمكن للمهاجم إثارة إجراءات غير مصرح بها أو استخراج البيانات عبر رسالة خبيثة؟
- [ ] لا — حتى مع وجود رسالة منتحلة (Spoofed Message)، لا يمكن الوصول إلى إجراءات أو بيانات حساسة  
- [ ] نعم — يمكن لرسالة مصممة خصيصاً **إثارة** تغييرات حساسة في حالة التطبيق (مثل تغيير كلمة المرور، تحديث الملف الشخصي)  
- [ ] نعم — يمكن لرسالة مصممة خصيصاً **أن تؤدي** إلى ثغرات DOM-based XSS أو استخراج رموز CSRF أو بيانات الجلسة (Session Data)
