## WSTG-CONF-02 — اختبار تكوين منصة التطبيق

يتضمن اختبار تكوين منصة التطبيق (Application platform configuration testing) تدقيق إعدادات خادم الويب (Web Server) الكامن، وخادم التطبيقات (Application Server)، وأطر العمل لضمان تحصينها ضد الاستغلالات الشائعة. يسعى المهاجمون بنشاط للبحث عن بيانات الاعتماد الافتراضية (Default Credentials)، ونقاط ضعف المنصة غير المرقعة (Unpatched platform vulnerabilities)، والواجهات الإدارية المكشوفة (Exposed administrative interfaces) للحصول على وصول غير مصرح به أو تنفيذ التعليمات البرمجية (Execute code). غالبًا ما يؤدي سوء التكوين (Misconfigurations) إلى الكشف عن متغيرات البيئة الحساسة (Environment variables)، أو مسارات النظام الداخلية (Internal system paths)، أو وجود تطبيقات تجريبية (Sample applications) غير ضرورية تزيد من سطح الهجوم (Attack surface). من منظور احترافي، تُعد المنصة التي تم تكوينها بشكل سيئ نقطة دخول ذات احتمالية عالية لتحقيق الوصول الأولي (Initial access) إلى البيئة المستهدفة.

| الحقل | القيمة |
|---|---|
| **معرف WSTG** | WSTG-CONF-02 |
| **CWE** | CWE-16 |
| **حالة الاختبار** | لم يتم إجراؤه |
| **الخطورة** | متوسطة / عالية* |

> *تصبح الخطورة عالية إذا كانت الواجهات الإدارية متاحة ببيانات الاعتماد الافتراضية أو إذا كانت البرمجيات النصية التجريبية تسمح بتنفيذ التعليمات البرمجية عن بُعد (Remote Code Execution - RCE).

**المراجع:**
* https://owasp.org/www-project-web-security-testing-guide/latest/4-Web_Application_Security_Testing/02-Configuration_and_Deployment_Management_Testing/02-Test_Application_Platform_Configuration  
* https://hacktricks.wiki/en/pentesting-web/web-vulnerabilities-methodology.html  

**الأدوات:** `Nmap`, `Nikto`, `WhatWeb`, `Wappalyzer`, `Burp Suite`, `Curl`

### هل بيانات الاعتماد أو الحسابات الافتراضية نشطة على المنصة؟
- [ ] لا — جميع الحسابات الافتراضية **معطلة** أو تم تغيير كلمات المرور الخاصة بها  
- [ ] نعم — الحسابات الافتراضية موجودة ولكن **لا يمكن الوصول إليها** من الشبكة الخارجية  
- [ ] نعم — الحسابات الافتراضية **نشطة** ويمكن الوصول إليها عبر الإنترنت *(حرجة)*  

### هل تفصح المنصة عن معلومات حساسة حول الإصدار عبر الترويسات أو صفحات الخطأ؟
- [ ] لا — سلاسل الإصدار (Version strings) **مخفية** أو عامة  
- [ ] نعم — معلومات الإصدار التفصيلية **مكشوفة** في ترويسات HTTP (مثل `Server`, `X-Powered-By`)  
- [ ] نعم — تتبع المكدس الكامل (Full stack traces) ومسارات النظام الداخلية **مكشوفة** في رسائل الخطأ التفصيلية  

### هل واجهات الإدارة أو التحكم مقيدة بشكل صحيح؟
- [ ] لا — الواجهات الإدارية **ليست مكشوفة**  
- [ ] نعم — الواجهات موجودة ولكنها **مقيدة** بواسطة القائمة البيضاء لعناوين IP أو المصادقة متعددة العوامل (Multi-Factor Authentication - MFA)  
- [ ] نعم — الواجهات (مثل `/admin`, `/manager`, `/console`) **متاحة للعامة** بمصادقة أحادية العامل  

### هل توجد ملفات تجريبية أو برمجيات نصية أو وثائق على خادم الإنتاج؟
- [ ] لا — تم **إزالة** جميع الملفات غير الضرورية  
- [ ] نعم — الملفات التجريبية أو الوثائق **موجودة** ولكنها لا تسرب معلومات حساسة  
- [ ] نعم — البرمجيات النصية التجريبية (مثل `info.php`, `examples/`) **موجودة** وتوفر ناقل هجوم (Attack vector) مباشر  

### هل تم تكوين الخادم لدعم طرق HTTP غير الآمنة؟
- [ ] لا — تم **تمكين** طرقتي `GET` و `POST` فقط  
- [ ] نعم — الطرق التي قد تنطوي على مخاطر مثل `PUT` أو `DELETE` أو `TRACE` **ممكنة** ولكنها مقيدة  
- [ ] نعم — الطرق الخطرة **ممكنة** وتسمح بتعديل الملفات بشكل غير مصرح به أو سرقة بيانات الاعتماد
