## WSTG-CLNT-04 — اختبار إعادة توجيه الروابط من جانب العميل (Client-Side URL Redirect)

تحدث عملية إعادة توجيه الروابط من جانب العميل (Client-Side URL Redirect) عندما يقبل تطبيق الويب مدخلات يتحكم فيها المستخدم — عادةً من خلال معاملات الرابط (URL parameters) أو أجزاء الهاش (hash fragments) — ويستخدمها داخل "بالوعة" (sink) لإعادة التوجيه من جانب العميل مثل `window.location` أو `document.location.href`. يتم استغلال هذه الثغرة بشكل أساسي من قبل المهاجمين لإجراء حملات تصيد احتيالي (Phishing) متطورة، حيث يثق الضحية في البداية بالنطاق (Domain) الشرعي قبل أن يتم توجيهه بصمت إلى موقع ضار. بالإضافة إلى التصيد الاحتيالي، يمكن غالباً ربط عمليات إعادة التوجيه من جانب العميل لتسريب البيانات الحساسة مثل رموز الوصول الخاصة بـ OAuth (OAuth access tokens) أو معرفات الجلسة (session identifiers) إذا حدثت إعادة التوجيه أثناء وجود تلك القيم في الرابط (URL) أو في ترويسة المرجع (Referer header). في تطبيقات الصفحة الواحدة (Single Page Applications - SPAs) الحديثة، تظهر هذه الثغرة بشكل متكرر في منطق التوجيه (routing logic)، أو معاملات "العودة إلى" (return to) بعد المصادقة، أو وظائف تبديل اللغة.


| الحقل | القيمة |
|---|---|
| **معرف WSTG** | WSTG-CLNT-04 |
| **CWE** | CWE-601 |
| **حالة الاختبار** | لم يتم التنفيذ |
| **الخطورة** | متوسطة* |

> *تصبح الخطورة عالية (High) إذا أمكن استخدام إعادة التوجيه لتسريب رموز OAuth، أو معرفات الجلسة، أو تجاوز ضوابط الأمان في تدفق المصادقة (authentication flow).

### هل يستخدم التطبيق سكربتات من جانب العميل لمعالجة عمليات إعادة التوجيه بناءً على مدخلات المستخدم؟
- [ ] لا — يتم التعامل مع إعادة التوجيه حصرياً من جانب الخادم (server-side) عبر أكواد حالة HTTP من فئة `3xx`  
- [ ] نعم — يستخدم التطبيق "بالوعات" (sinks) لـ JavaScript مثل `window.location` للتنقل بناءً على معاملات الرابط  
- [ ] نعم — يستخدم التطبيق موجه إطار عمل من جانب العميل (client-side framework router) مثل (`React Router` أو `Vue Router`) الذي يعالج المسارات المقدمة من المستخدم  

### هل تم تنفيذ ضوابط التحقق من المدخلات أو القائمة البيضاء (whitelisting) للرابط الوجهة؟
- [ ] نعم — يتم فرض قائمة بيضاء (**whitelist**) صارمة للنطاقات/المسارات المسموح بها والتجاوز **غير ممكن**  
- [ ] نعم — يوجد تحقق ولكن يعتمد على تعبيرات نمطية (regex) ضعيفة أو قوائم سوداء (blacklists)، والتجاوز **ممكن**  
- [ ] لا — يقبل التطبيق أي سلسلة نصية عشوائية كوجهة لإعادة التوجيه  

### هل يمكن تجاوز منطق إعادة التوجيه باستخدام تقنيات التعمية (obfuscation) الشائعة؟
- [ ] لا — تتعامل الضوابط بشكل صحيح مع الأحرف المرمزة (encoded characters)، والروابط النسبية للبروتوكول (`//`)، والشرطات المائلة العكسية (backslashes)  
- [ ] نعم — التجاوز **ممكن** باستخدام ترميز الرابط (URL encoding) أو الترميز المزدوج (double-encoding)  
- [ ] نعم — التجاوز **ممكن** باستخدام الروابط النسبية للبروتوكول أو رموز `@` (مثال: `https://expected.com@attacker.com`)  
- [ ] نعم — التجاوز **ممكن** باستخدام ثغرات معينة في المتصفح أو حقن البايت الفارغ (null byte injection)  

### هل تكشف عملية إعادة التوجيه عن بيانات حساسة للموقع الوجهة؟
- [ ] لا — لا توجد بيانات حساسة في الرابط أو يتم إرسالها عبر الترويسات (headers) أثناء إعادة التوجيه  
- [ ] نعم — يتم تسريب بيانات حساسة (مثل رموز الجلسة أو مفاتيح API) عبر ترويسة المرجع (`Referer` header) إلى الموقع الخارجي  
- [ ] نعم — البيانات الحساسة الموجودة في جزء الرابط (`#`) (URL fragment) **يمكن الوصول إليها** بواسطة سكربتات الموقع الوجهة  

### هل من الممكن تنفيذ JavaScript عبر "بالوعة" إعادة التوجيه (DOM-based XSS)؟
- [ ] لا — تسمح "البالوعة" (sink) فقط بالتنقل إلى بروتوكولات `http` أو `https`  
- [ ] نعم — تسمح بالوعة إعادة التوجيه بمعرفات موارد `javascript:` أو `data:`، مما يجعل هجمات Cross-Site Scripting القائمة على DOM (DOM-based XSS) **ممكنة**
