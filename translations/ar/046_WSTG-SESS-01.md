## WSTG-SESS-01 — اختبار مخطط إدارة الجلسة (Testing for Session Management Schema)

يركز اختبار مخطط إدارة الجلسة (Session Management Schema Testing) على تحليل كيفية تعامل التطبيق مع دورة حياة وهيكل معرفات الجلسة لضمان قوتها ضد التنبؤ والاختطاف (Hijacking). يقوم المهاجمون بالتقاط رموز جلسة (Session Tokens) متعددة لإجراء تحليل إحصائي، بحثاً عن أنماط، أو عشوائية منخفضة (Low Entropy)، أو زيادات متوقعة تسمح لهم بتزييف جلسات صالحة لمستخدمين آخرين. غالباً ما تظهر نقاط الضعف في المخطط على شكل ثغرات تثبيت الجلسة (Session Fixation) أو استخدام قيم غير عشوائية بشكل كافٍ، والتي توجد عادةً في ملفات تعريف الارتباط (HTTP Cookies)، أو معلمات URL، أو تنفيذات الترويسة المخصصة (Custom Header). يعد اختراق مخطط الجلسة هجوماً عالي التأثير يمنح الخصم وصولاً كاملاً إلى الحالة المصادق عليها (Authenticated State) للضحية دون الحاجة إلى أوراق اعتمادهم الأساسية.

### هل معرف الجلسة معقد وعشوائي بما يكفي؟
- [ ] نعم — الرمز (Token) يجتاز اختبارات العشوائية الإحصائية (عشوائية عالية/High Entropy) والتجاوز (Bypass) **غير ممكن**  
- [ ] نعم — الرمز طويل ولكنه يحتوي على أنماط متوقعة أو أجزاء ثابتة  
- [ ] لا — الرمز تسلسلي، أو يعتمد على طوابع زمنية متوقعة، أو لديه عشوائية **منخفضة للغاية** *(Critical)*  

### أين يتم نقل وتخزين معرف الجلسة؟
- [ ] يتم تخزين المعرف في ملف تعريف ارتباط محمي بـ `Secure` و `HttpOnly` *(الأكثر أماناً)*  
- [ ] يتم تخزين المعرف في ملف تعريف ارتباط ولكنه يفتقر إلى وسوم `Secure` أو `HttpOnly`  
- [ ] يتم نقل المعرف عبر معلمات URL أو طلبات `GET` **فقط** *(مخاطر عالية/High Risk)*  

### هل يصدر التطبيق معرف جلسة جديد عند نجاح عملية المصادقة (Authentication)؟
- [ ] نعم — **يتم إنشاء** معرف جلسة (Session ID) جديد وفريد مباشرة بعد تسجيل الدخول  
- [ ] لا — يبقى معرف الجلسة كما هو قبل وبعد تسجيل الدخول *(احتمالية حدوث Session Fixation)*  

### هل معرف الجلسة مرتبط بعنوان IP محدد أو وكيل مستخدم (User-Agent) لمنع إعادة الاستخدام؟
- [ ] نعم — يتم إبطال الجلسة إذا تغيرت بصمة (Fingerprint) العميل بشكل ملحوظ  
- [ ] لا — **يمكن** إعادة استخدام الجلسة عبر عناوين IP أو أجهزة مختلفة دون تحقق إضافي  

### هل يتم إبطال معرفات الجلسة بشكل صحيح من جانب الخادم (Server-side) بعد تسجيل الخروج أو انتهاء المهلة؟
- [ ] نعم — يتم **تدمير حالة الجلسة تماماً** من جانب الخادم عند تسجيل الخروج  
- [ ] لا — تظل الجلسة صالحة على الخادم حتى لو تم حذف ملف تعريف الارتباط من جانب العميل  
- [ ] لا — الجلسة **لا تنتهي صلاحيتها أبداً** أو لديها فترة انتهاء مهلة (Timeout) طويلة بشكل مفرط
