## WSTG-INPV-16 — HTTP Request Smuggling

تحدث ثغرة تهريب طلبات HTTP (HTTP Request Smuggling - HRS) عندما تقوم سلسلة من خوادم HTTP (مثل موازن التحميل (Load balancer) وخادم ويب خلفي) بتفسير حدود الطلب بشكل مختلف، وعادةً ما يكون ذلك بسبب تعارض ترويسات `Content-Length` و `Transfer-Encoding`. يستغل المهاجم عدم التزامن (Desynchronization) هذا لـ "تهريب" إدخال إلى ذاكرة تخزين الطلبات الخاصة بالخادم الخلفي، مما يؤدي فعلياً إلى إضافة مقطع يتحكم فيه المهاجم كبادئة لطلب المستخدم الشرعي التالي. تسمح هذه التقنية بشن هجمات خطيرة تشمل اختطاف الجلسة (Session hijacking)، وتجاوز ضوابط الأمان، وتسميم ذاكرة التخزين المؤقت (Cache poisoning). يتم الاستغلال عادةً من خلال التحليل القائم على التوقيت (Timing-based analysis) أو عن طريق مراقبة الاستجابات غير المتوقعة من الخادم الخلفي عند إرسال طلبات لاحقة إلى الخادم.


| الحقل | القيمة |
|---|---|
| **WSTG ID** | WSTG-INPV-16 |
| **CWE** | CWE-444 |
| **حالة الاختبار** | لم يتم التنفيذ |
| **الخطورة** | حرجة / عالية |

### هل تتعامل الخوادم الأمامية (Front-end) والخلفية (Back-end) مع ترويسة `Transfer-Encoding: chunked` بشكل متسق؟
- [ ] نعم — كلا الخادمين يتعاملان مع الترميز المقسم (Chunked encoding) بشكل متطابق وإلغاء التزامن **غير ممكن**  
- [ ] لا — تفسر الخوادم الترويسات بشكل مختلف ولكن يتم تطبيق التطهير أو التحسين (Normalization)  
- [ ] لا — إلغاء التزامن من نوع CL.TE **ممكن** *(حرجة)*  
- [ ] لا — إلغاء التزامن من نوع TE.CL **ممكن** *(حرجة)*  

### هل يمكن للمهاجم تسميم ذاكرة التخزين المؤقت لجهة الخادم أو العميل باستخدام طلب مهرب؟
- [ ] لا — التخزين المؤقت (Caching) **معطل** أو غير معرض للتسميم  
- [ ] نعم — الطلبات المهربة **يمكنها** إعادة توجيه المستخدمين إلى نطاقات خبيثة أو حقن سكربتات عبر تسميم ذاكرة التخزين المؤقت  

### هل من الممكن التقاط طلبات مستخدمين آخرين أو رموز الجلسة (Session tokens) عبر دمج الطلبات؟
- [ ] لا — التهريب **لا** يسمح بالكشف عن بيانات المستخدمين الآخرين  
- [ ] نعم — يسمح التهريب بإلحاق طلب المستخدم التالي بجسم طلب `POST` يتحكم فيه المهاجم، مما يجعل تسريب البيانات (Exfiltration) **ممكناً**  

### هل تستخدم البنية التحتية بروتوكول HTTP/2، وهل هي معرضة لإلغاء التزامن من نوع H2.CL أو H2.TE؟
- [ ] لا — التطبيق يستخدم HTTP/1.1 فقط أو تم تنفيذ HTTP/2 بشكل آمن دون تخفيض الإصدار  
- [ ] نعم — تحدث عمليات تخفيض إصدار HTTP/2 إلى HTTP/1.1 (Cleartext downgrades) وإلغاء التزامن **ممكن**  

### هل يتم التعامل مع الترويسات المشوهة (على سبيل المثال، `Transfer-Encoding:  chunked` مع مسافة) بشكل آمن؟
- [ ] نعم — يتم رفض الترويسات المشوهة أو توحيدها (Normalized) عبر جميع الطبقات  
- [ ] لا — إلغاء التزامن من نوع TE.TE **ممكن** بسبب التعامل غير المتسق مع الترويسات المشوهة أو المخفية
