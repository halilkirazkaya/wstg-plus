## WSTG-INPV-07 — حقن لغة الترميز القابلة للتوسيع (XML Injection)

يحدث حقن لغة الترميز القابلة للتوسيع (XML Injection) عندما يدمج التطبيق البيانات المقدمة من المستخدم في مستند أو تدفق XML دون التحقق من صحتها (Validation) أو تطهيرها (Sanitization) أو معالجة رموز الهروب (Escaping) للرموز الوصفية (Meta-characters). من خلال حقن وسوم XML محددة أو عناصر هيكلية، يمكن للمهاجم التلاعب بمنطق المستند، أو تجاوز آليات المصادقة، أو التدخل في سلامة البيانات التي يتم معالجتها في الخلفية (Backend). توجد هذه الثغرة بشكل شائع في خدمات الويب القائمة على بروتوكول SOAP، وواجهات برمجة التطبيقات (API) من نوع REST التي تقبل XML، وتأكيدات SAML، والتطبيقات التي تنشئ ملفات تكوين XML أو تقارير بشكل ديناميكي. من وجهة نظر المهاجم، الهدف هو كسر سياق البيانات المقصود لتغيير بنية المستند، مما قد يؤدي إلى تصعيد الصلاحيات (Privilege Escalation) غير المصرح به أو تنفيذ منطق الأعمال (Business Logic) غير المقصود.

### هل يقبل التطبيق ويعالج المدخلات المنسقة بتنسيق XML من المستخدم؟
- [ ] لا — التطبيق **لا** يعالج مدخلات XML  
- [ ] نعم — يتم معالجة مدخلات XML عبر واجهات برمجة التطبيقات (SOAP/REST)  
- [ ] نعم — يتم معالجة XML عبر رفع الملفات (SVG, DOCX, إلخ.)  

### هل يتم معالجة رموز الهروب أو تطهير الرموز الوصفية لـ XML (مثل `<`, `>`, `&`, `'`, `"`) بشكل صحيح؟
- [ ] نعم — يتم معالجة رموز الهروب أو تطهير جميع الرموز الوصفية **بشكل صحيح** (الأكثر أماناً)  
- [ ] نعم — يتم تطبيق بعض الفلترة ولكن التجاوز **ممكن** باستخدام الترميز (Encoding)  
- [ ] لا — يتم دمج المدخلات الخام مباشرة في هياكل XML **بدون** تطهير  

### هل يتم فرض التحقق من صحة المخطط (Schema Validation) من جانب الخادم (XSD/DTD) لجميع مدخلات XML؟
- [ ] نعم — التحقق الصارم من صحة المخطط **مُفعل** ويمنع التغييرات الهيكلية  
- [ ] نعم — التحقق **مُفعل** ولكنه **لا يمنع** حقن الوسوم داخل حقول البيانات  
- [ ] لا — التحقق من صحة المخطط **معطل** أو غير مطبق  

### هل يمكن للمهاجم حقن وسوم XML جديدة لتغيير بنية المستند أو المنطق؟
- [ ] لا — تبقى البنية سليمة بغض النظر عن المدخلات  
- [ ] نعم — يمكن حقن الوسوم ولكن **لا يمكن** تغيير منطق الأعمال  
- [ ] نعم — حقن الوسوم **ممكن** ويسمح بتجاوز المنطق أو تعديل البيانات (حرجة)  

### هل يمكن التلاعب بمحلل XML باستخدام مقاطع CDATA أو تعليقات XML؟
- [ ] لا — المحلل يتعامل بشكل صحيح مع علامات CDATA/التعليقات أو يرفضها  
- [ ] نعم — حقن `<![CDATA[...]]>` أو `<!-- ... -->` **ممكن** لإخفاء أو تجاوز الفلاتر
