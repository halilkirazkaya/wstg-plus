## WSTG-IDNT-05 — اختبار ضعف أو عدم فرض سياسة أسماء المستخدمين (Testing for Weak or Unenforced Username Policy)

يركز اختبار سياسات أسماء المستخدمين الضعيفة أو غير المفروضة على القواعد التي تحكم إنشاء معرفات الحسابات ومدى قابليتها للاستقصاء (Enumeration) أو الانتحال (Spoofing). فغالبًا ما تسمح السياسات الضعيفة بتسلسلات يمكن التنبؤ بها، أو كلمات القاموس الشائعة، أو المعرفات التي تعكس هويات الموظفين الداخلية، مما يقلل بشكل كبير من مساحة البحث لهجمات القوة الغاشمة (Brute Force) وحشو بيانات الاعتماد (Credential Stuffing). من وجهة نظر المهاجم، يعد تحديد هذه الأنماط الخطوة الأولى في اكتشاف الحسابات على نطاق واسع، وهو ما يتم تحقيقه غالبًا من خلال تحليل رسائل خطأ التسجيل، أو سلوك إعادة تعيين كلمة المرور، أو البيانات الوصفية (Metadata) في الملفات الشخصية العامة. يمنع ضمان وجود سياسة قوية المهاجمين من رسم خريطة منهجية لقاعدة المستخدمين ويسهل الدفاع ضد التصيد الاحتيالي (Phishing) المستهدف ومحاولات تجاوز المصادقة (Authentication Bypass) الآلية.


| الحقل | القيمة |
|---|---|
| **معرف WSTG** | WSTG-IDNT-05 |
| **CWE** | CWE-521 |
| **حالة الاختبار** | لم يتم تنفيذه |
| **الخطورة** | منخفضة / متوسطة |

### هل تعتمد أسماء المستخدمين على أنماط متسلسلة أو يمكن التنبؤ بها للغاية؟
- [ ] لا — أسماء المستخدمين عشوائية، أو يحددها المستخدم، أو عبارة عن سلاسل نصية ذات إنتروبيا عالية (High-Entropy)
- [ ] نعم — تتبع أسماء المستخدمين تنسيقًا يمكن التنبؤ به (على سبيل المثال، `user1001` و `user1002`) ولكن المصادقة قوية  
- [ ] نعم — أسماء المستخدمين متسلسلة تمامًا أو تتبع تنسيقًا مؤسسيًا معروفًا، مما يسهل عملية الاستقصاء (Enumeration)

### هل يفرض التطبيق متطلبات الحد الأدنى للطول والتعقيد لأسماء المستخدمين؟
- [ ] نعم — يتم فرض سياسات صارمة للطول ومجموعات المحارف أثناء التسجيل  
- [ ] نعم — توجد سياسات ولكنها تسمح بأسماء مستخدمين قصيرة للغاية (مثل 1-2 محرف) أو بسيطة جدًا  
- [ ] لا — لا توجد سياسة مفروضة بخصوص طول اسم المستخدم أو أنواع المحارف  

### هل يمكن استقصاء أسماء المستخدمين الصالحة من خلال استجابات التطبيق؟
- [ ] لا — يعيد التطبيق رسائل خطأ عامة ويظهر توقيتًا ثابتًا (Consistent Timing) لجميع المحاولات  
- [ ] نعم — يعيد التطبيق رسائل خطأ متميزة (مثل "اسم المستخدم مأخوذ بالفعل") ولكن يتم تطبيق تحديد معدل الطلبات (Rate Limiting)  
- [ ] نعم — يسرب التطبيق أسماء مستخدمين صالحة من خلال أخطاء التسجيل، أو إعادة تعيين كلمة المرور، أو فروق التوقيت  

### هل تمنع السياسة تسجيل أسماء المستخدمين الإدارية الشائعة أو المحجوزة؟
- [ ] نعم — يحظر التطبيق الأسماء الشائعة (مثل `admin` و `root` و `support`) والمصطلحات المحجوزة للنظام  
- [ ] لا — يمكن للمستخدمين تسجيل حسابات بأسماء حساسة أو إدارية  

### هل سياسة اسم المستخدم متسقة عبر جميع نقاط الدخول (API، الهاتف المحمول، الويب)؟
- [ ] نعم — يتم تطبيق السياسات باستمرار عبر جميع الواجهات والإصدارات  
- [ ] لا — لا تفرض نقاط النهاية القديمة (Legacy Endpoints) أو إصدارات API محددة نفس قيود اسم المستخدم
