## WSTG-CONF-10 — الاستيلاء على النطاق الفرعي (Subdomain Takeover)

يحدث الاستيلاء على النطاق الفرعي (Subdomain Takeover) عندما يشير سجل نظام أسماء النطاقات (DNS entry) - عادةً ما يكون سجل CNAME، وأحياناً سجلات A أو MX - إلى مورد تم إيقاف تشغيله أو غير موجود لدى مزود خدمة سحابية (Cloud Provider) أو خدمة استضافة تابعة لجهة خارجية. يحدد المهاجمون سجلات نظام أسماء النطاقات "المعلقة" (Dangling DNS records) هذه من خلال البحث عن رسائل خطأ محددة من مزودين مثل AWS أو GitHub أو Azure تشير إلى أن المورد لم يعد مطلوباً أو مملوكاً لأحد. ومن خلال تسجيل اسم المورد المهجور على منصة المزود، يكتسب المهاجم سيطرة كاملة على النطاق الفرعي، مما يسمح له باستضافة محتوى ضار، وتسريب ملفات تعريف الارتباط للجلسة (Session Cookies) المخصصة للنطاق الأساسي، وتجاوز حماية سياسة أمان المحتوى (CSP) أو مشاركة الموارد عبر الأصول المختلفة (CORS). وتعد هذه الثغرة الأمنية خطيرة بشكل خاص لأنها تستغل الثقة المرتبطة بهيكل النطاق المشروع للمؤسسة لتسهيل عمليات التصيد الاحتيالي (Phishing) وسرقة البيانات عالية التأثير.

### هل يستخدم التطبيق استضافة خارجية أو خدمات سحابية عبر نطاقات فرعية؟
- [ ] لا — جميع النطاقات الفرعية تشير إلى مساحة عناوين IP تسيطر عليها المؤسسة.
- [ ] نعم — تشير النطاقات الفرعية إلى مزودي خدمة خارجيين (S3, GitHub Pages, Heroku, إلخ).

### هل توجد أي سجلات DNS "معلقة" تشير إلى موارد غير موجودة؟
- [ ] لا — جميع سجلات DNS التي تم تحديدها يتم حلها إلى موارد نشطة وتحت السيطرة.
- [ ] نعم — توجد سجلات CNAME أو A لموارد **لم تعد موجودة** لدى المزود.
- [ ] نعم — يتم حل سجلات DNS إلى NXDOMAIN أو صفحات خطأ خاصة بالمزود *(مثل "NoSuchBucket")*.

### هل يمكن الاستيلاء بنجاح على المورد المعلق الذي تم تحديده؟
- [ ] لا — لدى المزود حماية ضد المطالبة بالأسماء المهجورة أو يتطلب التحقق من الحساب.
- [ ] نعم — **يمكن** تسجيل اسم المورد على منصة الطرف الثالث بواسطة أي مستخدم.

### هل يستخدم النطاق الأساسي ملفات تعريف ارتباط بنطاق "Domain" يمكن اختراقها؟
- [ ] لا — ملفات تعريف الارتباط محصورة بدقة في نطاقات فرعية محددة أو تستخدم أعلام `Host-Only`.
- [ ] نعم — ملفات تعريف الارتباط الحساسة (معرفات الجلسة، JWTs) مخصصة للنطاق الأصل و **يمكن** تسريبها عبر نطاق فرعي مختطف.

### هل يمكن استخدام النطاق الفرعي لتجاوز ترويسات الأمان أو تدفقات المصادقة؟
- [ ] لا — لا توجد تبعيات أمنية على النطاقات الفرعية المحددة.
- [ ] نعم — النطاق الفرعي مدرج في القائمة البيضاء في توجيهات CSP `script-src` أو `connect-src`.
- [ ] نعم — النطاق الفرعي هو URI إعادة توجيه (Redirect URI) صالح لتكوينات OAuth أو SSO.
