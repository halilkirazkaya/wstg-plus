## WSTG-ATHZ-03 — اختبار تصعيد الامتيازات (Testing for Privilege Escalation)

يحدث تصعيد الامتيازات (Privilege Escalation) عندما يستغل المهاجم نقاط الضعف للحصول على وصول إلى موارد أو وظائف مخصصة لمستخدمين ذوي مستويات تصريح أعلى أو هويات مختلفة. في التصعيد الرأسي (Vertical Escalation)، يحاول مستخدم عادي الوصول إلى وظائف إدارية، بينما يتضمن التصعيد الأفقي (Horizontal Escalation) الوصول إلى بيانات مملوكة لمستخدم آخر بنفس مستوى الامتياز. تظهر هذه العيوب عادةً في قوائم التحكم في الوصول (ACLs) الضعيفة، أو مراجع الكائنات المباشرة غير الآمنة (IDOR)، أو التلاعب بالمعلمات (Parameter Tampering) ضمن رموز الجلسة أو الهوية. من وجهة نظر المهاجم، يتم تحقيق ذلك غالبًا عن طريق التلاعب بالمعرفات الرقمية، أو تعديل المعلمات المستندة إلى الأدوار في ملفات تعريف الارتباط (Cookies) أو رموز (JWT)، أو استدعاء نقاط نهاية واجهة برمجة التطبيقات (API) المخفية التي تفتقر إلى فحوصات التصريح من جانب الخادم.

### هل يطبق التطبيق مستويات امتياز متعددة أو تعدد المستأجرين (Multi-tenancy)؟
- [ ] لا — التطبيق هو نظام لمستخدم واحد أو دور واحد  
- [ ] نعم — تم تعريف أدوار أو مستأجرين متعددين وتتطلب اختبار التصريح  

### هل تصعيد الامتيازات الأفقي (الوصول على نفس المستوى) ممكن؟
- [ ] لا — يتم تطبيق فحوصات التصريح على جميع معرفات الموارد وأصحاب الجلسات  
- [ ] نعم — الوصول إلى بيانات المستخدمين الآخرين ممكن من خلال `IDOR` أو التلاعب بالمعلمات  
- [ ] نعم — تسريب البيانات ممكن ولكن تعديل بيانات المستخدمين الآخرين غير ممكن  

### هل تصعيد الامتيازات الرأسي (من منخفض إلى عالي) ممكن؟
- [ ] لا — نقاط النهاية الإدارية تفرض بصرامة ضوابط الوصول المستندة إلى الأدوار (RBAC)  
- [ ] نعم — يمكن الوصول إلى الوظائف الإدارية من قبل مستخدمين ذوي امتيازات منخفضة عبر الوصول المباشر إلى رابط URL  
- [ ] نعم — يمكن الوصول إلى الوظائف الإدارية عن طريق التلاعب بالترويسات المتعلقة بالأدوار أو ملفات تعريف الارتباط أو ادعاءات (JWT claims)  

### هل يمكن تعديل أدوار المستخدم أو أذوناته عبر التعيين الجماعي (Mass Assignment) أو تلوث المعلمات (Parameter Pollution)؟
- [ ] لا — الحقول المستندة إلى الأدوار هي للقراءة فقط ولا يمكن تعديلها من قبل المستخدمين  
- [ ] نعم — يمكن رفع صلاحيات المستخدم من خلال تضمين معلمات مخفية (مثل `role=admin`) في تحديثات الملف الشخصي أو التسجيل  

### هل يتم فرض فحوصات التصريح بشكل متسق عبر جميع إصدارات API وطرق HTTP؟
- [ ] نعم — يتم تطبيق الضوابط بشكل متسق عبر جميع الإصدارات والطرق  
- [ ] لا — إصدارات API القديمة (مثل `/v1/`) أو طرق HTTP محددة (مثل `PUT`, `DELETE`, `PATCH`) تتجاوز التصريح  
- [ ] لا — الوظائف الإدارية مخفية في واجهة المستخدم ولكنها مفعلة على مستوى API لجميع المستخدمين
