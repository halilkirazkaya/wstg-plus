## WSTG-ATHN-02 — اختبار بيانات الاعتماد الافتراضية (Testing for Default Credentials)

يتضمن اختبار بيانات الاعتماد الافتراضية (Default Credentials) تحديد واجهات الإدارة أو الخدمات أو مكونات التطبيق التي لا تزال تستخدم أسماء المستخدمين وكلمات المرور المحددة من قبل المصنع أو المورد. تُعد هذه الثغرة هدفاً عالي الأولوية للمهاجمين لأنها غالباً ما توفر مساراً فورياً للاختراق الكامل للنظام، أو الوصول الإداري، أو تنفيذ تعليمات برمجية عن بُعد (Remote Code Execution) بأقل جهد ممكن. تحدث هذه الثغرة عادةً في البرمجيات الجاهزة، ومنصات إدارة المحتوى (CMS)، وأدوات إدارة قواعد البيانات، والأجهزة المتكاملة مثل أجهزة إنترنت الأشياء (IoT) أو أجهزة الشبكة. يتم الاستغلال عادةً عن طريق مطابقة إصدارات البرامج المحددة مع قواعد بيانات كلمات المرور الافتراضية العامة أو استخدام أدوات حشو بيانات الاعتماد (Credential Stuffing) المؤتمتة خلال مراحل الاستطلاع والاستغلال الأولية.

| الحقل | القيمة |
|---|---|
| **WSTG ID** | WSTG-ATHN-02 |
| **CWE** | CWE-1392 |
| **حالة الاختبار** | لم يتم تنفيذه |
| **الخطورة** | حرجة / عالية |

### هل واجهات الإدارة أو التحكم معروضة للإنترنت أو لقطاعات شبكة غير موثوقة؟
- [ ] لا — لا يمكن الوصول إلى أي واجهات إدارة  
- [ ] نعم — الواجهات موجودة ولكنها مقيدة بضوابط على مستوى الشبكة (على سبيل المثال، VPN، قائمة العناوين المسموح بها - IP Allowlist)  
- [ ] نعم — الواجهات **متاحة** للعامة وتتطلب مصادقة (Authentication)  

### هل تم تغيير بيانات الاعتماد الافتراضية المقدمة من المورد لجميع المكونات المحددة؟
- [ ] نعم — تم تغيير جميع بيانات الاعتماد الافتراضية في كافة المكونات *(الأكثر أماناً)*  
- [ ] نعم — تم تغيير بيانات الاعتماد للتطبيق الرئيسي، ولكن المكونات الثانوية (مثل CMS، وقواعد البيانات) لا تزال تستخدم البيانات الافتراضية  
- [ ] لا — بيانات الاعتماد الافتراضية **فعالة** على واجهة واحدة أو أكثر من الواجهات المحددة *(حرجة)*  

### هل يفرض التطبيق تغيير كلمة المرور عند تسجيل الدخول الأول للحسابات الجديدة أو الإدارية؟
- [ ] نعم — تغيير كلمة المرور **إلزامي** قبل اتخاذ أي إجراء إداري  
- [ ] لا — **يسمح** التطبيق باستخدام بيانات الاعتماد الافتراضية لأجل غير مسمى  

### هل آليات الحظر أو ضوابط تحديد المعدل (Rate Limiting) فعالة لمنع الاختبار الآلي لبيانات الاعتماد الافتراضية؟
- [ ] نعم — يتم تطبيق تحديد المعدل الصارم أو حظر الحساب (Account Lockout)  
- [ ] نعم — الضوابط موجودة ولكن التجاوز **ممكن** عبر التلاعب بالترويسات (Header Manipulation) أو تدوير عنوان الآي بي (IP Rotation)  
- [ ] لا — لم يتم تفعيل أي آليات لتحديد المعدل أو الحظر  

### هل تستخدم الإضافات الخارجية أو البرمجيات الوسيطة أو الأدوات الإدارية (مثل phpMyAdmin، Tomcat Manager) بيانات اعتماد فريدة؟
- [ ] لا — لا توجد مكونات من طرف ثالث في البيئة  
- [ ] نعم — تستخدم جميع مكونات الطرف الثالث بيانات اعتماد فريدة وغير افتراضية  
- [ ] لا — يمكن الوصول إلى مكون واحد على الأقل من مكونات الطرف الثالث عبر بيانات اعتماد افتراضية معروفة
