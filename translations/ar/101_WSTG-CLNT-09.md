## WSTG-CLNT-09 — اختبار اختطاف النقرات (Clickjacking)

يحدث اختطاف النقرات (Clickjacking)، المعروف أيضاً باسم إعادة تصميم واجهة المستخدم (UI redressing)، عندما يستخدم المهاجم طبقات شفافة أو معتمة لخداع المستخدم للنقر على زر أو رابط في صفحة أخرى بينما كان ينوي النقر على الصفحة العلوية. تؤدي هذه الثغرة إلى المساس بسلامة إجراءات المستخدم، مما قد يؤدي إلى تعديل غير مصرح به للبيانات، أو تغييرات في الحساب، أو تحويلات مالية يتم إجراؤها نيابة عن الضحية. عادةً ما يستغل المهاجمون ذلك عن طريق تضمين الموقع المستهدف داخل `<iframe>` في موقع ضار يسيطرون عليه ووضع محتوى مخادع أو مغريات فوقه. وتحدث هذه الثغرة بشكل متكرر في الصفحات التي تؤدي إجراءات حساسة لتغيير الحالة مثل أزرار "حذف الحساب"، أو نماذج إعادة تعيين كلمة المرور، أو لوحات التكوين الإدارية.


| الحقل | القيمة |
|---|---|
| **WSTG ID** | WSTG-CLNT-09 |
| **CWE** | CWE-1021 |
| **حالة الاختبار** | Not Performed |
| **الخطورة** | متوسطة / عالية* |

> *تصبح الخطورة عالية إذا كان من الممكن تشغيل إجراءات حساسة (مثل تحويل الأموال، أو حذف الحساب، أو تصعيد الامتيازات (Privilege Escalation)) عبر اختطاف النقرات.

### هل يطبق التطبيق رؤوس استجابة (Headers) من جهة الخادم لمنع التأطير (Framing) غير المصرح به؟
- [ ] نعم — يتم تطبيق `X-Frame-Options` أو سياسة أمان المحتوى (CSP) وتمنع التأطير *(الأكثر أماناً)*  
- [ ] نعم — الرؤوس موجودة ولكنها **سيئة التكوين** (مثل `X-Frame-Options: ALLOW-FROM` التي تفتقر إلى دعم واسع من المتصفحات)  
- [ ] لا — لم يتم تطبيق أي رؤوس حماية ضد التأطير  

### هل تم تنفيذ توجيه `frame-ancestors` ضمن سياسة أمان المحتوى (CSP)؟
- [ ] نعم — يتم تطبيق `frame-ancestors 'none'` أو `'self'`  
- [ ] نعم — توجيه `frame-ancestors` موجود ولكنه **يسمح** بمصادر غير موثوقة أو واسعة النطاق بشكل مفرط  
- [ ] لا — التوجيه **مفقود** أو لم يتم تنفيذ CSP  

### هل تم تكوين رأس `X-Frame-Options` (XFO) بشكل صحيح لدعم المتصفحات القديمة؟
- [ ] نعم — يتم تطبيق `DENY` أو `SAMEORIGIN`  
- [ ] نعم — رأس XFO موجود ولكن يتم **تجاهله** بواسطة المتصفحات الحديثة بسبب وجود توجيه `frame-ancestors` في CSP  
- [ ] لا — رأس XFO **مفقود** أو تم ضبطه على قيمة غير آمنة  

### هل يمكن تجاوز حماية التأطير باستخدام تقنيات معروفة؟
- [ ] لا — التجاوز **غير ممكن** عبر التقنيات القياسية (التأطير المزدوج، إلخ)  
- [ ] نعم — **يمكن** تجاوز الحماية بسبب تعبير نمطي (Regex) ضعيف أو منطق خاطئ في القائمة البيضاء لـ `frame-ancestors`  
- [ ] نعم — **يمكن** تجاوز الحماية لأن التطبيق يعتمد فقط على كسر الإطارات (Frame-busting) المستند إلى JavaScript  

### هل يمكن استغلال إجراء حساس عبر إثبات مفهوم (Proof-of-concept) لاختطاف النقرات؟
- [ ] لا — التأطير محظور أو لا يمكن الوصول إلى إجراءات حساسة  
- [ ] نعم — إعادة تصميم واجهة المستخدم **ممكنة** ولكنها تتطلب تفاعلاً معقداً من المستخدم  
- [ ] نعم — إعادة تصميم واجهة المستخدم **ممكنة** وتسمح بإجراءات فورية لتغيير الحالة *(حرج)*
