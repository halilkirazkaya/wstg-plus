## WSTG-BUSL-02 — اختبار القدرة على تزوير الطلبات (Test Ability to Forge Requests)

يتضمن تزوير الطلبات (Forging requests) ضمن سياق منطق الأعمال (Business Logic) التلاعب بالمعلمات المحددة من قبل التطبيق لتنفيذ إجراءات خارج سير العمل المقصود أو مستوى الصلاحيات (Authorization) المسموح به. يستهدف المهاجمون العمليات متعددة الخطوات، مثل أنظمة الدفع أو تسجيل الحسابات، حيث يتم الحفاظ على الحالة عبر معلمات من جانب العميل (Client-side parameters) يفشل الخادم في إعادة التحقق منها مقابل مصدر خلفي موثوق (Trusted backend source). من خلال تعديل الحقول المخفية (Hidden fields)، أو قيم JSON، أو معلمات REST مثل الأسعار أو الكميات أو أعلام الحالة الداخلية، يمكن للمهاجم تجاوز متطلبات الدفع، أو تصعيد الصلاحيات (Privilege Escalation)، أو تحفيز تغييرات غير مقصودة في الحالة. تظهر هذه الثغرة عادةً في نماذج الويب التي تعتمد على الحالة (Stateful web forms) أو واجهات برمجة التطبيقات (APIs) حيث يثق الخادم ضمنياً في تمثيل العميل لحالة العمل أثناء المعاملة.


| الحقل | القيمة |
|---|---|
| **معرف WSTG** | WSTG-BUSL-02 |
| **CWE** | CWE-602 |
| **حالة الاختبار** | لم يتم إجراؤه |
| **الخطورة** | عالية / حرجة* |

> *تصبح الخطورة حرجة إذا سمح الطلب المزور بوقوع خسارة مالية، أو إجراءات إدارية غير مصرح بها، أو الاستيلاء الكامل على الحساب (Full account takeover).

### هل يقوم التطبيق بالتحقق من صحة معلمات المعاملات مقابل "مصدر حقيقة" (Source of Truth) من جانب الخادم؟
- [ ] نعم — يتم التحقق من جميع المعلمات الحساسة (السعر، الدور، المعرف) مقابل قاعدة البيانات والتجاوز **غير ممكن**  
- [ ] نعم — يتم تطبيق التحقق ولكن يمكن تجاوزه عبر تلويث المعلمات (Parameter Pollution) أو الترميز البديل  
- [ ] لا — يثق التطبيق في القيم القادمة من جانب العميل لتحديد تكلفة أو نتيجة المعاملة *(حرجة)*  

### هل يمكن تعديل القيم الحساسة للأعمال (مثل الكميات، المبالغ) إلى قيم غير مصرح بها؟
- [ ] لا — يتم رفض القيم السالبة، أو القيم الصفرية، أو القيم العالية بشكل مفرط من قبل الخادم  
- [ ] نعم — يقبل الخادم القيم المعدلة ولكن فقط ضمن نطاق محدود  
- [ ] نعم — **يمكن** إرسال قيم سالبة أو صفرية، مما يؤدي إلى تجاوزات مالية أو منطقية  

### هل تُستخدم الحقول المخفية أو معلمات API للحفاظ على الحالة عبر العمليات متعددة الخطوات؟
- [ ] لا — يتم الحفاظ على الحالة بشكل آمن عبر جلسات جانب الخادم أو الرموز المميزة الموقعة (Signed tokens)  
- [ ] نعم — يتم تمرير الحالة عبر العميل ولكن فحوصات النزاهة (HMAC/Signatures) **مفعلة** وقوية  
- [ ] نعم — يتم تمرير الحالة عبر العميل وفحوصات النزاهة **معطلة** أو يمكن إزالتها  

### هل من الممكن تزوير طلبات لتخطي خطوات وسيطة إلزامية في سير العمل (Workflow)؟
- [ ] لا — يفرض الخادم تسلسلاً صارماً للعمليات لكل معاملة  
- [ ] نعم — **يمكن** تخطي الخطوات ولكن الإجراء النهائي لا يزال يتطلب بيانات صالحة من الخطوات السابقة  
- [ ] نعم — يمكن تزوير طلب "الإرسال" أو "الإتمام" النهائي مباشرة لتجاوز خطوات المصادقة أو الدفع
