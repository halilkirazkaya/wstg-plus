## WSTG-ATHN-07 — اختبار سياسة كلمة المرور الضعيفة (Testing for Weak Password Policy)

يتضمن اختبار سياسات كلمة المرور الضعيفة تقييم متطلبات التعقيد، والطول، والعشوائية (Entropy) التي يفرضها التطبيق أثناء إنشاء الحساب وتحديث بيانات الاعتماد. تؤدي السياسات غير الكافية إلى المساس المباشر بسرية حسابات المستخدمين بجعلها عرضة لهجمات القاموس الآلية (Automated Dictionary Attacks)، ومحاولات التخمين بالقوة الغاشمة (Brute Force)، وحشو الاعتمادات (Credential Stuffing). تكمن هذه الثغرات عادةً في نقاط نهاية التسجيل (Registration Endpoints)، وسير عمل إعادة تعيين كلمة المرور، ولوحات إدارة المستخدمين الإدارية. من وجهة نظر المهاجم، تقلل السياسة المتساهلة بشكل كبير من التكلفة الحسابية والوقت المطلوب لاختراق الاعتمادات بنجاح باستخدام قوائم الكلمات الشائعة أو الكسر القائم على الأنماط (Pattern-based Cracking).


| الحقل | القيمة |
|---|---|
| **معرف WSTG** | WSTG-ATHN-07 |
| **CWE** | CWE-521 |
| **حالة الاختبار** | لم يتم تنفيذه |
| **الخطورة** | متوسطة / منخفضة* |

> *تصبح الخطورة "عالية" إذا كان التطبيق يفتقر إلى آليات قفل الحساب (Account Lockout) أو تقييد المعدل (Rate Limiting) لمنع التخمين الآلي.

**المراجع:**
* https://owasp.org/www-project-web-security-testing-guide/latest/4-Web_Application_Security_Testing/04-Authentication_Testing/07-Testing_for_Weak_Authentication_Methods  
* https://hacktricks.wiki/en/pentesting-web/login-bypass/index.html  

**الأدوات:** `Burp Suite (Intruder)`، `ZAP (Fuzzer)`، `Hydra`، `Hashcat`، `Cewl`

### هل يفرض التطبيق حداً أدنى لطول كلمة المرور؟
- [ ] نعم — الحد الأدنى للطول هو 12 حرفًا فأكثر *(الأكثر أمانًا)*  
- [ ] نعم — الحد الأدنى للطول يتراوح بين 8 و 11 حرفًا  
- [ ] نعم — الحد الأدنى للطول **أقل من** 8 حروف  
- [ ] لا — لا يتم فرض حد أدنى للطول  

### هل يتم فرض متطلبات التعقيد (أحرف كبيرة، أحرف صغيرة، أرقام، رموز)؟
- [ ] نعم — استخدام فئات متعددة من الأحرف إلزامي وتجاوزها **غير ممكن**  
- [ ] نعم — يُقترح استخدام فئات الأحرف ولكن **لا يتم** فرضها  
- [ ] لا — لا يتم تطبيق أي متطلبات تعقيد  

### هل يقوم التطبيق بحظر كلمات المرور الشائعة/الضعيفة أو كلمات المرور التي تحتوي على بيانات خاصة بالمستخدم؟
- [ ] نعم — كلمات المرور الضعيفة المعروفة (مثل "password123") وكلمات المرور القائمة على اسم المستخدم **لا يمكن** استخدامها  
- [ ] نعم — يتم حظر كلمات المرور الشائعة، ولكن يمكن استخدام البيانات الخاصة بالمستخدم (مثل اسم المستخدم، البريد الإلكتروني)  
- [ ] لا — يتم قبول أي سلسلة نصية تستوفي متطلبات الطول/التعقيد  

### هل يمكن تجاوز متطلبات التعقيد أو الطول عبر تعديل من جهة العميل (Client-side)؟
- [ ] لا — يتم فرض السياسات بصرامة من **جهة الخادم (Server-side)**  
- [ ] نعم — يتم فرض السياسات عبر JavaScript و **يمكن** تجاوزها عن طريق اعتراض الطلب  

### هل يتم إبلاغ المستخدم بسياسة كلمة المرور بوضوح دون تسريب تفاصيل التنفيذ؟
- [ ] نعم — السياسة مرئية أثناء الإدخال وتوفر رسائل فشل عامة  
- [ ] نعم — السياسة مرئية ولكن رسائل الفشل تكشف عن تعبيرات نمطية (Regex) محددة أو ثغرات في المنطق  
- [ ] لا — السياسة **غير** مرئية، مما يضطر المستخدم لاكتشافها عبر التجربة والخطأ
