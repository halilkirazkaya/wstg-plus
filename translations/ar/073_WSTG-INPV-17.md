## WSTG-INPV-17 — اختبار حقن ترويسة المضيف (Host Header Injection)

يحدث حقن ترويسة المضيف (Host Header Injection) عندما يعتمد التطبيق على ترويسة `Host` التي يوفرها المستخدم ويدمجها في منطق جانب الخادم (Server-side logic)، مثل إنشاء الروابط أو عمليات إعادة التوجيه، دون التحقق الكافي من صحتها. يقوم المهاجمون بالتلاعب بهذه الترويسة لتسهيل تسميم ذاكرة التخزين المؤقت للويب (Web Cache Poisoning)، أو تسميم إعادة تعيين كلمة المرور (Password Reset Poisoning) عن طريق إعادة توجيه الرموز الحساسة (Tokens) إلى نطاق خارجي، أو تجاوز ضوابط الأمان التي تعتمد على التوجيه المستند إلى الترويسة. يتيح الاستغلال الناجح للمهاجم تسريب بيانات الجلسة، أو الاستيلاء على الحساب (Account Takeover)، أو إعادة توجيه المستخدمين إلى بنية تحتية خبيثة. توجد هذه الثغرة بشكل شائع في بيئات موازنة الحمل (Load-balanced environments) أو التطبيقات التي تنشئ عناوين URL مطلقة ديناميكيًا بناءً على سياق الطلب الوارد.

| الحقل | القيمة |
|---|---|
| **WSTG ID** | WSTG-INPV-17 |
| **CWE** | CWE-20 |
| **حالة الاختبار** | لم يتم التنفيذ (Not Performed) |
| **الخطورة** | متوسطة / عالية* |

> *تصبح الخطورة "عالية" إذا تم استخدام ترويسة Host لإنشاء روابط حساسة في رسائل البريد الإلكتروني، مثل روابط إعادة تعيين كلمة المرور.

### هل يعكس التطبيق قيمة ترويسة `Host` في الروابط، عمليات إعادة التوجيه، أو السكربتات؟
- [ ] لا — يستخدم التطبيق عناوين URL أساسية ثابتة (Hardcoded) أو إعدادات تم التحقق منها بصرامة.
- [ ] نعم — تنعكس ترويسة `Host` ولكن يتم التحقق منها بصرامة مقابل قائمة بيضاء (Whitelist).
- [ ] نعم — تنعكس ترويسة `Host` ومن الممكن التجاوز (Bypass) عبر قيم مشوهة (مثل: `expected.com:attacker.com`).
- [ ] نعم — تنعكس ترويسة `Host` دون أي عملية تحقق من الصحة.

### هل تتم معالجة ترويسات بديلة متعلقة بالمضيف من قبل التطبيق أو البنية التحتية؟
- [ ] لا — يتم تجاهل الترويسات مثل `X-Forwarded-Host` أو `X-Host` أو `Forwarded`.
- [ ] نعم — تتم معالجة الترويسات البديلة ولكنها لا تتجاوز المنطق الداخلي.
- [ ] نعم — يمكن استخدام الترويسات البديلة لتجاوز قيمة ترويسة `Host` الأساسية.

### هل يمكن التلاعب بترويسة `Host` لتسميم رسائل البريد الإلكتروني التي ينشئها النظام (مثل إعادة تعيين كلمة المرور)؟
- [ ] لا — تستخدم روابط البريد الإلكتروني نطاقًا ثابتًا وموثوقًا من إعدادات الخادم.
- [ ] نعم — تؤثر ترويسة `Host` على روابط البريد الإلكتروني ولكن لا يمكن تجاوز التحقق من الصحة.
- [ ] نعم — يمكن التلاعب بترويسة `Host` لإعادة توجيه رموز إعادة تعيين كلمة المرور إلى نطاق يسيطر عليه المهاجم *(حرجة - Critical)*.

### هل التطبيق عرضة لتسميم ذاكرة التخزين المؤقت للويب (Web Cache Poisoning) عبر ترويسة `Host`؟
- [ ] لا — لا توجد آلية تخزين مؤقت أو أن ترويسة `Host` جزء من مفتاح التخزين المؤقت (Cache key).
- [ ] نعم — يوجد تخزين مؤقت ولكنه يتحقق بصرامة من ترويسة `Host` أو يتجاهلها للمدخلات غير المفتاحية (Unkeyed inputs).
- [ ] نعم — يوجد تخزين مؤقت ويمكن تسميمه بواسطة ترويسة `Host` محقونة لتقديم محتوى خبيث لمستخدمين آخرين.

### هل يقبل الخادم ترويسات `Host` عشوائية لتوجيه المضيف الافتراضي (Virtual Host Routing)؟
- [ ] لا — يعيد الخادم خطأً أو صفحة افتراضية لترويسات `Host` غير المعترف بها.
- [ ] نعم — يقبل الخادم أي ترويسة `Host`؛ مما يفيد في عمليات الاستطلاع أو تعداد الخدمات الداخلية (Internal service enumeration).
